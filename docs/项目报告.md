> 项目版本：v0.1.0-MVP

## 第一章 需求分析

### 1.1 开发背景

在快节奏的现代生活中，情绪管理与自我记录的需求持续增长，但传统日记应用存在以下痛点：
- **门槛高**：用户需主动打开空白文本框，不知道从何下笔
- **坚持难**：结构化输入带来的认知负担让用户难以养成日记习惯
- **价值低**：普通日记 App 仅做文字存档，缺乏结构化数据沉淀与情绪洞察能力

**Echo（回响）** 的核心出发点是：通过 AI 角色扮演"老朋友"的方式引导对话，让"记日记"变成一段轻松的睡前闲聊。对话结束后，AI 自动将内容转化为结构化日记，同时提取情绪、事件、消费、地点、标签等多维数据，帮助用户构建个人生活数据库，实现长期价值的自然沉淀。

项目定名"Echo（回响）"，寓意每一天的记录都是生命中的一次回声，被永久留存。

### 1.2 面向用户

| 用户画像 | 说明 |
|---|---|
| 核心用户 | 18-35岁，希望养成日记习惯但难以坚持的年轻人 |
| 次要用户 | 关注情绪健康与自我成长，有记录生活意愿的人群 |
| 潜在用户 | 对 AI 产品有接受度，愿意尝试智能生活辅助工具的用户 |

### 1.3 主要功能

| 功能模块 | 核心能力 | 状态 |
|---|---|---|
| 用户注册/登录 | 手机号+密码，JWT Token 鉴权 | ✅ 已实现 |
| AI 陪伴对话 | 与 AI「Echo」进行自然语言对话，AI 主动引导记录 | ✅ 已实现 |
| 日记自动生成 | 对话结束后 AI 自动生成完整日记正文及结构化五表数据 | ✅ 已实现 |
| 日记编辑（Path A） | 用户手动修改正文，触发 AI 重新提取结构化数据 | ✅ 已实现 |
| 日记补充（Path B） | 已完成会话继续追加 AI 对话，保存后重建日记 | ✅ 已实现 |
| 历史回忆 | 日历视图、时间轴、跨日五维数据表格 | ✅ 已实现 |
| 打卡与希冀 | 连续打卡统计、习惯追踪、长期目标进度 | 🔨 UI原型 |
| 个人中心 | 统计数据、昵称修改、偏好设置 | ✅ 已实现 |

### 1.4 主要性能指标

- AI 单次调用超时上限：**90 秒**（GLM-5 模型特性决定）
- JWT Token 有效期：**30 天**（减少用户重复登录摩擦）
- 分页默认每页 **20 条**，最大 **100 条**
- 历史时间轴单次最大返回 **500 条**
- AI 对话上下文窗口：**最近 20 条消息**（防止 Token 超限）

### 2.2 功能模块层次结构

```
Echo 系统
├── 认证模块 (auth)
│   ├── 注册（手机号+密码，同步初始化用户设置）
│   └── 登录（bcrypt 验证，签发 30天 JWT）
│
├── 用户模块 (users)
│   ├── 查询个人信息（手机号脱敏返回）
│   ├── 更新昵称/头像
│   └── 统计数据（记录天数/连续打卡/总事件数）
│
├── 设置模块 (settings)
│   └── 时区/输入偏好/提醒设置 CRUD
│
├── 会话模块 (conversations)  ← 核心交互入口
│   ├── 获取/创建今日会话
│   ├── 发送消息（同步返回 AI 回复）
│   │   └── is_supplement=true → 允许 completed 会话继续接收
│   └── 完成会话 → 触发 AI 日记生成
│
├── 日记录模块 (daily-records)  ← 数据沉淀核心
│   ├── 获取今日记录（含 body_text 正文）
│   ├── 手动补录（指定历史日期，从正文直接创建）
│   ├── 轻量编辑（user_note/keywords/tags）
│   ├── 更新正文 Path A（手动改→AI重提结构化数据）
│   └── 保存补充 Path B（对话追加→AI重建日记）
│
└── 历史模块 (history)  ← 数据消费层
    ├── 日记列表（分页+多条件筛选）
    ├── 指定日期详情（带五子表全量数据）
    ├── 日历视图（按月份高亮打卡点）
    ├── 时间轴（按月分组倒序）
    └── 五维数据表（events/emotions/expenses/locations/tags）
```

## 第三章 详细设计

### 3.1 界面设计

**设计风格**：移动端优先（最大宽度 430px），暖色系（主色调 `#F5A623`），Inter 字体，卡片式布局，底部四 Tab 导航栏，全局圆角（`--radius-lg: 20px`），玻璃磨砂效果（`backdrop-filter: blur`）。

**底部导航四 Tab**：

| Tab | 页面 | 核心功能 |
|---|---|---|
| ✍️ 今日 | today.html | AI 对话记录、日记查看与编辑 |
| 📖 回忆 | index.html | 历史日记列表、日历、五表视图 |
| 🌱 希冀 | hope.html | 连续打卡、习惯追踪、目标进度 |
| 👤 我 | profile.html | 个人统计、昵称修改、设置 |

**典型使用流程**：

```
[login.html]
  注册/登录 → 自动跳转
       ↓
[today.html] 今日页
  ├── 未记录 → 点击「开始记录」
  │           → 创建会话 → 发送消息 ↔ AI 追问

[index.html] 回忆页
  ├── 列表 Tab：历史日记卡片（分页）

  ├── 事件 Tab / 情绪 Tab / 消费 Tab / 地点 Tab / 标签 Tab

[profile.html] 个人中心
  ├── 统计卡片（记录天数/连续打卡/事件总数）
```

**关键 UI 交互**：
- `today.html`：Bottom Sheet（`sheetSlideUp` 动画，translateY 0→100%）用于查看编辑历史日期
- `today.html`：内联编辑面板（`todayInlinePanels`）用于当天日记编辑和补充对话
- 补充对话通过 `is_supplement: true` 标志允许已完成会话继续接收消息
- 所有操作结果通过 Toast 组件（右下角弹出，2.2s 自动消失）反馈给用户

### 3.2 数据库设计

#### 3.2.1 数据库总览

系统共 9 张表，均继承 `Base` 基类（含 `created_at / updated_at / deleted_at`，全局软删除机制）。

**三层实体架构**（源自 `docs/02_数据库总览与关系设计初稿.md`）：

| 层级 | 表 | 职责 |
|---|---|---|
| 身份与配置层 | `users`, `user_settings` | 账户管理、个性化配置 |
| 交互记录层 | `conversations`, `conversation_messages` | 每日 AI 对话的原始记录 |
| 沉淀复盘层 | `daily_records` + 五子表 | 结构化日记数据，是查询与分析的基础 |

#### 3.2.2 ER 关系图

```
users (1) ──────────────── (1) user_settings
  │
  ├── (1:N) conversations
  │              │
  │              ├── (1:N) conversation_messages
  │              │
  │              └── (1:1) daily_records ──── (1:N) record_events
  │                                      ├── (1:N) record_emotions
  │                                      ├── (1:N) record_expenses
  │                                      ├── (1:N) record_locations
  │                                      └── (1:N) record_tags
  │
  └── (1:N) daily_records（user_id 冗余索引，支持跨日查询）
```

#### 3.2.3 主要表结构说明

**`users` 表**

| 字段 | 类型 | 说明 |
|---|---|---|
| id | UUID(PK) | 主键，UUID v4 |
| phone | VARCHAR(20) | 主登录凭据，唯一约束（部分索引，deleted_at IS NULL）|
| email | VARCHAR(100) | 预留，MVP 不启用 |
| hashed_password | VARCHAR(255) | bcrypt 哈希，禁止明文 |
| nickname | VARCHAR(50) | 用户昵称 |
| status | VARCHAR(20) | `active` / `banned` |
| last_login_at | TIMESTAMPTZ | 最近登录时间 |

**`conversations` 表**

| 字段 | 类型 | 说明 |
|---|---|---|
| id | UUID(PK) | 主键 |
| user_id | UUID(FK→users) | 所属用户 |
| record_date | DATE | 用户本地自然日（依据时区换算）|
| status | VARCHAR(20) | `recording` / `completing` / `completed` |

> 联合唯一约束：`(user_id, record_date) WHERE deleted_at IS NULL`

**`conversation_messages` 表**

| 字段 | 类型 | 说明 |
|---|---|---|
| id | UUID(PK) | 主键 |
| conversation_id | UUID(FK→conversations) | 所属会话 |
| role | VARCHAR(20) | `user` / `ai` |
| content | TEXT | 消息正文 |
| sequence_number | INTEGER | 会话内有序序号（从1递增）|

> 联合唯一约束：`(conversation_id, sequence_number)`

**`daily_records` 表**（核心沉淀表）

| 字段 | 类型 | 说明 |
|---|---|---|
| id | UUID(PK) | 主键 |
| user_id | UUID(FK→users) | 所属用户 |
| conversation_id | UUID(FK, UNIQUE) | 关联会话（强 1:1）|
| record_date | DATE | 归属日期（冗余自 conversation）|
| body_text | TEXT | 日记正文（可被用户/AI重写的权威源）|
| summary_text | TEXT | 摘要（供列表展示）|
| emotion_overall_score | INTEGER | 综合情绪分 1-10 |
| keywords | JSONB | 关键词数组（无需深度过滤，适合JSONB）|
| user_note | TEXT | 用户私密备注 |

> 联合唯一约束：`(user_id, record_date) WHERE deleted_at IS NULL`

**五张结构化子表（共同模式）**

| 子表 | 特有字段 | 说明 |
|---|---|---|
| `record_events` | content(TEXT), source, is_user_confirmed | 当日事件 |
| `record_emotions` | emotion_label(VARCHAR), intensity(1-5), source | 情绪标注 |
| `record_expenses` | amount(NUMERIC 10,2), currency, category, description | 消费记录 |
| `record_locations` | name(VARCHAR 200), source | 地点信息 |
| `record_tags` | tag_name(VARCHAR 50), source | 标签（无 is_user_confirmed）|

> 所有子表均冗余 `user_id`（违反 3NF 但支持高效跨日聚合查询，是合理的反范式设计）
> 所有子表均有 `record_id`(FK→daily_records)、`source`(`ai`/`user`)、`deleted_at`

#### 3.2.4 数据库设计决策说明

| 决策 | 理由 |
|---|---|
| `keywords` 存 JSONB 而非独立表 | 关键词作为整体展示，无需单词级筛选，JSONB 降低建表冗余 |
| 五子表独立建表而非 JSONB | 支持跨日聚合（如"过去一月消费超 200 元的记录"），结构约束更严格 |
| `user_id` 冗余至子表 | 支持 `WHERE user_id=? AND record_date BETWEEN ? AND ?` 的跨日查询，无需 JOIN `daily_records` |
| `body_text` 独立于 `summary_text` | Path A 用户改正文时，AI 重建不会覆盖摘要；两者语义分离 |
| 软删除全局策略 | 保留审计链，支持越权操作检测，防止事务失败留下脏空洞 |

### 3.3 关键算法与核心技术

#### 3.3.1 AI 日记生成流程

```
会话消息列表（conversation_messages）
       │
       ▼
序列化为对话文本
[用户]: 今天下午去见了老朋友...
[AI助手]: 听起来是久违的重逢...
       │
       ▼
构造 Prompt
  System: 你是专业日记助手，必须输出指定 JSON 格式...
  User:   以下是今日对话记录，请提取 body_text/summary_text/
          emotion_overall_score/keywords/events/emotions/
          expenses/locations/tags
       │
       ▼
调用 TokenHub GLM-5
  temperature=0.3, timeout=90s
       │
  ┌────┴────┐
  │ 成功    │ 失败
  ▼         ▼
JSON 解析  Fallback 规则引擎
  ↓ 多格式兼容  ↓ 正则提取金额
  ↓ markdown代码块 ↓ 关键词匹配情绪
  ↓ 正则兜底    ↓ 截断句子生成事件
       │
       ▼
规范化（数值范围 clamp、去重、截断长度）
       │
       ▼
同一事务写入 daily_records + 五子表
（失败全量回滚，conversation.status 保持 completing）
       │
       ▼
更新 conversation.status → "completed"
```

#### 3.3.2 两条日记更新路径

**Path A（手动改正文）**：
```
用户提交新 body_text
→ 校验非空
→ 更新 daily_records.body_text
→ 软删旧五子表（含 source=user 的，保留用户手动加的 tags）
→ AI 从新 body_text 重提取五子表
→ 写入新子表记录
→ 派生更新 summary_text / emotion_overall_score / keywords
→ 同一事务提交，失败全量回滚
```

**Path B（AI 补充对话重建）**：
```
用户发送补充消息（is_supplement=true）
→ 跳过 completed 状态校验
→ 正常写入消息 + AI 回复

用户点击"保存本次补充"
→ 拉取会话全量消息历史
→ AI 重新生成完整 body_text + 五子表
→ 软删旧子表中 source=ai 的记录（保留用户确认过的条目）
→ 写入新提取数据
→ 同一事务提交
```

#### 3.3.3 连续打卡计算算法

```python
# 取用户近 365 天日记日期，按倒序排列
dates = [record_date₁, record_date₂, ...]  # 降序

streak = 0
expected = today  # 若最近一条不是今天，改为 yesterday

for d in dates:
    if d == expected:
        streak += 1
        expected = expected - 1day
    elif d < expected:
        break  # 日期中断，停止计数
```

#### 3.3.4 安全设计

| 安全措施 | 实现方式 |
|---|---|
| 密码安全 | `bcrypt` 哈希（单向，不可逆）|
| 令牌鉴权 | `python-jose` 签发 HS256 JWT，有效期 30 天 |
| 接口鉴权 | FastAPI `Depends(get_current_user)` 全局注入，不在路由函数内单独校验 |
| 越权防护 | 所有数据查询强制绑定 `WHERE user_id = current_user.id` |
| 手机号脱敏 | 返回时中间四位替换为 `****`（如 `138****8000`）|
| 数据库连接 | asyncpg + SSL（Neon Pooler），去除不兼容的 sslmode 参数 |
| 软删除隔离 | 所有查询默认 `WHERE deleted_at IS NULL`，防止软删数据泄露 |

### 6.2 克服的主要技术难点

**难点 1：Vercel + asyncpg + Neon 三方兼容性**

Neon PostgreSQL 的标准连接串携带 `sslmode=require` 参数，而 asyncpg 驱动不支持该 URL 参数形式（需通过代码配置 SSL context）。同时，pydantic-settings 在 Vercel Serverless 冷启动时无法稳定读取环境变量。

解决方案：在 `app/core/config.py` 的 `SQLALCHEMY_DATABASE_URI` 属性中，优先通过 `os.environ.get("DATABASE_URL")` 直接读取，再执行 URL 参数剥离（去除 `sslmode/ssl/channel_binding`），并通过 `connect_args` 传入 SSL 配置。

**难点 2：AI JSON 输出的健壮性解析**

GLM-5 模型有时将 JSON 包裹在 markdown 代码块中（` ```json ... ``` `），有时输出被截断，有时字段名大小写不一致。

解决方案：实现多格式兼容的解析链：先尝试直接 `json.loads()`，失败则用正则从代码块中提取，再失败则启动 Fallback 规则引擎（关键词匹配情绪、正则提取金额、句子截断生成事件）。

**难点 3：日记重建的事务一致性**

日记重建涉及：读取旧子表 → 批量软删 → AI 提取新数据 → 写入新子表 → 更新主表，共 5 个步骤。任何一步失败都可能留下数据不一致状态（旧记录被删但新记录未写入）。

解决方案：全部步骤在同一个 SQLAlchemy async session 事务内执行，任何异常触发 `rollback()`，`conversation.status` 保持不变（重建失败时保持 `completing`，前端可重试）。

**难点 4：补充对话与日记状态机冲突**

原有设计中 `completed` 状态的会话拒绝一切新消息（设计合理但需要为「补充场景」开口）。若直接放开 completed 状态会破坏原有逻辑。

解决方案：引入 `is_supplement` 标志而非新增状态值，服务层在检测到该标志时跳过状态校验，最大程度复用原有代码路径，避免引入状态机复杂度。
