# 《03_认证与用户模块》后端设计文档

## 1. 模块目标

本模块负责用户身份的建立、验证与基础信息管理，是整个系统所有受鉴权接口的前置依赖。

具体职责包括：
- 提供手机号（或邮箱）与密码的注册与登录能力
- 签发 JWT Token，作为后续所有受保护接口的鉴权凭据
- 提供当前用户基础信息的查询与编辑能力
- 维护账户状态（正常、封禁）

本模块不包含用户偏好配置（如时区、AI 伴侣设定），该部分归属于设置模块（`user_settings`）。

---

## 2. 涉及的数据表

本模块直接操作以下数据表：

| 表名 | 读/写 | 说明 |
|------|-------|------|
| `users` | 读写 | 账户主表，负责注册写入与登录查询 |
| `user_settings` | 写（新用户注册时初始化） | 建议在注册成功后同步初始化默认 `user_settings`；若初始化失败，整笔注册事务回滚。 |

---

## 3. `users` 表结构建议（设计说明）

以下为字段级别的设计说明，不含 SQL 或 ORM 代码。

| 字段名 | 类型建议 | 说明 |
|--------|----------|------|
| `id` | UUID | 主键，使用 UUID v4，不对外暴露自增序列 |
| `phone` | VARCHAR(20) | 主要登录凭据，建议设唯一约束；MVP 阶段注册必填，不允许省略 |
| `email` | VARCHAR(100) | 备用登录凭据，允许为空；设唯一约束（预留字段，MVP 阶段不启用） |
| `hashed_password` | VARCHAR(255) | 使用 bcrypt 哈希后的密码，禁止明文存储 |
| `nickname` | VARCHAR(50) | 用户昵称，初始可由系统随机生成；AI 在对话中使用此称谓 |
| `avatar_url` | VARCHAR(255) | 用户头像的 OSS 存储路径；MVP 阶段可为空，默认使用系统占位图 |
| `status` | VARCHAR(20) | 账户状态。落库及 API 返回值统一为小写字符串：`active`（正常）、`banned`（封禁）。代码层枚举成员名可使用大写（如 `UserStatus.ACTIVE`），但实际写入值与接口返回值均为小写。 |
| `last_login_at` | TIMESTAMPTZ | 最近一次登录时间，每次登录成功时更新 |
| `created_at` | TIMESTAMPTZ | 注册时间，系统自动写入 |
| `updated_at` | TIMESTAMPTZ | 最近更新时间，系统自动维护 |
| `deleted_at` | TIMESTAMPTZ | 软删除标记，默认为 NULL |

**设计权衡说明：**

> **关于 `phone` 必填与 `email` 扩展字段的边界说明**  
> MVP 阶段注册接口要求 `phone` 字段必填，不允许两者同时为空。`email` 字段保留仅为后续扩展预留结构位，当前注册流程不启用。接口参数验证层须在入参阶段强制校验 `phone` 的存在性，不依赖数据库约束兜底。

> **关于 `phone` 与 `email` 字段均允许为空的数据库层设计**  
> 原因：MVP 阶段以手机号登录为主，但后续可能引入邮箱登录。若两个字段均设为 NOT NULL，则后续兼容扩展时需要大量历史数据补填。  
> 建议：两个字段在数据库层均允许为空，但在应用层约束"至少有一项不为空"，同时各自加唯一约束（部分索引：`WHERE deleted_at IS NULL`），避免软删除影响唯一性判定。

| 字段 | 位置 | 说明 |
|------|------|------|
| `hashed_password` | `users` 表 | 采用 bcrypt 算法加密，不可逆 |
| `status` | `users` 表 | 登录时须校验该字段，`banned` 状态应拒绝签发 Token |
| `last_login_at` | `users` 表 | 每次成功登录后更新，可用于后续会话异常检测 |

## 5. 接口清单

| 序号 | 方法 | 路径 | 是否需要鉴权 | 说明 |
|------|------|------|--------------|------|
| 01 | POST | `/api/v1/auth/register` | 否 | 注册新账户 |
| 02 | POST | `/api/v1/auth/login` | 否 | 登录并获取 Token |
| 03 | GET | `/api/v1/users/me` | 是 | 获取当前用户基础信息 |
| 04 | PUT | `/api/v1/users/me` | 是 | 修改当前用户基础信息（昵称、头像） |

> 说明：注册与登录逻辑分离为两个接口，而非合并为"登录或注册"的隐式自动注册模式。原因是隐式注册模式会导致前端无法区分当前是首次注册还是重复登录，无法展示新手引导流程。若后续产品要求合并，可在不破坏接口结构的前提下调整业务逻辑。

## 6. 接口详情

---

### 6.1 注册

**路径**：`POST /api/v1/auth/register`  
**鉴权**：不需要

#### 请求参数（Body JSON）

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `phone` | string | **是** | 手机号，11 位纯数字。MVP 阶段注册必填，不允许省略。 |
| `password` | string | 是 | 原始密码，长度 8~50 位，后端加密存储 |
| `nickname` | string | 否 | 昵称；不传则系统随机生成 |

> **说明**：`email` 字段在注册接口中不开放，当前版本不接受该参数。

#### 请求示例

```json
{
  "phone": "13800138000",
  "password": "MyPassword123",
  "nickname": "云淡"
}
```

#### 返回参数

| 字段 | 类型 | 说明 |
|------|------|------|
| `user_id` | string (UUID) | 注册成功后生成的用户 ID |
| `nickname` | string | 最终使用的昵称（系统生成或用户传入） |

#### 返回示例（成功）

```json
{
  "code": 20000,
  "message": "success",
  "data": {
    "user_id": "e7a1f3c2-84b0-4d1e-9a23-1c5d8e2f7b90",
    "nickname": "云淡"
  },
  "trace_id": "req-reg-001"
}
```

#### 错误情况

| HTTP 状态码 | 内部 code | 说明 |
|-------------|-----------|------|
| 400 | 40001 | 手机号格式错误或密码不符合规则 |
| 409 | 40901 | 手机号已被注册 |

---

### 6.2 登录

**路径**：`POST /api/v1/auth/login`  
**鉴权**：不需要

#### 请求参数（Body JSON）

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `phone` | string | 是 | 注册时使用的手机号 |
| `password` | string | 是 | 原始密码 |

#### 请求示例

```json
{
  "phone": "13800138000",
  "password": "MyPassword123"
}
```

#### 返回参数

| 字段 | 类型 | 说明 |
|------|------|------|
| `access_token` | string | JWT Token，后续所有受保护接口均需携带 |
| `token_type` | string | 固定值 `"bearer"` |
| `expires_in` | integer | Token 有效期，单位：秒 |

#### 返回示例（成功）

```json
{
  "code": 20000,
  "message": "success",
  "data": {
    "access_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
    "token_type": "bearer",
    "expires_in": 2592000
  },
  "trace_id": "req-login-001"
}
```

#### 错误情况

| HTTP 状态码 | 内部 code | 说明 |
|-------------|-----------|------|
| 400 | 40001 | 请求参数缺失或格式错误 |
| 401 | 40102 | 手机号或密码错误 |
| 403 | 40301 | 账户处于 `banned` 状态，拒绝登录 |

---

### 6.3 获取当前用户信息

**路径**：`GET /api/v1/users/me`  
**鉴权**：需要（`Authorization: Bearer <token>`）

#### 请求参数

无请求体。用户身份从 Token 解析获取，不接受通过 URL 或 Body 传入 `user_id`。

#### 返回参数

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | string (UUID) | 用户唯一 ID |
| `phone` | string | 手机号（脱敏，中间四位替换为 `****`） |
| `nickname` | string | 当前昵称 |
| `avatar_url` | string \| null | 头像地址，无头像时为 null |
| `status` | string | 账户状态，返回值为小写：`active` 或 `banned` |
| `created_at` | string (ISO 8601) | 注册时间 |
| `last_login_at` | string \| null | 最近登录时间 |

#### 返回示例（成功）

```json
{
  "code": 20000,
  "message": "success",
  "data": {
    "id": "e7a1f3c2-84b0-4d1e-9a23-1c5d8e2f7b90",
    "phone": "138****8000",
    "nickname": "云淡",
    "avatar_url": null,
    "status": "active",
    "created_at": "2024-05-20T10:30:00Z",
    "last_login_at": "2024-06-01T08:15:22Z"
  },
  "trace_id": "req-me-001"
}
```

#### 错误情况

| HTTP 状态码 | 内部 code | 说明 |
|-------------|-----------|------|
| 401 | 40101 | Token 未携带 |
| 401 | 40102 | Token 已过期或无效 |

---

### 6.4 修改当前用户信息

**路径**：`PUT /api/v1/users/me`  
**鉴权**：需要（`Authorization: Bearer <token>`）

> 使用 `PUT` 语义，本接口仅支持对 `nickname` 和 `avatar_url` 两个字段的覆盖更新。调用时若某字段不传，则该字段保持不变；若显式传入 `null`，则视为清空该字段（`avatar_url` 适用此逻辑，`nickname` 不允许为 null）。

#### 请求参数（Body JSON，全部可选）

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `nickname` | string | 否 | 新昵称，长度 1~50 字符，不允许为 null |
| `avatar_url` | string | 否 | 新头像地址；头像文件上传逻辑由独立的文件上传接口处理，此处只接收已上传后的 URL；可传 null 以清空头像 |

#### 请求示例

```json
{
  "nickname": "霜降"
}
```

#### 返回参数

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | string (UUID) | 用户 ID |
| `nickname` | string | 更新后的昵称 |
| `avatar_url` | string \| null | 更新后的头像地址 |
| `updated_at` | string (ISO 8601) | 本次更新时间 |

#### 返回示例（成功）

```json
{
  "code": 20000,
  "message": "success",
  "data": {
    "id": "e7a1f3c2-84b0-4d1e-9a23-1c5d8e2f7b90",
    "nickname": "霜降",
    "avatar_url": null,
    "updated_at": "2024-06-02T09:00:00Z"
  },
  "trace_id": "req-update-001"
}
```

#### 错误情况

| HTTP 状态码 | 内部 code | 说明 |
|-------------|-----------|------|
| 400 | 40001 | 字段格式不合规（如昵称超长或为 null） |
| 401 | 40102 | Token 无效或过期 |

## 7. Token 方案建议

**方案**：采用 JWT（JSON Web Token），签发后存储于客户端，服务端无状态验证。

| 参数 | 建议值 | 说明 |
|------|--------|------|
| 算法 | HS256 | 对称签名算法，适合单体服务 |
| 有效期 | 30 天 | MVP 阶段，减少用户频繁重新登录的摩擦感 |
| Payload 内容 | `user_id`、`exp`（过期时间） | 不在 Token 中存储敏感信息（如密码、手机号） |
| 携带方式 | `Authorization: Bearer <token>` | 标准 HTTP 头部，全局统一 |
| 验证位置 | FastAPI 全局依赖注入（`Depends`） | 不在各路由函数内部单独验证 |

**设计权衡说明：**

> **关于是否引入 Refresh Token 双 Token 机制**  
> 建议：MVP 阶段不引入。双 Token 机制需要维护 Refresh Token 的服务端存储（Redis 或数据库），以及 Token 轮换的完整逻辑，开发成本较高。  
> 原因：30 天的 Access Token 有效期在 C 端产品中已足够覆盖大部分用户的正常使用周期。若后续有安全审计要求，再引入 Token 吊销机制（黑名单策略）即可。

## 8. 用户信息接口设计建议

1. **身份来源唯一化**：所有需要当前用户身份的接口，`user_id` 必须且只能从 JWT Token 解析获取，禁止通过 URL 路径（如 `/api/v1/users/{user_id}`）或请求体传入用户 ID。这是防止越权访问（IDOR）的基本措施。

2. **手机号脱敏**：`GET /api/v1/users/me` 返回的手机号须在应用层进行中间四位脱敏处理（如 `138****8000`），不返回完整手机号。

3. **密码修改接口独立**：本文档中不包含修改密码的接口，建议将其作为独立接口（如 `POST /api/v1/auth/change-password`）在后续安全模块中设计，避免与基础资料编辑接口耦合。

4. **软删除用户**：当某账户执行注销操作时，对 `users` 表的 `deleted_at` 字段写入当前时间即可，不做物理删除；对应的该手机号可通过部分索引（`WHERE deleted_at IS NULL`）的唯一约束允许重新注册。

## 11. 风险点与实现建议

### 风险 1：密码明文传输
- **风险描述**：前端若通过 HTTP 明文传输原始密码，存在中间人攻击风险。
- **建议**：生产环境必须启用 HTTPS。开发测试阶段可暂时豁免，但部署前须强制落地。

### 风险 2：弱密码策略缺失
- **风险描述**：若不设置最低密码复杂度要求，用户可能设置极简密码（如 `12345678`），导致账户安全性极低。
- **建议**：在接口参数验证层设置密码规则（如：长度至少 8 位，须包含字母与数字的组合），通过 Pydantic 的字段校验器在入参阶段拦截。

### 风险 3：注册接口被批量调用（撞库/注册滥用）
- **风险描述**：注册接口无鉴权保护，可能被脚本批量请求，消耗服务器资源或制造大量无效账户。
- **建议**：MVP 阶段可通过 IP 频率限制（Rate Limiting）中间件做基础防护，后续可升级为短信验证码校验。

### 风险 4：Token 无法主动失效
- **风险描述**：JWT 的无状态特性意味着一旦签发，在过期前无法主动吊销（如用户修改密码后，旧 Token 仍然有效）。
- **建议**：MVP 阶段可接受此限制，但需在文档中记录此已知约束。若后续出现账户安全事件，可通过引入 Token 黑名单（存储于 Redis）的方式补救，无需修改接口结构。

### 风险 5：账户状态校验遗漏
- **风险描述**：若 Token 验证中间件只校验签名有效性而不检查 `users.status`，则被封禁的用户仍可凭旧 Token 访问受保护接口。
- **建议**：在鉴权依赖注入逻辑中，完成 Token 签名验证后，须额外查询数据库确认当前用户的 `status` 字段值为 `active`，否则返回 `403`。代码层可使用枚举常量（如 `UserStatus.ACTIVE`）进行比对，但比对目标的实际值应为小写字符串 `"active"`，与落库值保持一致。

# 《04_用户设置模块》后端设计文档

> 本文档为「回响（Echo）」MVP 阶段的用户设置模块设计文档。  
> 关联文档：`docs/01_后端全局约束与命名规范.md`、`docs/02_数据库总览与关系设计初稿.md`、`docs/03_认证与用户模块.md`  
> 本文档不包含代码、ORM、SQL，仅输出接口设计与模块规范。

---

## 1. 模块目标

本模块负责管理用户的个人化配置项，是聊天行为、推送提醒与 AI 人格注入的配置中枢。

具体职责包括：
- 提供当前用户设置的读取能力
- 提供当前用户设置的更新能力（覆盖式局部更新）
- 管理每日提醒开关与提醒时间的配置
- 管理用户输入偏好（文字 / 语音 / 混合）
- 提供用户主动发起的"删除本人全部数据"入口

本模块不包含用户基础信息（如昵称、头像）的管理，该部分归属于认证与用户模块（`users` 表）。

---

## 2. 涉及的数据表

本模块直接操作以下数据表：

| 表名 | 读/写 | 说明 |
|------|-------|------|
| `user_settings` | 读写 | 用户个性化配置主表，本模块的核心操作对象 |
| `users` | 读（鉴权） / 写（删除数据时同步软删用户行） | 通过 Token 解析 `user_id`；执行数据清除时同步操作 |

## 4. 与 `users` 表的关系

`user_settings` 与 `users` 之间为 **1:1 关系**，通过 `user_settings.user_id` 外键向 `users.id` 建立关联。

| 约束 | 建议 |
|------|------|
| 外键约束 | `fk_user_settings_user_id_users`，引用 `users.id` |
| 唯一约束 | `uq_user_settings_user_id`，保证一个用户最多只有一条配置记录 |
| 初始化时机 | 用户注册成功后，在同一事务内同步写入一条默认 `user_settings`；若写入失败，整笔注册事务回滚 |
| 数据清除一致性 | 执行删除用户数据时，`user_settings` 记录**不执行软删除**，而是在同一事务内将所有可配置字段重置为默认值，保持记录行存续；其余业务数据表的软删除与该重置操作须在同一事务内完成 |

> **说明**：`user_settings` 的分离建表理由已在 `docs/02_数据库总览与关系设计初稿.md` 第 3.2 节阐明，此处不重复论述。

### 6.2 更新当前用户设置

**路径**：`PUT /api/v1/settings`  
**鉴权**：需要（`Authorization: Bearer <token>`）

> `reminder_enabled` 与 `input_preference` 不允许显式传入 `null`，必须提供合法值。  
> **提醒规则（已定稿）**：若请求中 `reminder_enabled` 为 `true`，则 `reminder_time` 必须同时提供合法的 `HH:MM` 值；不满足此条件时接口返回 `400`。

## 7. 字段设计建议

### 7.1 `reminder_enabled` 命名说明

此处沿用既定字段名 `reminder_enabled`，不再改名。  
**说明**：《01_后端全局约束与命名规范》第 10 节建议布尔值字段以 `is_` 或 `has_` 开头；本项目在早期设计中已使用 `reminder_enabled` 作为约定字段名，为保持命名一致性，本模块不执行重命名，后续其他模块引用此字段时应与此保持一致。

---

### 7.2 `reminder_time` 字段设计建议

**建议**：使用数据库 `TIME` 类型存储，格式为 `HH:MM`（24 小时制），不携带日期信息，不携带时区。  
**原因**：提醒时间本质上是一个"本地墙钟时刻"，所对应的实际 UTC 时间需结合 `user_settings.timezone` 在推送任务触发时动态计算。若将其存储为带时区的 `TIMESTAMPTZ`，则每次用户更改时区后都需要联动更新提醒时间字段，维护成本较高。

**关联逻辑**：推送系统在生成每日提醒任务时，从 `user_settings` 同时读取 `reminder_time` 与 `timezone`，在计算层将其转换为实际的 UTC 触发时刻，不在 `user_settings` 中存储预计算结果。

**约束规则（已定稿）**：`reminder_time` 仅在 `reminder_enabled` 为 `false` 时允许为 `NULL`。当 `reminder_enabled` 为 `true` 时，`reminder_time` 必须为合法的 `HH:MM` 值；接口层在业务校验阶段强制拦截不满足此条件的请求，返回 `400`。

---

### 7.3 `input_preference` 字段设计建议

**建议**：枚举值落库及接口返回均使用小写字符串：`text` / `voice` / `mixed`；代码层枚举成员名使用全大写（如 `InputPreference.TEXT`）。  
**原因**：与《01_后端全局约束与命名规范》第 11 节枚举值规范保持一致；同时与 `users.status` 字段的大小写处理策略对齐，降低接口消费方的理解成本。

**设计权衡**：

> **建议**：不使用数据库原生 `ENUM` 类型，以定长 `VARCHAR(20)` 存储。  
> **原因**：若后续新增输入偏好类型（如 `image`），`VARCHAR` 方案只需在应用层调整枚举定义，无需执行 `ALTER TYPE`；数据库 `ENUM` 类型的新增选项需要对整个类型执行 DDL 修改，在生产环境中存在锁表风险。

---

### 7.4 `timezone` 字段设计建议

**建议**：使用 IANA 时区字符串（如 `Asia/Shanghai`、`America/New_York`），而非 UTC 偏移量（如 `+08:00`）。  
**原因**：UTC 偏移量在夏令时地区会随季节变化，导致同一地区的用户在夏令时开始/结束时需要主动更新配置。IANA 时区标识已内置夏令时规则，可在应用层（依赖 `pytz` 或 `zoneinfo`）自动处理偏移量计算。

**应用层校验**：`timezone` 字段写入前，应在应用层校验字符串是否为合法 IANA 时区标识，不合法时返回 `400`。

## 8. 删除用户数据接口的处理建议

### 8.1 操作范围

本接口执行的是**业务数据清除**，不注销用户账户。执行完成后：
- `users` 表中的用户行保持不变（`deleted_at` 仍为 `NULL`，用户仍可登录）
- `user_settings` 记录行保持不变，所有可配置字段重置为注册时的默认值（`timezone` 恢复 `Asia/Shanghai`、`reminder_enabled` 恢复 `false`、`reminder_time` 恢复 `NULL`、`input_preference` 恢复 `text`、`ai_companion_name` 恢复 `NULL`）
- 第 2 节所列全部业务数据表（`conversations`、`conversation_messages`、`media_files`、`daily_records`、`record_*` 系列、`weekly_reports`）中与该 `user_id` 关联的所有行，`deleted_at` 字段统一写入当前 UTC 时间戳

### 8.2 事务边界

`user_settings` 字段重置与全部关联业务表的软删除操作须在**单一数据库事务**内完成。任意操作失败（如锁等待超时、约束冲突）应触发整体回滚，向调用方返回 `500`，不允许出现部分清除的中间状态。

### 8.3 二次确认机制

二次确认为**前端交互责任**，不体现在接口请求参数中。前端须在 UI 层完成用户的明确二次确认（如弹窗提示）后，方可向后端发起本接口的调用。后端接口不接收任何确认字段，不依赖请求体做确认校验。

### 8.4 OSS 文件的处理

`media_files` 表的软删除操作仅标记元数据记录，不触发对象存储（OSS）上实际文件的物理删除。物理文件的清理应由独立的后台清理任务（如定期扫描 `deleted_at` 非空的 `media_files` 记录并调用 OSS 删除接口）处理，不在本次接口的同步链路内执行，以避免拖慢接口响应时间。

### 8.5 日志审计

本接口属于高危操作，建议在服务器日志中记录以下信息（不含敏感数据）：操作时间、`user_id`、请求 `trace_id`、各表软删除行数。

## 11. 风险点与实现建议

### 风险 1：`user_settings` 记录缺失

- **风险描述**：若注册流程中 `user_settings` 初始化失败且事务未正确回滚，可能导致 `GET /api/v1/settings` 返回 404，影响聊天模块正常启动。
- **建议**：在注册事务中强制同步初始化 `user_settings`，确保两个写入操作处于同一数据库事务。同时在 `GET /api/v1/settings` 接口中加入兜底逻辑：若查询结果为空，自动为当前用户创建默认配置记录并返回，不直接返回 404。

---

### 风险 2：`timezone` 非法值写入

- **风险描述**：若应用层未对 `timezone` 字段的合法性进行校验，写入非法 IANA 时区字符串（如 `UTC+8`、`China Standard Time`）后，时区相关计算（归属日期判定、提醒触发时刻计算）将产生预期外的错误。
- **建议**：在 `PUT /api/v1/settings` 接口的入参校验阶段（Pydantic 验证器层），调用标准时区库（如 `zoneinfo` 或 `pytz`）对传入的时区字符串进行合法性验证，不合法则返回 `400`，不进入数据库写入流程。

---

### 风险 3：`reminder_enabled` 为 `true` 但 `reminder_time` 为 `NULL`

- **风险描述**：若接口入参校验不完整，可能出现 `reminder_enabled=true` 与 `reminder_time=NULL` 同时落库的情况，推送系统读取时触发计算错误。
- **建议**：接口层在业务校验阶段强制拦截此组合（见 6.2 节错误码 `40002`），不允许该状态落库。推送任务调度层同时加入防御性判断作为双重保障：若遇此状态则跳过并记录日志，不抛出异常。

---

### 风险 4：数据清除事务过大导致超时

- **风险描述**：若某用户的历史数据量极大（如数百条 `conversation_messages`），单一事务中覆盖所有关联表的软删除操作可能触发事务超时。
- **建议**：MVP 阶段用户数据量有限，单事务方案可接受。若后期出现超时问题，可将清除操作拆分为：首先在事务内完成 `user_settings` 重置（阻断新数据写入），其余关联表的软删除通过异步后台任务分批处理。但此方案引入了中间状态，需要配套补偿机制，MVP 阶段不建议提前引入。

---

### 风险 5：输入偏好与实际可用功能不一致

- **风险描述**：若用户将 `input_preference` 设置为 `voice`，但语音输入功能在当前版本尚未上线，可能导致前端出现功能不可用但配置已保存的状态矛盾。
- **建议**：MVP 初期若语音功能未上线，建议在前端层面限制 `voice` 和 `mixed` 选项的可选性；后端接口层不对此做额外拦截，正常存储即可，不引入版本标记字段。该字段已预留，待功能上线后前端同步开放。

## 1. 模块目标

本模块负责用户与 AI 之间每日对话的全生命周期管理，是 MVP 核心闭环的主体执行层。

具体职责包括：
- 管理每日会话的创建与状态流转
- 承载用户消息与 AI 回复的双向消息写入
- 提供会话消息的有序列表查询能力
- 接收用户发出的"结束今日记录"指令，并触发后续总结生成流程

本模块不包含：
- AI 总结内容的生成逻辑（由总结生成任务异步执行，归属于日记录模块的写入侧）
- 历史记录的查看（归属于历史模块）
- 周报生成（归属于周报模块）
- 媒体文件的上传与 ASR 转写（归属于媒体文件模块）

## 5. 会话状态设计建议

`conversations.status` 字段定义会话的生命周期阶段。落库及接口返回值均使用小写字符串，代码层枚举成员名使用大写。

| 状态值（落库/返回） | 枚举成员名 | 含义 |
|---------------------|-----------|------|
| `recording` | `ConversationStatus.RECORDING` | 记录进行中，接受用户输入与 AI 回复 |
| `completing` | `ConversationStatus.COMPLETING` | 用户已触发结束，AI 总结生成任务正在异步执行；此状态下拒绝新消息写入 |
| `completed` | `ConversationStatus.COMPLETED` | AI 总结已生成完毕，`daily_records` 已写入；会话生命周期结束 |

**状态流转路径：**

```
recording  →（用户调用结束接口）→  completing  →（总结任务完成）→  completed
```

**约束说明：**
- 仅 `recording` 状态下允许接受 `POST /messages` 请求；其他状态下返回 `409`。
- `completing` → `completed` 的状态推进由异步任务（总结生成服务）完成，不由本模块的接口直接触发。
- 不存在从 `completed` 回退到 `recording` 的路径；若用户在已完成的当日再次进入，前端应直接读取历史记录，不创建新会话。

**设计权衡：**

> **关于是否增加 `failed` 状态**  
> **建议**：MVP 阶段不引入 `failed` 状态。  
> **原因**：总结生成失败时可通过重试机制在 `completing` 状态下重复触发，无需引入独立失败状态。后续若需要精细化的任务状态管控，可在不破坏当前接口结构的前提下扩展新状态。

---

## 6. 消息顺序字段设计建议

`conversation_messages.sequence_number` 为会话内消息的有序序号，从 `1` 开始单调递增。

**设计选择：使用 `sequence_number` 而非单纯依赖 `created_at` 排序**

| 对比维度 | `sequence_number` | 仅用 `created_at` |
|----------|-------------------|-------------------|
| 排序稳定性 | 确定，业务层保证单调递增 | 受服务器时钟精度影响，毫秒级并发可能出现相同时间戳 |
| 并发写入 | 需要在业务层保证序号分配的原子性（见风险 4） | 无需额外保证 |
| AI 消息插入 | 可明确表达"AI 回复紧跟用户消息之后"的位置关系 | 依赖时间先后，AI 响应写入时间难以精确紧跟 |
| 消息计数 | 直接从序号推断当前会话消息总数 | 需额外 COUNT 查询 |

**分配规则**：
- 用户消息写入时分配下一个可用序号；AI 回复消息紧跟用户消息序号 +1 写入。
- 同一次"发送消息"请求中，用户消息序号与 AI 回复消息序号应在同一业务操作中连续分配，不允许中间插入其他序号。
- 序号的实际分配方式（如数据库行锁、原子递增等）属于实现细节，不在本文档中展开。

## 7. 接口清单

| 序号 | 方法 | 路径 | 是否需要鉴权 | 说明 |
|------|------|------|--------------|------|
| 01 | GET | `/api/v1/conversations/today` | 是 | 获取今天的会话状态；若今日尚无会话，返回空状态而非 404 |
| 02 | POST | `/api/v1/conversations` | 是 | 创建今天的会话；今日会话已存在时返回 409 |

| 05 | POST | `/api/v1/conversations/{conversation_id}/complete` | 是 | 结束今日记录，触发 AI 总结生成异步任务 |

> **路径设计说明**：`/conversations/today` 作为具名子资源路径，语义上明确表示"今日会话"这一业务实体，优于通过 query 参数过滤（如 `?date=today`）。注意路由注册顺序须确保 `/today` 在 `/{conversation_id}` 之前匹配，防止字符串 `today` 被误解析为 UUID。

### 8.1 获取今日会话状态

**路径**：`GET /api/v1/conversations/today`  
**鉴权**：需要（`Authorization: Bearer <token>`）

#### 请求参数

### 8.4 发送用户消息并获取 AI 回复

**路径**：`POST /api/v1/conversations/{conversation_id}/messages`  
**鉴权**：需要（`Authorization: Bearer <token>`）

> 本接口采用同步响应模式，详见第 9 节设计建议。  
> 仅 `status` 为 `recording` 的会话接受本接口调用；`completing` 或 `completed` 状态下返回 `409`。  
> 接口须校验 `conversation_id` 归属当前用户。

| 参数名 | 类型 | 说明 |

| `conversation_id` | string (UUID) | 目标会话 ID |

| 字段 | 类型 | 必填 | 说明 |

## 9. 发送消息并同步返回 AI 回复的接口设计建议

### 9.1 同步模式的实现路径

本模块 MVP 阶段采用**同步响应模式**：

1. 后端接收用户消息，写入 `conversation_messages`（`role=user`）
2. 从 `conversations` 加载当前会话的消息历史，构建 LLM 请求上下文
3. 调用 AI 服务，等待模型返回回复内容
4. 将 AI 回复写入 `conversation_messages`（`role=ai`）
5. 将用户消息与 AI 回复作为整体一次性返回给前端

**优点**：前端实现简单，无需轮询或 WebSocket；接口状态明确，成功即表示 AI 已完成回复。  
**缺点**：接口响应时间受 LLM 调用延迟影响，通常在 2~10 秒；需设置合理超时时间（建议 30 秒）。

### 9.2 超时处理建议

**设计权衡：**

> **建议**：将后端 LLM 调用的超时时间设置为 30 秒，接口整体超时时间设置为 35 秒，并向调用方返回专用错误码（内部 code `50401`）。  
> **原因**：同步模式下若不设置明确的超时时间，网络波动或模型负载高峰期可能导致请求长时间挂起，占用服务器连接资源。前端应在收到超时响应后提示用户重试，不允许前端自行重传相同消息（重传应由用户主动发起，以避免重复写入）。

### 9.3 关于流式响应（Streaming）的扩展说明

**设计权衡：**

> **建议**：MVP 阶段不引入流式（Server-Sent Events / WebSocket）响应。  
> **原因**：流式响应需要前端处理增量文本拼接、断线重连、消息边界判定等复杂逻辑，同时需要后端处理流式写入与错误中断的补偿策略。MVP 阶段产品验证优先，同步接口在功能闭环上完全满足需求。若后续用户反馈等待体验差，可在接口层以相同路径引入 `Accept: text/event-stream` 的内容协商机制，不需要变更接口路径设计。

### 9.4 LLM 上下文构建建议

后端在调用 AI 服务前，需从 `conversation_messages` 中读取当前会话的历史消息作为上下文。

**设计权衡：**

> **当前实现**：按 `sequence_number` 升序读取当前会话的全部消息作为上下文，不再按条数截断。  
> **原因**：日记记录场景需要 AI 记住本次会话完整脉络，避免早期信息在后续追问或总结前被遗漏。

## 10. 「结束今日记录」接口应触发的动作

`POST /api/v1/conversations/{conversation_id}/complete` 接口的执行链路如下：

### 10.1 接口同步执行（在接口响应前完成）

1. **校验会话归属与状态**：确认 `conversation_id` 属于当前用户，且 `status` 为 `recording`
2. **状态流转**：将 `conversations.status` 由 `recording` 更新为 `completing`，同步更新 `updated_at`
3. **返回响应**：向调用方返回含 `status: "completing"` 的成功响应

### 10.2 接口异步触发（后台任务，不阻塞接口响应）

4. **触发总结生成任务**：向后台任务队列投递一条包含 `conversation_id` 的总结生成请求

> 任务执行内容属于日记录模块职责，本模块只负责投递任务，不关心任务内部实现。任务完成后，后台任务负责：将 `conversations.status` 推进至 `completed`；向 `daily_records` 写入 AI 生成的总结内容及结构化子表数据。

**设计权衡：**

> **关于是否在接口中同步等待总结生成完成**  
> **建议**：MVP 阶段异步触发，接口立即返回 `completing` 状态，不等待总结完成。  
> **原因**：总结生成需要读取全量对话消息并调用 LLM，耗时通常在 5~30 秒，不适合在接口链路中同步等待。前端可通过轮询 `GET /api/v1/conversations/today` 的 `status` 字段感知总结是否完成。

### 10.3 对消息写入的影响

接口成功返回后，`conversations.status` 已变更为 `completing`，此后所有发送消息的请求（`POST /messages`）均应返回 `409`，不再接受新消息写入。

## 13. 风险点与实现建议

### 风险 1：用户时区配置缺失或非法

- **风险描述**：若 `user_settings.timezone` 缺失或为非法 IANA 时区字符串，系统无法正确换算用户当地自然日，造成 `record_date` 写入错误数据。
- **建议**：不允许在时区字段缺失或非法时静默降级为默认值后继续写入。应立即终止当前操作，向调用方返回 `500`（内部 code `50002`），并记录高优先级告警供运维排查。时区字段的合法性保障应在 `PUT /api/v1/settings` 的入参校验阶段前置拦截（见用户设置模块），本模块以强依赖方式处理此异常，不做兜底。

---

### 风险 2：同一用户同一自然日创建重复会话

- **风险描述**：前端在网络超时场景下可能重复调用 `POST /api/v1/conversations`，导致尝试写入重复会话记录。
- **建议**：在数据库层对 `(user_id, record_date)` 建立联合唯一约束（部分索引：`WHERE deleted_at IS NULL`），当应用层捕获到约束冲突异常时，将其转化为 `409` 响应返回；不允许出现重复会话记录落库的情况。

---

### 风险 3：LLM 调用超时或失败时的消息状态一致性

- **风险描述**：用户消息已写入数据库后，若 AI 调用超时或报错，导致 AI 回复消息未落库，前端将收到错误响应，但用户消息已存在于数据库中，造成"用户消息孤立无 AI 回复"的不一致状态。
- **建议**：用户消息写入与 AI 回复写入须在整体流程上设计补偿逻辑。MVP 阶段建议：用户消息先写入，AI 调用失败时不回滚用户消息（保留原始记录），但在接口响应中返回错误；前端收到错误后提示用户重试，重试时由用户重新发送消息内容，后端将其作为新消息顺序写入，不尝试覆盖已存在的孤立消息。孤立的用户消息不影响摘要生成（总结任务可识别无 AI 回复的消息行并跳过或处理）。

---

### 风险 4：`sequence_number` 并发分配冲突

- **风险描述**：若多设备或网络重试同时对同一会话发送消息，`sequence_number` 的并发分配可能产生冲突，触发 `uq_conversation_messages_conversation_id_sequence_number` 唯一约束违反。
- **建议**：`sequence_number` 的分配应在数据库层以原子操作保障（如使用 `SELECT MAX(sequence_number) FOR UPDATE` 行级锁后递增），不在应用层并发计算后写入。当唯一约束冲突发生时，应用层捕获异常后可选择自动重试一次，重试仍失败时返回 `500`。

---

### 风险 5：结束记录后总结生成任务失败

- **风险描述**：异步总结生成任务在执行中可能失败（LLM 服务不可用、Token 超限等），导致 `conversations.status` 长期停留在 `completing`，不推进至 `completed`。
- **建议**：后台任务须实现幂等重试机制，失败后按退避策略最多重试 3 次。若全部重试均失败，仍将状态保持为 `completing`（不回退至 `recording`），并通过告警通道通知运维介入。前端可对 `completing` 状态设置超时提示（如超过 5 分钟仍未变为 `completed` 时展示"生成中，请稍后查看"），不将此状态视为用户侧操作失败。

---

### 风险 6：`/conversations/today` 路由与 `/{conversation_id}` 的路径冲突

- **风险描述**：若路由注册顺序不当，字符串 `today` 可能被框架误匹配为 `{conversation_id}` 的 UUID 路径参数，导致 `today` 请求进入错误的处理函数并因 UUID 解析失败返回 `400` 或 `422`。
- **建议**：在 FastAPI 路由注册时，确保 `/conversations/today` 的路由定义在 `/conversations/{conversation_id}` 之前注册；同时在 `{conversation_id}` 路径的处理函数中加入 UUID 格式的入参校验，非 UUID 格式的请求在入参层即返回 `422`，不进入业务逻辑。

# 《06_历史记录模块》后端设计与开发规范/实施清单

## 一、 模块目标
* **角色定位**：历史记录模块在 Echo 后端中承担着“用户记忆检索中枢”的角色，负责结构化、高性能地呈现用户过往累积的日记录数据。
* **独立性**：它绝非 `today record` 的附属接口，因为 `today` 关注的是**当天数据的生成、增量修改与状态维护**；而 `history` 关注的是**跨周期的数据只读、复杂筛选与多维度聚合展示**。因此，必须将它们在架构上独立开来。
* **面向未来的扩展**：模块的底层架构不仅服务于当下的简单列表查询，还会成为未来“时间轴长卷”、“日历视图打卡情况”、“基于灵感与情绪的深度查询”乃至“系统自动生成周报/统计图表”的基座，因此在数据拼装和加载策略上必须解耦且高效。

## 二、 功能分层规划

### 1. 本轮直接实现
以下功能是当前可以直接进入编码阶段的核心链路：
*   **历史记录列表** (`GET /api/v1/history/daily-records`)：支持分页、按时间倒序、多条件筛选（日期范围、关键词、灵感、情绪分区间）。
*   **指定日期详情** (`GET /api/v1/history/daily-records/{record_date}`)：复用 `daily_record` 模型，带出底层五张子表的完整单日数据。
*   **历史日历视图** (`GET /api/v1/history/calendar`)：按月份高亮记录情况，返回含轻量摘要的日子数组。
*   **历史时间轴视图** (`GET /api/v1/history/timeline`)：按日期范围拉取聚合列表（建议按月分组或按日倒序，前端更易处理时间卷轴渲染）。

### 2. 本轮预留不实现（后续扩展）
以下能力在代码注释或 Schema 定义中留出设计空位，但**本轮坚决不落到业务代码里**：
*   全文搜索引擎集成（ES等）
*   复杂灵感交集筛选（本轮只做单一灵感检索）
*   消费统计聚合
*   情绪趋势图（将交由独立的数据图表接口处理）
*   历史导出（PDF/CSV等）
*   相似记录推荐
*   周报/月报联动预计算

## 三、 文件清单

1.  **`app/schemas/history.py` (新增/修改)**
    *   **职责**：存放历史专用的 Pydantic Schema。例如 `HistoryListResponse` 及各项查询、视图返回的模型封装。
2.  **`app/services/history_service.py` (新增/修改)**

3.  **`app/api/routes/history.py` (新增/修改)**
    *   **职责**：实现历史模块的 `Router` 及参数校验，不写业务逻辑，通过调用 `history_service` 打包 `ApiResponse`。

## 五、 Service 设计脚本 (`HistoryService`)
规划以下核心方法：

1.  **`list_daily_records(...)`**
    *   **入参**：`session`, `user_id`, `page`, `page_size`, `start_date`, `end_date`, `keyword`, `tag`, `min_emotion_score`, `max_emotion_score`。
    *   **出参**：`HistoryListResponse`。
    *   **逻辑**：通过过滤组装 `filters`。第一步通过 `select(func.count)` 取总数；第二步做 `desc(DailyRecord.record_date)` 排序和 `offset`, `limit` 切片。
    *   **查不到时**：直接根据前置检查，若总数为 0，则立刻返回 `total_count=0, records=[], total_pages=0`，不报错且不进行下一道检索逻辑。

2.  **`get_daily_record_detail_by_date(...)`**
    *   **入参**：`session`, `user_id`, `record_date: date`。
    *   **出参**：`DailyRecordDetailResponse`（复用结构）。
    *   **逻辑**：精准匹配。由于历史模块的特殊性，可直接调用复用 `DailyRecordService.get_record_by_date` 方法来实现深度查询。

3.  **`get_calendar_view(...)`**
    *   **入参**：`session`, `user_id`, `year: int`, `month: int`。
    *   **出参**：`HistoryCalendarResponse`。
    *   **逻辑**：构造日期起止判定跨度（获取该月的 1 日和月底）。检索范围内的记录字段，只返回存在记录的日子并映射成 `HistoryCalendarDayItemResponse` 返回，无该日数据则不用包含对象下发过去。

4.  **`get_timeline_view(...)`**
    *   **入参**：`session`, `user_id`, `start_date: date`, `end_date: date`。
    *   **出参**：`HistoryTimelineResponse`。
    *   **逻辑**：查询跨度内日期的核心轻字段元素。推荐用于 Echo 的显示：**按月分组** （`strftime("%Y-%m")`）。这更符合心境与记忆的月度时间轴长卷划分。

## 六、 Router 设计脚本

*   Router 绝不编写 SQL 操作、也不对复杂业务数据流进行拼合重构，纯依靠注入与声明。

1.  **历史列表**：
    *   路径：`GET /api/v1/history/daily-records`
    *   函数名：`get_history_daily_records`
    *   参数：`start_date: Optional[date]`, `end_date: Optional[date]`, 以及 `page`, `page_size`、各种筛选作为 Query 传入。调用对应方法注入。
2.  **详情接口**：
    *   路径：`GET /api/v1/history/daily-records/{record_date}`
    *   函数名：`get_history_daily_record_detail`
    *   参数：`record_date: date` 作为 Path 直接强类型声明传入。将 `current_user` 和它一起推入 Service。
3.  **日历接口**：
    *   路径：`GET /api/v1/history/calendar`
    *   函数名：`get_history_calendar`
    *   参数：`year: int`, `month: int` （利用 `Query(...)` 对参数进行约束检查）。
4.  **时间轴接口**：
    *   路径：`GET /api/v1/history/timeline`
    *   函数名：`get_history_timeline`
    *   参数：`start_date: date`, `end_date: date`。

## 七、 查询与加载策略
1.  **列表接口**：
    *   必须维持其“轻字段”。不能且不应该顺挂着带出五张大尺寸的附属子表属性，以此确保即便每页拉 50-100 条分页依旧极高吞吐率。多条件筛选叠加走 Filter 构建。
2.  **详情接口**：
    *   这完全是相反的策略。为了在拉取当天的细节时保障性能避免延迟递推查库：必须用 `selectinload` 把附属直接查透带出：
        `selectinload(DailyRecord.events)`
        `selectinload(DailyRecord.emotions)`
        `selectinload(DailyRecord.expenses)`
        `selectinload(DailyRecord.locations)`
        `selectinload(DailyRecord.inspirations)`
3.  **日历接口**：
    *   只下发轻量概要摘要。前端绘制日历点阵只需要确认 `has_record` 和情绪等核心标志，绝不要把关联级的数据塞入月度检索流中。
4.  **时间轴接口**：
    *   从体验维度，采取“按月份分组”下发方案更理想。时间流不仅在横向上拉伸，还可以按月建立视图节点边界供前端展示段落点。

## 八、 参数与校验规则
1.  **日期使用规范**：
    *   无论 Router 的任何接口，`record_date`, `start_date`, `end_date` 等，均在 Router 直接用 `date` 原生声明拦截。绝不应当在 Service 去自己引入 `strptime` 解析（FastAPI 原生即可包揽并抛出合规 422 警告）。
2.  **分页防飞坡**：
    *   必须绑定 `page: int = Query(1, ge=1)` 和 `page_size: int = Query(20, ge=1, le=100)` 来抑制恶意暴走拖库请求。
3.  **空结果 fallback**：
    *   任何列表/多项查询当获取到自身约束总数为 `total_count = 0` 时，安全下放且明确返回：`{ "records": [], "total_pages": 0 }` 等符合前端预期的空壳，不能引发框架崩塌或者给前端奇怪的 None。
4.  **强检索规定**：
    *   `keyword` 限定模糊找（当前实现覆盖 `summary_text`、`body_text`、`user_note`、`keywords`、`events.content`、`inspirations.content`、`locations.name`、`expenses.description`、`expenses.category`）。
    *   回忆页搜索和 `search.html?q=关键词` 搜索结果页复用该接口，不额外新增搜索专用接口。
    *   `tag` 当前按灵感内容筛选日记录。
    *   极值情绪由 `Query(None, ge=... )` 卡住前后区间，然后入 `where()` 。

## 九、 安全与软删除规则
1.  **最底层越界防护**：
    *   在获取、统计或任何一根查询流水里，必须永远绑定：`DailyRecord.user_id == current_user.id`。没绑此项属于重大高风险渗透缺陷！
2.  **关于软删除的基线**：
    *   不要在开发脚本上强制采用像 `.and_(deleted_at == null)` 这类会由于不同框架或环境导致的不稳定字符串。全部查询必须遵循系统现行的基准规范过滤掉已软删除行（以当前项目中统一依赖基线为准）。
    *   附属查询若自带独立的软删属性也要同步过滤，或者依附于未删主表的映射传递。

## 十、 联调顺序
1.  优先利用脚本或者完成好的每日总结流程，给当下账号**灌入或堆叠出跨越十多天的 `daily_records`（包含情绪标签子表的丰满项）**。
2.  **调列表接口**：裸数据量查询无误后，分批组合测试（比如只筛某种关键字、指定三天范围日等），确认 `total_pages` 和 `records` 无误。
3.  **调详情**：用刚刷出的某两天下发到详情 Path。观测所有事件与开支数组是否一并获取、字段类型是否存在缺失。
4.  **本日历与轴**：验证分组正确叠加与缺省日期表现情况。
5.  **测查边界问题**：指定一月查空集、模拟未鉴权用户的串号（期望失败）、送入非常规参数结构等。

## 十一、 风险提醒
1.  **时区模糊的风险**：使用 `date` 时很容易与基于 UTC 格式持久化的系统生成偏移，造成查跨天找不到某一天。一定要警惕并使用原生类型规避时区差解析出错。
2.  **贪图方便造成查询污染**：日后增加时间轴相关，容易手一滑把所有内容塞一起给前端解析，造成庞杂性能过载。必须遵守前文说的“只有在单一详情路由里才能放纵查全子表”。
3.  **组合叠加逻辑运算错误**：在拼接 filter 列表的 `append` 过程中，逻辑的 “AND” 条件和不经意间的“OR”需要通过 SQLAlchemy 严格明确组合规范。
4.  **复用历史方法入侵**：将来出“周报总结”等能力一定不能强行挤入现有的 History 历史纯看读接口组，周报必然带复杂 LLM/Map-Reduce 算力调用。

## 十二、 最终开发顺序建议

*   **第一阶段：基石查询**
    完成 列表 (`GET /api/v1/history/daily-records`) 与 详情 (`GET /api/v1/history/daily-records/{record_date}`) 的底层 Schema 对齐和全功能实装，它是任何模块能够跑通的生命基盘。
*   **第二阶段：多元化视图呈现**
    实装 日历视图 和 时间轴视图 相关的业务与分组计算。
*   **第三阶段：进阶与除虫**
    在保证跑通的业务闭环下，对分页参数等强进行越界调优；补全并反复交叉验证“标签 & 缩紧时间 & 指定心情”等深度组合检索。

## 1. 模块目标

本模块负责每日聊天结束后，承接基于对话生成的总结内容及其结构化衍生数据的存储、查询与用户轻量修正。

具体职责包括：
- 接收总结生成任务的处理结果，将 AI 生成的今日总结及提取的结构化数据一次性统一入库
- 提供今日总结及历史某日记录的查询能力
- 允许用户对 AI 生成结果进行轻量的修正（灵感、备注、关键词）

本模块不包含：
- 跨日聚合列表与历史检索（归属于历史模块）
- 周报生成与查询（归属于周报模块）

## 9. `daily_records` 与 `conversations` 的关联及状态流转建议

`daily_records` 与 `conversations` 之间为**强 1:1 关系**，通过 `daily_records.conversation_id` 外键关联。

### 状态流转机制

本模块不再引入独立的状态流转，完全以 `conversations` 表的当前状态作为单点事实来源：

1. **触发与锁定**：当用户调用聊天模块的“结束今日记录”(`complete`)接口时，聊天模块同步将 `conversations.status` 更新为 `completing`。此状态代表“总结生成任务进行中”。
2. **异步执行与落库**：总结生成任务成功后，在同一数据库事务中：
   - 将 AI 生成的结果一次性写入 `daily_records` 记录。
   - 插入各相关子表的记录。
   - 将 `conversations.status` 推进为 `completed`。
3. **异常处理**：若任务失败，`conversations.status` 暂时停留在 `completing`；此时不会有 `daily_records` 行被写入，交由重试策略或系统告警处理。

这种设计确保了状态转移的原子性，避免了结构化日志数据处于“部分写入”或“生成失败”等复杂的中间状态体系。

## 12. 结构化数据字段设计建议

### 12.1 `keywords` 适用 JSONB 的考量

**结论：MVP 阶段适合存储为 JSONB。**

`keywords` 字段承载当日内容的概要性字符串标注（如 `["重逢", "散步", "满足"]`）。

**理由：**
- **作为一个整体展示结构使用**，没有基于单一关键词的高频独立检索、过滤需求。
- 用户每次操作均是整体替换，无需精确定位并编辑其中某一项元组。
- 符合项目《后端全局约束与命名规范》中，高内聚且无需深度关系过滤场景下允许使用 JSONB 降低独立建表冗余的要求。

*补充：当后续确实出现“关键词作为用户灵感筛选的全局搜索依据”时，才需将其升级拆解为关联词典表。*

---

### 12.2 `emotion_overall_score` 的明确定义

**建议：INTEGER 存储，取值范围 1~10。它是一个“展示型综合分”。**

- **纯展示属性**：它是一个直观的综合评分（1=极度负面，5=中性，10=极度正面），方便前端直接映射为情绪温度计或其他展示效果。
- **静态一次性生成**：本分数仅在 AI **生成当日总结过程中**一次性计算并得出写入。
- **无需且不联动重算**：在 MVP 阶段，它作为一个当期的快照写入。之后如果用户在其余流程上针对 `record_emotions` 等内容或者强度有轻微认同度或状态调整，该 `emotion_overall_score` **不进行自动联动重算**。借此将复杂的主子关联重新计分的交互开销削减，极大降低数据修改链路的复杂度。

---

### 12.3 哪些字段适合保留 `is_user_confirmed`

此字段仅保留在AI提取存在不确定且未来需要作为严谨统计数据支持的**业务级子表**内。

| 模块范畴 / 表 | 是否保留 | 原因说明 |
|---------------|----------|----------|
| `record_events` | ✅ 必须保留 | 事件存在理解或概括误差，业务需要体现其为可信资产 |
| `record_emotions` | ✅ 必须保留 | 情绪判定多为主观推论，被确认的数据才能纳入正式的情绪报表 |
| `record_expenses` | ✅ 必须保留 | 金额和货币是强准确敏感型数据，未确认的内容绝不能并入真正的收支统计 |
| `record_locations` | ✅ 必须保留 | 模糊地名的有效地址解析确认需要存在依赖 |
| `record_inspirations` | ❌ 不设置 | 灵感体系以自然存在（或不进行软删除）即可默认其为合理分类器，不增加中间认知障碍 |
| *主表总结明细* | ❌ 不设置 | 对系统叙述不需要布尔化认同，仅通过 `user_note` 补充主观定调 |

---

### 12.4 为何维持专门的业务结构化子表替代单一 JSONB

1. **为跨日计算铺路**：历史与周报模块一定会面临“提取上月超过某金额的 `expenses` 金额相加”、“提取过去七天的所有正向 `emotions` 组合”等查询需求。在 JSONB 内执行计算、聚合查询不但可读性极差且性能损耗严重。
2. **便于后续实现数据点精修与独立的软删除控制**：每个事件或消费是业务上的一个独立 Entity，应当能让用户未来实现“仅删除特定一笔花费”并在日志表中留下标准的操作时间轨迹与 `deleted_at` 追溯，而大型 JSONB 使得更新特定局部数组面临“先读取再序列化存回”的脏写风险。
3. **严格化字段约束**：`amount` 为 `NUMERIC(10, 2)` 需要绝对规避使用类似弱类型进行串改的不良边界，这仅能在严格的字段定义上强制。

## 15. 风险点与实现建议

### 风险 1：事务中断带来的系统性数据断层

- **风险特征**：总结完成后大量表需要连环下盘（主表以及 5 个子表）。若期间连接闪断或插入异常，很容易出现生成了一半的内容存库断层差异。
- **处置必须**：必须依赖完备的数据库本地事务执行原子级的单一写入封装；发生任何外键依赖问题抑或超时报错立马进行整体回滚，并将 `conversations.status` 保持为 `completing` 而不变，在重试排查后重新由调度中心触发即可。

### 风险 2：灵感软删除越权问题

- **风险特征**：由于 PUT `/daily-records/{record_id}` 的提交仅以一个 ID 数组 `inspirations_to_remove` 进行，容易导致传输他人的灵感 ID 造成越界软删除。
- **处置必须**：软删除语句必须绑定类似 `WHERE id IN (受检集合) AND record_id = {传入record_id} AND user_id = {当前验权user_id}`。保证越界的任何处理都不产生有效执行。

### 风险 3：时区容错失效

- **风险特征**：获取不正确的时区可能会导致自然日转换时跨日历错归，甚至将该日数据强加错配给上一日。
- **处置必须**：获取 `today` 查询若未能顺利解析出有效规范的 `timezone` 或者该请求无时区支撑时，应该阻断获取直接抛出 `500`。

- 需要的方法：`update_record(session, user_id, record_id, req)`

  - `AsyncSession`
  - 当前用户 ID
  - `record_id`
  - `UpdateDailyRecordRequest`
- 出参：
  - `DailyRecordDetailResponse`

    - 记录存在
    - 记录未被软删除

    - 对请求数组中的每个灵感内容先调用 `strip()`

    - 去重后新增灵感
    - 避免与当前记录中已有有效灵感同名重复
  - `inspirations_to_remove`

- 灵感冲突处理：

## 1. 模块目标

### 为什么 history 不再只是"daily_records 列表"

原有的 history 模块以"按天展示摘要列表"为中心，每一行对应一天，用户关注的是"哪天有记录"。这种结构适合时间轴翻阅，但不适合"我想看最近两周发生了哪些事件"或"我的情绪在这段时间里是什么状态"这类横向跨日查询。

### 为什么五张结构化子表应该成为历史主页面的核心

Echo 的核心数据资产是从对话中提炼出来的 `events`, `emotions`, `expenses`, `locations`, `inspirations` 五张结构化子表。用户真正感兴趣的不是"这一天"本身，而是：

- 最近一个月我去了哪些地方？
- 最近一周我都花了什么钱？
- 这段时间我出现最多的情绪是什么？

这些问题天然是横跨多天、围绕某一类结构化数据的查询，必须让五张子表"独立出镜"，而不能永远藏在某一天的详情里。

### 为什么"记录主页"应该由"今日总结 + 五表主视图"组成

这不是单纯的产品设计选择，而是数据架构的自然分工：
- **上半部分（今日总结区）**：时间点是"今天"，接口已有，调用现有的 `DailyRecordService.get_today_record` 即可，不需要新接口。
- **下半部分（五表主视图）**：时间跨度是用户感兴趣的一段范围，需要五个独立的轻量表格接口来支撑，这正是本轮要新增的内容。

- 查询逻辑：join 主表取 `record_date`，过滤子表、主表软删除，支持 `emotion_label` 精准匹配和 intensity 范围筛选
- 筛选边界校验：`min_intensity > max_intensity` 时返回 400

**3. `list_expenses(session, user_id, page, page_size, start_date, end_date, category, min_amount, max_amount)`**

- 出参：`HistoryExpenseItemResponse` 分页列表
- 查询逻辑：同上，`amount` 用 `>=` / `<=` 构建，join 主表取 `record_date`
- 筛选边界校验：`min_amount > max_amount` 时返回 400
- `category` 参数需做 strip 清洗后再做精准匹配

**4. `list_locations(session, user_id, page, page_size, start_date, end_date, keyword)`**

- 出参：`HistoryLocationItemResponse` 分页列表
- 查询逻辑：keyword 对 `name` 做 ilike 匹配

**5. `list_inspirations(session, user_id, page, page_size, start_date, end_date, keyword)`**

- 出参：`HistoryInspirationItemResponse` 分页列表
- 查询逻辑：`keyword` 对 `content` 做模糊匹配，join 主表取 `record_date`，过滤子表软删除（按现有项目规范）

## 7. 查询与性能策略

### 为什么五表接口不能直接返回完整 daily_record

五表接口的前端用途是"表格展示"，每一行只需要该条记录自身的字段 + 所属日期 + 所属记录 UUID。如果为了取 `record_date` 就把整个 `DailyRecord` 的所有字段都加载进来，等于把今日详情级别的重量数据放进一个翻页 20 条的列表里，是典型的过度加载。

### 正确的查询方式

五表接口应该：

1. 直接在子表（如 `record_events`）上发起查询
2. 通过 `JOIN DailyRecord ON record_events.record_id = daily_records.id` 取到 `record_date`
3. 同时过滤：
   - `record_events.user_id == user_id`（子表本身有 user_id 字段）
   - `record_events.deleted_at.is_(None)`（子表软删除）
   - `daily_records.deleted_at.is_(None)`（主表软删除，防止查到已删主记录下的子条目）
4. 只 select 所需字段，不要 select 整个实体

### expenses 金额字段处理

- 数据库层是 `Numeric(10, 2)`，接口层 Schema 建议 `amount: float`（Pydantic 会自动转换）
- 不要在 `amount` 筛选时用字符串比较，必须用数值类型的 `>=` / `<=`
- `min_amount` / `max_amount` 均为 `Optional[float]`，各自独立生效，不强制两者同时传

### 避免拖库的策略

- 所有五表接口都必须有分页（`page`, `page_size`）
- 日期范围如果前端不传，建议 Service 层默认限制最近 N 天（例如 90 天）或通过文档约定前端必须传日期范围，不允许全量拉取

## 8. 搜索能力设计

### 历史详情页的搜索：建议本轮采用方案 A（前端页内过滤）

**结论：当前 MVP 阶段，历史详情页的搜索无需落后端。**

**理由：**

1. 单日详情数据量极其有限。一天的 events / emotions / expenses / locations / inspirations 每类最多几条到十几条，前端 JS 做内存过滤绰绰有余，不存在性能瓶颈。
2. 现有 `GET /api/v1/history/daily-records/{record_date}` 已经一次性返回完整数据结构（五张子表都有），前端根本不需要再向后端发额外请求。
3. 增加 `GET /api/v1/history/daily-records/{record_date}/search?q=...` 这类接口，在当前阶段只会造成接口膨胀，测试与维护成本不成比例。
4. 如果未来用户量增大，或单日数据量因 AI 分析精细化显著增加，届时再补后端搜索接口成本可控。

**前端实现建议（供参考）：**

拿到详情接口数据后，前端对 `events[].content`, `inspirations[].content`, `expenses[].category`, `locations[].name` 等字段做简单的 `.filter(item => item.xxx.includes(keyword))` 即可，响应速度瞬时，用户体验不逊于后端搜索。

## 9. 参数与校验规则

### 通用规则（对所有五表接口生效）

- `start_date`, `end_date`：Router 直接声明为 `date` 类型，FastAPI 自动校验格式
- `page`：`Query(1, ge=1)`
- `page_size`：`Query(20, ge=1, le=100)`
- `start_date > end_date`：Service 层做交叉校验，返回 `ErrorResponseAPIException(status_code=400, code=40001, ...)`
- 空结果时：`records = []`, `total_count = 0`, `total_pages = 0`，不抛异常

### 各表额外筛选参数建议

| 接口 | 额外参数 | 处理建议 |
|---|---|---|
| events | `keyword: Optional[str]` | strip 后 ilike，空字符串跳过 |
| events | `source: Optional[str]` | 精准匹配，可选 |
| emotions | `emotion_label: Optional[str]` | 精准匹配，strip |

| locations | `keyword: Optional[str]` | strip 后 ilike |
| inspirations | `keyword: Optional[str]` | strip 后 ilike |

## 10. 安全与软删除规则

- 所有五表接口的 SQL 查询都必须在子表层绑定 `user_id`（`record_events.user_id == user_id`），子表模型已有 `user_id` 字段，不需要绕道主表
- 必须同时过滤：子表 `deleted_at.is_(None)` + 主表 `DailyRecord.deleted_at.is_(None)`，防止查到主记录已软删但子条目残留的肮脏数据
- 不使用 `.and_(deleted_at == null)` 这种不稳定写法，以 SQLAlchemy 2.0 标准 `.is_(None)` 为准
- 分页计算不暴露数据量差异，空结果时一律返回空壳，不报 404

## 11. 联调顺序

1. **确认单日基准数据**：先调 `GET /api/v1/history/daily-records/{record_date}`，确认某天的全部子表数据已正常返回（events, emotions, expenses, locations, inspirations 齐全）
2. **联调 events 接口**：先以已知有数据的日期范围调用 `GET /api/v1/history/events`，验证 `record_date`, `daily_record_id` 等关联字段正确
3. **联调其余四表**：emotions → expenses → locations → inspirations 依次验证基础查询和字段返回
4. **验证所有筛选参数**：日期范围、keyword、各表特有筛选，逐个验证生效
5. **验证分页**：指定 `page_size=1`，翻到 `page=2`，确认切片逻辑正确
6. **验证空结果**：传入无记录的日期范围，确认返回 `records=[]`, `total_pages=0`
7. **验证跳转链路**：前端拿到某行的 `record_date`，跳转到 `GET /api/v1/history/daily-records/{record_date}`，确认详情页数据匹配
8. **确认详情页搜索走前端**：访问详情页后在前端做内存过滤，确认体验可接受，不需要补后端接口

## 12. 风险提醒

- **五表接口命名爆炸**：路径必须统一风格 `/history/events`, `/history/emotions` 等，不要出现 `/history/record-events`, `/history/history-emotions` 这种前缀叠加或命名不一致的情况
- **子表 user_id 绑定漏掉**：子表模型已有独立的 `user_id` 字段，但开发时容易误以为"join 了主表就安全了"，实际上必须显式绑定子表的 `user_id`，否则存在越权风险
- **expenses 金额范围筛选错误**：`min_amount / max_amount` 必须是数值类型，不要接受字符串，否则数据库比较结果不可预期。交叉校验（`min > max`）不能漏
- **inspirations 接口和历史列表的 tag 筛选混淆**：`/history/inspirations` 是"查灵感记录流"，返回每条灵感的详情及所属日期；`/history/daily-records?tag=xxx` 是"按灵感内容筛选日记录"。两者用途不同，不要合并或互相干扰
- **把五表接口做重**：这些接口是轻量表格接口，每个接口只返回该子表的核心字段，不要贪图方便把主表大字段（`summary_text`, `extra_json` 等）也塞进去
- **详情页提前塞后端搜索**：本轮已明确前端做页内过滤，不要在联调时"顺手"加 search 接口，避免范围蔓延
