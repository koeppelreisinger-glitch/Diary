# 详细联调操作教程（基于 integration test checklist）

本文基于现有的最小联调清单 [08_最小联调步骤.md](./08_最小联调步骤.md) 整理，并已对照当前仓库中的真实路由与服务实现进行修正。

当前代码实现的关键口径如下：

- `POST /api/v1/conversations/{conversation_id}/complete` 不是纯异步占位，而是先将会话置为 `completing`，随后同步完成总结生成与落库，正常情况下响应里会直接带回 `daily_record`，最终状态应为 `completed`
- 新注册用户默认时区是 `UTC`，如果不先修改 `timezone`，今天的 `record_date` 可能和联调人员的本地日期不一致
- 补写消息的 `is_supplement` 在当前实现里是请求体字段，不是 query 参数
- `history` 相关日期参数现在兼容这 4 种格式：`YYYY-MM-DD`、`YYYY/MM/DD`、`YYYY.MM.DD`、`YYYYMMDD`；联调时仍推荐优先使用 `YYYY-MM-DD`

---

## 1. 联调目标

本教程用于跑通下面这条完整闭环：

1. 注册/登录
2. 设置用户时区
3. 创建今日会话
4. 发送聊天消息
5. 完成会话并生成日记录
6. 校验今日记录、历史详情、历史列表与五张结构化子表
7. 可选验证补写、正文重建、轻量编辑

## 2. 联调前准备

### 2.1 启动服务

在项目根目录执行：

```bash
.venv\Scripts\uvicorn.exe app.main:app --host 127.0.0.1 --port 8000 --reload
```

启动后确认下面两个地址可访问：

- Swagger 文档：`http://127.0.0.1:8000/docs`
- 健康检查：`http://127.0.0.1:8000/health`

### 2.2 推荐联调方式

推荐优先使用 Swagger UI 联调，因为可以直接看到参数模型和返回结构。

如果要用脚本快速回归，仓库里已有：

- `test_all_endpoints.py`
- `test_history_and_complete.py`

### 2.3 准备一组测试账号

如果还没有账号，先注册一个。例如：

```json
{
  "phone": "13900001111",
  "password": "Password123!",
  "nickname": "联调测试"
}
```

### 2.4 推荐测试语料

为了让 `events / emotions / expenses / locations / tags` 五类数据尽量一次性被命中，建议联调时发送下面两段文本：

```json
{
  "content_type": "text",
  "content": "今天在公司开了很多会，还见了客户，整个人有点累。"
}
```

```json
{
  "content_type": "text",
  "content": "晚上去咖啡店买了一杯咖啡花了32元，之后回家了。"
}
```

这两段文本可以覆盖当前规则中的这些抽取点：

- `开会` -> events
- `累` -> emotions
- `咖啡` -> expenses
- `公司 / 咖啡店 / 家` -> locations
- `开会 / 客户 / 咖啡` -> tags 和 keywords

## 3. Swagger 授权方式

### 3.1 注册

接口：`POST /api/v1/auth/register`

操作：

1. 打开 Swagger 文档页
2. 找到 `POST /api/v1/auth/register`
3. 点击 `Try it out`
4. 填入注册请求体
5. 点击 `Execute`

预期结果：

- HTTP 状态码 `200`
- 返回体 `code = 20000`
- `data` 中可看到当前用户基础信息

### 3.2 登录

接口：`POST /api/v1/auth/login`

你可以用两种方式登录：

方式 A：直接调登录接口，手动复制 `access_token`

方式 B：使用 Swagger 右上角 `Authorize`

推荐使用方式 B，步骤如下：

1. 点击 Swagger 右上角 `Authorize`
2. 在弹窗里填写：
   - `username` 填手机号
   - `password` 填密码
3. 点击 `Authorize`
4. 关闭弹窗

注意：

- 这里不要手动加 `Bearer ` 前缀，Swagger 会自动处理
- 当前项目给 Swagger 单独提供了 `swagger_login` 适配接口，所以右上角授权可以直接用

验证方法：

调用 `GET /api/v1/users/me`，如果返回成功，说明授权生效。

## 4. 主线联调步骤

## 4.1 先修正用户时区

接口：`PUT /api/v1/settings`

推荐请求体：

```json
{
  "timezone": "Asia/Shanghai",
  "input_preference": "text",
  "reminder_enabled": false
}
```

为什么这一步必须先做：

- 新用户默认时区是 `UTC`
- 会话的 `record_date` 是按用户时区换算“今天”得到的
- 如果不先改时区，后续你看到的“今天”可能不是你想要的自然日

预期结果：

- `code = 20000`
- 返回体中 `timezone = "Asia/Shanghai"`

### 4.2 确认今天还没有会话

接口：`GET /api/v1/conversations/today`

预期结果之一：

```json
{
  "code": 20000,
  "message": "success",
  "data": {
    "has_today": false,
    "conversation": null
  }
}
```

如果已经有会话，不必强行新建，直接记录它的 `conversation.id` 并继续后续步骤。

### 4.3 创建今天会话

接口：`POST /api/v1/conversations`

请求体：无

预期结果：

- `status = "recording"`
- 记下返回的 `data.id`，后续称为 `conversation_id`
- `record_date` 应符合你刚设置的时区日期

典型返回示例：

```json
{
  "code": 20000,
  "message": "success",
  "data": {
    "id": "e456d9a4-1234-abcd-8f90-111122223333",
    "status": "recording",
    "record_date": "2026-04-18",
    "created_at": "2026-04-18T10:00:00Z"
  }
}
```

如果返回 `409`，通常表示当天会话已存在，此时改用 `GET /api/v1/conversations/today` 获取已有会话 ID。

### 4.4 发送第一条消息

接口：`POST /api/v1/conversations/{conversation_id}/messages`

请求体：

```json
{
  "content_type": "text",
  "content": "今天在公司开了很多会，还见了客户，整个人有点累。",
  "is_supplement": false
}
```

预期结果：

- 返回 `user_message` 和 `ai_message`
- 两条消息都应成功写入
- `user_message.role = "user"`
- `ai_message.role = "ai"`

### 4.5 发送第二条消息

接口仍然是：`POST /api/v1/conversations/{conversation_id}/messages`

请求体：

```json
{
  "content_type": "text",
  "content": "晚上去咖啡店买了一杯咖啡花了32元，之后回家了。",
  "is_supplement": false
}
```

预期结果：

- 继续返回一组 `user_message` + `ai_message`
- 第二次发送后，会话中应累计有 4 条消息

### 4.6 可选：拉取消息列表做中间校验

接口：`GET /api/v1/conversations/{conversation_id}/messages`

建议检查：

- `total_count = 4`
- `messages` 按 `sequence_number` 升序排列
- 角色顺序应类似 `user -> ai -> user -> ai`

### 4.7 完成会话并触发总结生成

接口：`POST /api/v1/conversations/{conversation_id}/complete`

请求体：无

当前实现的真实行为：

1. 先将会话状态更新为 `completing`
2. 立刻调用总结生成服务
3. 生成 `daily_record` 主记录及五张子表
4. 将会话最终置为 `completed`
5. 在响应中直接返回 `daily_record`

联调时重点检查：

- `code = 20000`
- `status` 最终应为 `completed`
- `daily_record.record_date` 与会话日期一致
- `daily_record.summary_text` 不为空
- `daily_record.body_text` 不为空
- `daily_record.emotion_overall_score` 有值
- `daily_record.events / emotions / expenses / locations / tags` 中至少有部分数据

基于上面的推荐语料，通常应能看到：

- `events` 至少 1 条
- `emotions` 至少 1 条
- `expenses` 至少 1 条，金额约 `32.0`
- `locations` 至少包含 `公司`、`咖啡店`、`家` 中的一部分
- `tags` 至少包含 `工作`、`消费` 中的一部分

### 4.8 校验今日记录接口

接口：`GET /api/v1/daily-records/today`

预期结果：

- `has_record = true`
- `is_generating = false`
- `record.id` 存在
- `record.body_text`、`record.summary_text`、五张子表都能返回

这一步是判断“聊天闭环是否真的落到日记录层”的关键检查点。

### 4.9 校验历史列表

接口：`GET /api/v1/history/daily-records`

建议检查：

- `total_count >= 1`
- `records[0].record_date` 包含本次生成的日期
- `summary_text` 和 `emotion_overall_score` 正常返回

### 4.10 校验历史详情

接口：`GET /api/v1/history/daily-records/{record_date}`

其中 `record_date` 使用上一步或 `complete` 响应中的日期。

当前支持的写法示例：

- `GET /api/v1/history/daily-records/2026-04-18`
- `GET /api/v1/history/daily-records/2026/04/18`
- `GET /api/v1/history/daily-records/2026.04.18`
- `GET /api/v1/history/daily-records/20260418`

其中仍推荐优先使用第一种标准写法。

建议检查：

- `body_text` 是否存在
- `summary_text` 是否存在
- `events / emotions / expenses / locations / tags` 是否齐全
- 子表内容是否和刚才聊天文本一致

到这里，最小联调闭环就跑通了。

## 5. 五表主视图扩展联调

如果本次还要验证“记录主页下半区”的五个历史表格，按下面顺序调。

说明：

- 本节所有 `start_date` / `end_date` 参数都支持 `YYYY-MM-DD`、`YYYY/MM/DD`、`YYYY.MM.DD`、`YYYYMMDD`
- 若你在 Swagger 中联调，最稳妥的仍然是填 `YYYY-MM-DD`

### 5.1 events

接口：`GET /api/v1/history/events`

推荐参数：

- `start_date=2026-04-01`
- `end_date=2026-04-30`
- `keyword=开会`

重点检查：

- 返回项里有 `daily_record_id`
- 返回项里有 `record_date`
- `content` 与聊天内容匹配

### 5.2 emotions

接口：`GET /api/v1/history/emotions`

推荐参数：

- `start_date=2026-04-01`
- `end_date=2026-04-30`
- `emotion_label=疲惫`
- `min_intensity=1`
- `max_intensity=5`

重点检查：

- `emotion_label`
- `intensity`
- `record_date`

### 5.3 expenses

接口：`GET /api/v1/history/expenses`

推荐参数：

- `start_date=2026-04-01`
- `end_date=2026-04-30`
- `category=餐饮`
- `min_amount=1`
- `max_amount=100`

重点检查：

- `amount` 为数值
- `currency` 正常返回
- `description` 与咖啡消费匹配

### 5.4 locations

接口：`GET /api/v1/history/locations`

推荐参数：

- `start_date=2026-04-01`
- `end_date=2026-04-30`
- `name=咖啡`

重点检查：

- 是否能模糊匹配出 `咖啡店`
- 每条记录都带 `daily_record_id`

### 5.5 tags

接口：`GET /api/v1/history/tags`

推荐参数：

- `start_date=2026-04-01`
- `end_date=2026-04-30`
- `tag_name=工作`

重点检查：

- `tag_name`
- `source`
- `record_date`

### 5.6 分页与空结果

建议再补两组验证：

分页验证：

- 给任一五表接口传 `page_size=1`
- 再请求 `page=2`
- 确认 `total_count / total_pages / current_page` 正确

空结果验证：

- 传一个没有数据的日期范围
- 推荐直接使用下面这个示例：
  `GET /api/v1/history/events?start_date=2026-01-01&end_date=2026-01-02&page=1&page_size=20`
  或
  `GET /api/v1/history/events?start_date=2026/01/01&end_date=2026/01/02&page=1&page_size=20`
- 预期返回空壳，不报错
- `records = []`
- `total_count = 0`
- `total_pages = 0`

如果这里看到的是“参数校验失败”，优先排查这两类问题：

- 日期字符串不属于上述 4 种支持格式
- `start_date` 晚于 `end_date`

## 6. 补写、重建与轻量编辑联调

这一部分不是最小闭环必测，但如果页面已经接到“补写”和“编辑”能力，建议一起验证。

### 6.1 已完成会话继续补写

接口：`POST /api/v1/conversations/{conversation_id}/messages`

注意：当前实现里 `is_supplement` 是请求体字段。

请求体示例：

```json
{
  "content_type": "text",
  "content": "补充一下，下午回家前还在公司楼下又见了一个客户。",
  "is_supplement": true
}
```

预期结果：

- 已 `completed` 的会话仍然允许发消息
- 仍然会同步返回 `ai_message`

如果你把 `is_supplement` 传成 `false`，预期会收到 `409`。

### 6.2 保存本次补充，重建日记录

接口：`POST /api/v1/daily-records/{record_id}/supplement`

请求体可以用：

```json
{}
```

预期结果：

- 服务端会读取该会话的全部 user 消息
- 重新生成最新 `body_text`
- 全量重建结构化数据
- 返回更新后的完整 `record`

### 6.3 手动修改正文并重建

接口：`PUT /api/v1/daily-records/{record_id}/body`

先确认这里用的是 `record_id`，不是 `record_date`。

`record_id` 的获取方式：

- 优先直接使用 `POST /api/v1/conversations/{conversation_id}/complete` 返回的 `data.daily_record.id`
- 或使用 `GET /api/v1/daily-records/today` 返回的 `data.record.id`

不要把 `2026-04-18` 这种日期值填进 `record_id` 路径位，否则会直接参数校验失败。

请求体示例：

```json
{
  "body_text": "今天主要在公司开会，也见了客户。晚上去咖啡店买了咖啡，回家后还是感觉有点累。"
}
```

预期结果：

- `body_text` 被更新
- `summary_text / emotion_overall_score / keywords / 五张子表` 同步重建

补充说明：

- 这条接口当前已实测可通
- 如果你是在命令行工具里手动调用，务必保证请求体是 UTF-8 JSON；否则中文正文可能出现乱码，导致提取结果异常

### 6.4 轻量编辑标签和备注

接口：`PUT /api/v1/daily-records/{record_id}`

请求体示例：

```json
{
  "user_note": "这天工作密度很高",
  "keywords": ["工作", "会议", "咖啡"],
  "tags_to_add": ["复盘"],
  "tags_to_remove": []
}
```

预期结果：

- `user_note` 更新成功
- `keywords` 被整体覆盖
- 新标签成功加入
- 不触发结构化重建

## 7. 常见报错与排查方法

### 7.1 `50002` 用户时区配置缺失或非法

常见原因：

- 没先设置 `timezone`
- 设置了错误的 IANA 时区标识

处理方式：

- 先调 `GET /api/v1/settings`
- 再调 `PUT /api/v1/settings`，例如把时区改成 `Asia/Shanghai`

### 7.2 `40901` 今日会话已存在

常见原因：

- 同一天重复调用 `POST /conversations`

处理方式：

- 改用 `GET /api/v1/conversations/today`
- 直接拿已有 `conversation_id`

### 7.3 `40902` 会话正在生成总结中

常见原因：

- 会话状态处于 `completing` 时继续发消息

处理方式：

- 等待 `complete` 结束
- 或重新查询今日记录状态

### 7.4 `40903` 今日记录已结束

常见原因：

- 已 `completed` 的会话继续按普通消息发送

处理方式：

- 如果是补写场景，在请求体里传 `"is_supplement": true`

### 7.5 `40002` 开启提醒时未传提醒时间

常见原因：

- `PUT /settings` 时把 `reminder_enabled` 设为 `true`，但没有传 `reminder_time`

处理方式：

- 补一个合法时间，例如 `"22:30"`

### 7.6 `422` 参数校验失败

常见原因：

- 日期格式不是 `YYYY-MM-DD`
- UUID 格式不合法
- 请求体字段类型不对

处理方式：

- 以 Swagger 模型为准重新填参

## 8. 建议验收结论

当下面这些条件全部满足时，可以判定本轮联调通过：

1. 能成功注册、登录并完成 Swagger 授权
2. 修改时区后，能正确创建“今天”的会话
3. 发送两轮消息后，能拿到用户消息和 AI 回复
4. 调用 `complete` 后能直接拿到 `daily_record`
5. `GET /daily-records/today` 能查到完整今日记录
6. `GET /history/daily-records/{record_date}` 能查到同一份详情
7. 五表接口至少能查到与本次聊天对应的事件、情绪、消费、地点、标签
8. 补写与编辑场景调用后，记录内容能正确刷新
