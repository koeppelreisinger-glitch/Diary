# 《10_记录主页与历史五表接口》后端开发脚本 / 实施清单

> 本文档是 Echo 项目"记录主页（今日总结 + 五表历史主视图）"与"历史详情页搜索能力"的后端规划文件。
> 不含任何代码，作为后续连续编码的唯一蓝图。

---

## 1. 模块目标

### 为什么 history 不再只是"daily_records 列表"

原有的 history 模块以"按天展示摘要列表"为中心，每一行对应一天，用户关注的是"哪天有记录"。这种结构适合时间轴翻阅，但不适合"我想看最近两周发生了哪些事件"或"我的情绪在这段时间里是什么状态"这类横向跨日查询。

### 为什么五张结构化子表应该成为历史主页面的核心

Echo 的核心数据资产是从对话中提炼出来的 `events`, `emotions`, `expenses`, `locations`, `inspirations` 五张结构化子表。用户真正感兴趣的不是"这一天"本身，而是：

- 最近一个月我去了哪些地方？
- 最近一周我都花了什么钱？
- 这段时间我出现最多的情绪是什么？

这些问题天然是横跨多天、围绕某一类结构化数据的查询，必须让五张子表"独立出镜"，而不能永远藏在某一天的详情里。

### 为什么"记录主页"应该由"今日总结 + 五表主视图"组成

这不是单纯的产品设计选择，而是数据架构的自然分工：
- **上半部分（今日总结区）**：时间点是"今天"，接口已有，调用现有的 `DailyRecordService.get_today_record` 即可，不需要新接口。
- **下半部分（五表主视图）**：时间跨度是用户感兴趣的一段范围，需要五个独立的轻量表格接口来支撑，这正是本轮要新增的内容。

## 3. 文件清单

### 建议：扩展到现有文件，不新建大模块

理由如下：

1. 五表接口的业务逻辑本质上是"历史数据查询"的延伸，和现有 `history_service.py` 同属一个关注域。

|---|---|---|
| `app/schemas/history.py` | 修改 | 追加五个表格 Item Response 模型和统一分页壳 |

## 4. 五表接口设计脚本

### 通用约定（对所有五个接口生效）

- 所有接口路径以 `GET /api/v1/history/` 为前缀
- 所有接口都支持：`start_date`, `end_date`, `page`, `page_size`
- 所有接口都必须绑定当前用户的 `user_id`
- 所有接口都必须过滤主表和子表的软删除
- 每个返回项都必须携带 `record_date` 和 `daily_record_id`，方便前端点击某行后跳转到对应历史详情页

---

### A. events 表格接口

**路径：** `GET /api/v1/history/events`

**返回字段（每条 item）：**

| 字段 | 说明 |
|---|---|
| `id` | 事件 UUID |
| `daily_record_id` | 所属日记录 UUID（用于前端跳转详情页） |
| `record_date` | 所属日期（join 自主表） |
| `content` | 事件内容文本 |
| `source` | 来源（ai / user） |
| `is_user_confirmed` | 用户是否确认 |
| `created_at` | 创建时间 |

**支持的筛选参数：**

- `start_date` / `end_date`：按 record_date 范围筛选
- `keyword`：对 `content` 做 ilike 模糊匹配（可选，MVP 阶段建议支持）
- `source`：按来源精准筛选（可选，可为后续预留）

**分页：** `page` / `page_size`

**前端跳转逻辑：** 点击某行时，用 `daily_record_id` 或 `record_date` 跳转至 `GET /api/v1/history/daily-records/{record_date}`

## 5. Schema 设计脚本

### 建议采用方案 2：统一分页壳 + 各 item 分别定义

理由：项目风格已经是 `HistoryListResponse` 这种外层壳 + 内层 item 的模式，五表接口跟进一致，减少认知负担。

### 需要在 `app/schemas/history.py` 中追加的模型

**分页壳（可复用）：**

`HistoryTableListResponse[T]`：通用分页容器，包含 `total_count`, `total_pages`, `current_page`, `page_size`, `records: List[T]`

如果 Pydantic 泛型在项目内使用体验不理想，可分别定义五个外壳（如 `HistoryEventListResponse`），核心字段保持一致，不要混用字段名。

**各 item 模型：**

- `HistoryEventItemResponse`：携带 `id`, `daily_record_id`, `record_date`, `content`, `source`, `is_user_confirmed`, `created_at`
- `HistoryEmotionItemResponse`：携带 `id`, `daily_record_id`, `record_date`, `emotion_label`, `intensity`, `source`, `is_user_confirmed`, `created_at`
- `HistoryExpenseItemResponse`：携带 `id`, `daily_record_id`, `record_date`, `amount`, `currency`, `category`, `description`, `source`, `is_user_confirmed`, `created_at`
- `HistoryLocationItemResponse`：携带 `id`, `daily_record_id`, `record_date`, `name`, `source`, `is_user_confirmed`, `created_at`
- `HistoryInspirationItemResponse`：携带 `id`, `daily_record_id`, `record_date`, `content`, `source`, `created_at`

## 6. Service 设计脚本

### 建议：继续扩展 `HistoryService`，不新建 Service

**理由：**

1. 五表查询的核心逻辑（绑定用户、日期范围筛选、分页）与现有 `list_daily_records` 高度相似，抽出共享逻辑更容易。
2. 新建 `HistoryStructuredService` 会造成调用链拆散，Router 需要同时依赖两个 Service，增加维护复杂度。
3. 当业务逻辑真正足够独立时，再拆不迟。

### 需要新增的方法

**1. `list_events(session, user_id, page, page_size, start_date, end_date, keyword, source)`**

- 入参：见上
- 出参：分页后的 `HistoryEventItemResponse` 列表及分页元信息
- 查询逻辑：join `record_events` → join `daily_records`（取 `record_date`），过滤 `record_events.user_id == user_id`，过滤子表软删除，过滤主表软删除
- 分页：两段查询（count + 数据）
- 筛选：日期范围走主表 `record_date`，keyword 走子表 `content.ilike`

**2. `list_emotions(session, user_id, page, page_size, start_date, end_date, emotion_label, min_intensity, max_intensity)`**

- 出参：`HistoryEmotionItemResponse` 分页列表
- 查询逻辑：join 主表取 `record_date`，过滤子表、主表软删除，支持 `emotion_label` 精准匹配和 intensity 范围筛选
- 筛选边界校验：`min_intensity > max_intensity` 时返回 400

**3. `list_expenses(session, user_id, page, page_size, start_date, end_date, category, min_amount, max_amount)`**

- 出参：`HistoryExpenseItemResponse` 分页列表
- 查询逻辑：同上，`amount` 用 `>=` / `<=` 构建，join 主表取 `record_date`
- 筛选边界校验：`min_amount > max_amount` 时返回 400
- `category` 参数需做 strip 清洗后再做精准匹配

**4. `list_locations(session, user_id, page, page_size, start_date, end_date, keyword)`**

- 出参：`HistoryLocationItemResponse` 分页列表
- 查询逻辑：keyword 对 `name` 做 ilike 匹配

**5. `list_inspirations(session, user_id, page, page_size, start_date, end_date, keyword)`**

- 出参：`HistoryInspirationItemResponse` 分页列表
- 查询逻辑：`keyword` 对 `content` 做模糊匹配，join 主表取 `record_date`，过滤子表软删除（按现有项目规范）

## 7. 查询与性能策略

### 为什么五表接口不能直接返回完整 daily_record

五表接口的前端用途是"表格展示"，每一行只需要该条记录自身的字段 + 所属日期 + 所属记录 UUID。如果为了取 `record_date` 就把整个 `DailyRecord` 的所有字段都加载进来，等于把今日详情级别的重量数据放进一个翻页 20 条的列表里，是典型的过度加载。

### 正确的查询方式

五表接口应该：

1. 直接在子表（如 `record_events`）上发起查询
2. 通过 `JOIN DailyRecord ON record_events.record_id = daily_records.id` 取到 `record_date`
3. 同时过滤：
   - `record_events.user_id == user_id`（子表本身有 user_id 字段）
   - `record_events.deleted_at.is_(None)`（子表软删除）
   - `daily_records.deleted_at.is_(None)`（主表软删除，防止查到已删主记录下的子条目）
4. 只 select 所需字段，不要 select 整个实体

### expenses 金额字段处理

- 数据库层是 `Numeric(10, 2)`，接口层 Schema 建议 `amount: float`（Pydantic 会自动转换）
- 不要在 `amount` 筛选时用字符串比较，必须用数值类型的 `>=` / `<=`
- `min_amount` / `max_amount` 均为 `Optional[float]`，各自独立生效，不强制两者同时传

### 避免拖库的策略

- 所有五表接口都必须有分页（`page`, `page_size`）
- 日期范围如果前端不传，建议 Service 层默认限制最近 N 天（例如 90 天）或通过文档约定前端必须传日期范围，不允许全量拉取

## 8. 搜索能力设计

### 历史详情页的搜索：建议本轮采用方案 A（前端页内过滤）

**结论：当前 MVP 阶段，历史详情页的搜索无需落后端。**

**理由：**

1. 单日详情数据量极其有限。一天的 events / emotions / expenses / locations / inspirations 每类最多几条到十几条，前端 JS 做内存过滤绰绰有余，不存在性能瓶颈。
2. 现有 `GET /api/v1/history/daily-records/{record_date}` 已经一次性返回完整数据结构（五张子表都有），前端根本不需要再向后端发额外请求。
3. 增加 `GET /api/v1/history/daily-records/{record_date}/search?q=...` 这类接口，在当前阶段只会造成接口膨胀，测试与维护成本不成比例。
4. 如果未来用户量增大，或单日数据量因 AI 分析精细化显著增加，届时再补后端搜索接口成本可控。

**前端实现建议（供参考）：**

拿到详情接口数据后，前端对 `events[].content`, `inspirations[].content`, `expenses[].category`, `locations[].name` 等字段做简单的 `.filter(item => item.xxx.includes(keyword))` 即可，响应速度瞬时，用户体验不逊于后端搜索。

## 9. 参数与校验规则

### 通用规则（对所有五表接口生效）

- `start_date`, `end_date`：Router 直接声明为 `date` 类型，FastAPI 自动校验格式
- `page`：`Query(1, ge=1)`
- `page_size`：`Query(20, ge=1, le=100)`
- `start_date > end_date`：Service 层做交叉校验，返回 `ErrorResponseAPIException(status_code=400, code=40001, ...)`
- 空结果时：`records = []`, `total_count = 0`, `total_pages = 0`，不抛异常

### 各表额外筛选参数建议

| 接口 | 额外参数 | 处理建议 |
|---|---|---|
| events | `keyword: Optional[str]` | strip 后 ilike，空字符串跳过 |
| events | `source: Optional[str]` | 精准匹配，可选 |
| emotions | `emotion_label: Optional[str]` | 精准匹配，strip |
| emotions | `min_intensity / max_intensity: Optional[int]` | 交叉校验，Query 约束 ge=0 |
| expenses | `category: Optional[str]` | strip 后精准匹配 |
| expenses | `min_amount / max_amount: Optional[float]` | 交叉校验，ge=0 |
| locations | `keyword: Optional[str]` | strip 后 ilike |
| inspirations | `keyword: Optional[str]` | strip 后 ilike |

## 10. 安全与软删除规则

- 所有五表接口的 SQL 查询都必须在子表层绑定 `user_id`（`record_events.user_id == user_id`），子表模型已有 `user_id` 字段，不需要绕道主表
- 必须同时过滤：子表 `deleted_at.is_(None)` + 主表 `DailyRecord.deleted_at.is_(None)`，防止查到主记录已软删但子条目残留的肮脏数据
- 不使用 `.and_(deleted_at == null)` 这种不稳定写法，以 SQLAlchemy 2.0 标准 `.is_(None)` 为准
- 分页计算不暴露数据量差异，空结果时一律返回空壳，不报 404

## 11. 联调顺序

1. **确认单日基准数据**：先调 `GET /api/v1/history/daily-records/{record_date}`，确认某天的全部子表数据已正常返回（events, emotions, expenses, locations, inspirations 齐全）
2. **联调 events 接口**：先以已知有数据的日期范围调用 `GET /api/v1/history/events`，验证 `record_date`, `daily_record_id` 等关联字段正确
3. **联调其余四表**：emotions → expenses → locations → inspirations 依次验证基础查询和字段返回
4. **验证所有筛选参数**：日期范围、keyword、各表特有筛选，逐个验证生效
5. **验证分页**：指定 `page_size=1`，翻到 `page=2`，确认切片逻辑正确
6. **验证空结果**：传入无记录的日期范围，确认返回 `records=[]`, `total_pages=0`
7. **验证跳转链路**：前端拿到某行的 `record_date`，跳转到 `GET /api/v1/history/daily-records/{record_date}`，确认详情页数据匹配
8. **确认详情页搜索走前端**：访问详情页后在前端做内存过滤，确认体验可接受，不需要补后端接口

## 12. 风险提醒

- **五表接口命名爆炸**：路径必须统一风格 `/history/events`, `/history/emotions` 等，不要出现 `/history/record-events`, `/history/history-emotions` 这种前缀叠加或命名不一致的情况
- **子表 user_id 绑定漏掉**：子表模型已有独立的 `user_id` 字段，但开发时容易误以为"join 了主表就安全了"，实际上必须显式绑定子表的 `user_id`，否则存在越权风险
- **expenses 金额范围筛选错误**：`min_amount / max_amount` 必须是数值类型，不要接受字符串，否则数据库比较结果不可预期。交叉校验（`min > max`）不能漏
- **inspirations 接口和历史列表的 tag 筛选混淆**：`/history/inspirations` 是"查灵感记录流"，返回每条灵感的详情及所属日期；`/history/daily-records?tag=xxx` 是"按灵感内容筛选日记录"。两者用途不同，不要合并或互相干扰
- **把五表接口做重**：这些接口是轻量表格接口，每个接口只返回该子表的核心字段，不要贪图方便把主表大字段（`summary_text`, `extra_json` 等）也塞进去
- **详情页提前塞后端搜索**：本轮已明确前端做页内过滤，不要在联调时"顺手"加 search 接口，避免范围蔓延

## 13. 最终开发顺序建议

### 第一阶段（直接开始编码）

按以下顺序实现，每个接口都做完整的 Schema + Service 方法 + Router 路由：

1. `GET /api/v1/history/events`（最典型，跑通后作为其他四个的模版）
