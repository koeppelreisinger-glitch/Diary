## 一、全局 AI 调用清单

在 Echo 项目中，共存在以下 **4 类** AI 调用场景（v1.1 将原场景 D+E 合并为单次调用）：

| # | 场景 | 触发时机 | 是否含图片 | 阻塞用户？ | 超时建议 |
|---|------|----------|-----------|-----------| ---------|
| A | **对话追问**（纯文字） | 用户发送消息 | ❌ | ✅ 同步阻塞 | 30s |
| B | **对话追问**（含图片） | 用户发送含图片消息 | ✅ | ✅ 同步阻塞 | 30s |
| C | **图片后台分析**（caption/tags） | 图片上传后异步触发 | ✅ | ❌ 后台静默 | 20s |
| D | **日记总结 + 结构化提取**（合并） | 用户点击「结束今日记录」后异步触发 | ❌ | ❌ 后台静默 | 60s |

> **v1.0 → v1.1 变更说明**：原方案将总结生成（D）与结构化提取（E）拆为两个并行 AI 调用，实际实现中已在 `DiaryAIService` 内合并为单次调用（一次性返回全量 JSON），省去一次 API 请求，并通过 `response_format=json_object` 保证结构合法。文档据此修订。

## 二、设计原则

1. **用户路径上的 AI 调用必须最快**：场景 A/B 在接口响应链路上，用流式（`stream: true`）减少首字等待。
2. **后台任务绝不串行等待**：场景 C 与 D 互相独立，能并行的一律并行。
3. **不同用户的请求天然隔离**：每个请求独立的 `httpx.AsyncClient`，无共享状态。
4. **同一用户同一时刻只有一个同步 AI 请求**：接口层通过会话状态（`recording` → `completing`）天然防止并发冲突。
5. **后台任务之间无写库竞争**：C 写 `daily_record_images`，D 写 `daily_records` 和子表，表不重叠。
6. **后台 AI 调用强制 JSON 输出**：场景 C、D 均使用 `response_format: {"type": "json_object"}`，直接 `json.loads()` 解析，不依赖字符串截取兜底。

## 三、各场景详细方案

### 3.1 场景 A — 纯文字对话追问（同步 · 流式）

```
用户发消息
    │
    ▼
[读取对话历史，取最近 20 条] ← 数据库查询（~5ms）
    │
    ▼
[POST /v1/chat/completions  stream=true]  ← 单次 AI 调用
    │   等待 AI 首 token，通常 1~3s
    ▼
[拼接流式 chunk，累计完整回复]
    │
    ▼
[写入用户消息 + AI 回复到数据库]
    │
    ▼
返回响应
```

**关键参数：**

```python
payload = {
    "model": "glm-5v-turbo",
    "messages": history[-20:] + [user_msg],  # 只取最近 20 条
    "stream": True,
    "max_tokens": 500,
    "temperature": 0.7
}
```

**为什么不并行？**
对话追问是严格顺序依赖的：必须先有用户消息，才能请求 AI，再落库 AI 回复。天然串行，无优化空间。

---

### 3.2 场景 B — 含图片对话追问（同步 · 流式）

```
用户发含图片消息
    │
    ├─[查询图片 URL]──────────────────────────────────── ~5ms
    │
    ▼
[构建 Vision 消息（image_url + text）]
    │
    ▼
[POST /v1/chat/completions  stream=true  model=glm-5v-turbo]
    │   等待 AI 首 token，通常 2~5s（Vision 略慢于纯文字）
    ▼
[拼接流式 chunk，累计完整回复]
    │
    ▼
[写入消息到数据库]
    │
    ▼
返回响应
```

**与场景 A 的区别**：消息的 `content` 字段从字符串变为列表（image_url + text），其余调用逻辑完全一致，**共用同一个流式处理函数**。

---

### 3.3 场景 C — 图片后台分析（异步 · 非流式 · 独立任务）

图片上传接口返回后，立即用 `asyncio.create_task` 触发，**完全不阻塞上传响应**。

```
upload_image() 返回 ──────────────────────────────────────────┐
                                                               │ 非阻塞
asyncio.create_task(_async_process_image(image_id, url))       │
    │                                                          │
    ▼ （后台并行运行）                                          │
┌──────────────────────────────────────────────────────────┐   │
│                                                          │   │
│  ┌────────────────────────────┐  ┌─────────────────────┐ │   │
│  │ 图像处理（Pillow）          │  │ AI Vision 调用       │ │   │
│  │  · resize → 缩略图 (~50ms) │  │  · stream=false     │ │   │
│  │  · quantize → 主色调       │  │  · response_format  │ │   │
│  │  （纯 CPU，线程池执行）     │  │    =json_object     │ │   │
│  └────────────┬───────────────┘  │  · 等待 ~3~8s       │ │   │
│               │                  └────────┬────────────┘ │   │
│               └───────────────┬───────────┘              │   │
│                               ▼ 两者完成后                │   │
│                        一次性写库                         │   │
│               (ai_caption / ai_tags / thumbnail_url)     │   │
└──────────────────────────────────────────────────────────┘   │
                                                    用户已看到上传结果
```

> **v1.1 优化**：图像处理（步骤1/2）与 AI Vision 调用（步骤3）改为 `asyncio.gather` 并行执行。前提：AI 调用使用图片 **URL**（而非 base64），两者之间无依赖关系，可完全重叠，总耗时从「处理 + AI」变为 `max(处理, AI)` ≈ AI 时间。

**为什么用 `stream=false`？**
后台分析需要完整的 JSON 结构（`caption` + `tags`），流式拼接反而增加解析复杂度。非流式一次性拿到结果，代码更简洁稳定。

**关键参数：**

```python
payload = {
    "model": "glm-5v-turbo",
    "messages": [vision_msg],
    "stream": False,
    "max_tokens": 200,
    "temperature": 0.3,
    "response_format": {"type": "json_object"}   # 强制合法 JSON
}
```

### 3.4 场景 D — 日记总结 + 结构化提取（异步 · 单次 AI 调用）

> **v1.1 关键修订**：原方案（v1.0）拆为两次并行 AI 调用（总结 + 提取）。实际实现（`DiaryAIService`）已合并为**单次调用**，一次性返回包含总结与结构化数据的完整 JSON。

#### 3.4.1 整体时序

```
用户点击「结束今日记录」
    │
    ▼
[conversations.status → completing]  ← 同步，立即返回给前端
    │
    ▼ （asyncio.create_task，后台执行）
┌──────────────────────────────────────────────────────────┐
│                                                          │
│  读取全量对话消息（一次数据库查询）                        │
│      │                                                   │
│      ▼                                                   │
│  POST /v1/chat/completions                               │
│      model=glm-5v-turbo                                  │
│      stream=false                                        │
│      max_tokens=1000                                     │
│      response_format=json_object                         │
│      等待 ~5~15s                                         │
│      │                                                   │
│      ▼ 单次返回全量 JSON：                                │
│      {                                                   │
│        "body_text": "...",                               │
│        "summary_text": "...",                            │
│        "emotion_overall_score": 8,                       │
│        "keywords": [...],                                │
│        "events": [...],                                  │
│        "emotions": [...],                                │
│        "expenses": [...],                                │
│        "locations": [...],                               │
│        "inspirations": [...]                                     │
│      }                                                   │
│      │                                                   │
│      ▼                                                   │
│  在同一事务中写库：                                       │
│      · daily_records（主表）                             │
│      · record_events / emotions / expenses               │
│      · record_locations / record_inspirations                    │
│      · conversations.status → completed                  │
└──────────────────────────────────────────────────────────┘
```

#### 3.4.2 Prompt 设计（合并后）

```python
COMBINED_SYSTEM = """
你是一个温暖的日记助手。根据用户今日的对话内容，严格按如下 JSON 结构返回，不加任何额外文字：
{
  "body_text":             "完整日记正文（200~400字，温柔叙述，第三人称）",
  "summary_text":          "精简摘要（50~80字）",
  "emotion_overall_score": 8,
  "keywords":              ["关键词1", "关键词2", "关键词3"],
  "events":                [{"content": "事件描述"}],
  "emotions":              [{"emotion_label": "开心", "intensity": 4}],
  "expenses":              [{"amount": 32.0, "currency": "CNY", "category": "餐饮", "description": "咖啡"}],
  "locations":             [{"name": "地点名称"}],
  "inspirations":                  [{"content": "灵感"}]
}
没有的字段返回空数组，emotion_overall_score 取 1~10 整数。
"""
```

#### 3.4.3 Python 实现（核心逻辑，见 `DiaryAIService`）

```python
async def build_record_payload_from_messages(
    self,
    messages: list[ConversationMessage]
) -> dict:
    plain_messages = [
        {"role": m.role, "content": m.content}
        for m in messages
    ]
    payload = {
        "model": settings.TOKENHUB_MODEL,          # "glm-5v-turbo"
        "messages": [
            {"role": "system", "content": COMBINED_SYSTEM}
        ] + plain_messages,
        "stream": False,
        "max_tokens": 1000,
        "temperature": 0.5,
        "response_format": {"type": "json_object"} # 强制合法 JSON
    }
    headers = {
        "Authorization": f"Bearer {settings.TOKENHUB_API_KEY}",
        "Content-Type": "application/json"
    }
    async with httpx.AsyncClient(timeout=60.0) as client:
        resp = await client.post(settings.TOKENHUB_BASE_URL, headers=headers, json=payload)
        resp.raise_for_status()
        raw = resp.json()["choices"][0]["message"]["content"]
        return json.loads(raw)   # response_format 保证此处不会抛 JSONDecodeError
```

**为什么 v1.1 不再拆成两次并行调用？**

| 对比项 | v1.0（D+E 并行） | v1.1（D+E 合并） |
|--------|-----------------|-----------------|
| API 调用次数 | 2 次 | 1 次 |
| 总等待时间 | max(D, E) ≈ 10~15s | 单次 ≈ 10~15s（相近） |
| JSON 解析 | 两段独立 JSON，各有兜底 | 一段，`response_format` 保证合法 |
| Prompt 维护 | 两套 system prompt | 一套 |
| API 配额消耗 | 2x | 1x |

> 总结：合并后等待时间基本不变（AI 生成量相近），但省去一次网络往返，降低配额消耗，且解析更稳定。

## 四、串并行决策树（快速参考）

```
有新的 AI 调用需求？
    │
    ├─ 用户在等待响应？
    │       │
    │       ├─ YES → 同步路径，使用 stream=true，超时 30s
    │       │         └─ 含图片？→ Vision 格式（image_url + text）
    │       │         └─ 纯文字？→ 普通 messages 格式
    │       │
    │       └─ NO  → 后台异步路径
    │                  │
    │                  ├─ 图片分析（场景 C）
    │                  │   → asyncio.create_task
    │                  │   → 图像处理 ‖ AI Vision（asyncio.gather）
    │                  │   → stream=false，response_format=json_object，超时 20s
    │                  │
    │                  └─ 日记生成（场景 D，合并版）
    │                      → asyncio.create_task
    │                      → 单次 AI 调用，response_format=json_object
    │                      → 完成后在同一事务写全部子表
    │
    └─ 与已有后台任务有写库冲突？
            │
            ├─ YES → 检查写入的表是否重叠
            │         重叠 → 改为串行，或合并为单次 AI 调用
            │
            └─ NO  → 放心并行
```

## 五、防打架规则

| 规则 | 说明 |
|------|------|
| **场景 A/B 互斥** | 同一会话 `recording` 状态下，接口层天然串行（客户端逐条发送，后端同步等待 AI）|
| **C 与 A/B 不冲突** | C 写 `daily_record_images`，A/B 写 `conversation_messages`，表不重叠 |
| **C 与 D 不冲突** | C 的 image_id 写入完成后，D 才可能被触发；即使并发，写入的行不同 |
| **D 内部无竞争** | 合并为单次调用后，主表与子表在同一事务顺序写入（flush → add 子表 → commit），外键约束保证一致性 |
| **重试幂等** | D 失败重试时，先检查 `daily_records` 是否已存在对应 `conversation_id`，避免重复写入 |
| **不共享 httpx.AsyncClient** | 每个 AI 调用创建独立的 `AsyncClient`，避免连接池竞争和状态污染 |

## 六、超时与降级策略

| 场景 | 超时 | 降级行为 |
|------|------|---------|
| A — 纯文字对话 | 30s | 返回 `504`，前端提示「AI 响应超时，请重试」；用户消息已入库，下次重试作为新消息 |
| B — 含图片对话 | 30s | 同上 |
| C — 图片分析 | 20s | 静默失败，`ai_caption / ai_tags` 保持 `NULL`；不影响用户查看图片 |
| D — 日记总结+提取 | 60s | 保持 `completing` 状态，3次退避重试（5s / 15s / 30s）；全部失败后告警；子表为空时前端优雅降级显示「结构化数据暂无」|
