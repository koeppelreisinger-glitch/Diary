# 《03_认证与用户模块》后端设计文档

> 本文档为「回响（Echo）」MVP 阶段的认证与用户模块设计文档。  
> 关联文档：`docs/01_后端全局约束与命名规范.md`、`docs/02_数据库总览与关系设计初稿.md`  
> 本文档不包含代码、ORM、SQL，仅输出接口设计与模块规范。

---

## 1. 模块目标

本模块负责用户身份的建立、验证与基础信息管理，是整个系统所有受鉴权接口的前置依赖。

具体职责包括：
- 提供手机号（或邮箱）与密码的注册与登录能力
- 签发 JWT Token，作为后续所有受保护接口的鉴权凭据
- 提供当前用户基础信息的查询与编辑能力
- 维护账户状态（正常、封禁）

本模块不包含用户偏好配置（如时区、AI 伴侣设定），该部分归属于设置模块（`user_settings`）。

---

## 2. 涉及的数据表

本模块直接操作以下数据表：

| 表名 | 读/写 | 说明 |
|------|-------|------|
| `users` | 读写 | 账户主表，负责注册写入与登录查询 |
| `user_settings` | 写（新用户注册时初始化） | 建议在注册成功后同步初始化默认 `user_settings`；若初始化失败，整笔注册事务回滚。 |

## 3. `users` 表结构建议（设计说明）

以下为字段级别的设计说明，不含 SQL 或 ORM 代码。

| 字段名 | 类型建议 | 说明 |
|--------|----------|------|
| `id` | UUID | 主键，使用 UUID v4，不对外暴露自增序列 |
| `phone` | VARCHAR(20) | 主要登录凭据，建议设唯一约束；MVP 阶段注册必填，不允许省略 |
| `email` | VARCHAR(100) | 备用登录凭据，允许为空；设唯一约束（预留字段，MVP 阶段不启用） |
| `hashed_password` | VARCHAR(255) | 使用 bcrypt 哈希后的密码，禁止明文存储 |
| `nickname` | VARCHAR(50) | 用户昵称，初始可由系统随机生成；AI 在对话中使用此称谓 |
| `avatar_url` | VARCHAR(255) | 用户头像的 OSS 存储路径；MVP 阶段可为空，默认使用系统占位图 |
| `status` | VARCHAR(20) | 账户状态。落库及 API 返回值统一为小写字符串：`active`（正常）、`banned`（封禁）。代码层枚举成员名可使用大写（如 `UserStatus.ACTIVE`），但实际写入值与接口返回值均为小写。 |
| `last_login_at` | TIMESTAMPTZ | 最近一次登录时间，每次登录成功时更新 |
| `created_at` | TIMESTAMPTZ | 注册时间，系统自动写入 |
| `updated_at` | TIMESTAMPTZ | 最近更新时间，系统自动维护 |
| `deleted_at` | TIMESTAMPTZ | 软删除标记，默认为 NULL |

**设计权衡说明：**

> **关于 `phone` 必填与 `email` 扩展字段的边界说明**  
> MVP 阶段注册接口要求 `phone` 字段必填，不允许两者同时为空。`email` 字段保留仅为后续扩展预留结构位，当前注册流程不启用。接口参数验证层须在入参阶段强制校验 `phone` 的存在性，不依赖数据库约束兜底。

> **关于 `phone` 与 `email` 字段均允许为空的数据库层设计**  
> 原因：MVP 阶段以手机号登录为主，但后续可能引入邮箱登录。若两个字段均设为 NOT NULL，则后续兼容扩展时需要大量历史数据补填。  
> 建议：两个字段在数据库层均允许为空，但在应用层约束"至少有一项不为空"，同时各自加唯一约束（部分索引：`WHERE deleted_at IS NULL`），避免软删除影响唯一性判定。

## 4. 认证相关字段建议

| 字段 | 位置 | 说明 |
|------|------|------|
| `hashed_password` | `users` 表 | 采用 bcrypt 算法加密，不可逆 |
| `status` | `users` 表 | 登录时须校验该字段，`banned` 状态应拒绝签发 Token |
| `last_login_at` | `users` 表 | 每次成功登录后更新，可用于后续会话异常检测 |

**Token 方案见第 7 节。**

## 5. 接口清单

| 序号 | 方法 | 路径 | 是否需要鉴权 | 说明 |
|------|------|------|--------------|------|
| 01 | POST | `/api/v1/auth/register` | 否 | 注册新账户 |
| 02 | POST | `/api/v1/auth/login` | 否 | 登录并获取 Token |
| 03 | GET | `/api/v1/users/me` | 是 | 获取当前用户基础信息 |
| 04 | PUT | `/api/v1/users/me` | 是 | 修改当前用户基础信息（昵称、头像） |

> 说明：注册与登录逻辑分离为两个接口，而非合并为"登录或注册"的隐式自动注册模式。原因是隐式注册模式会导致前端无法区分当前是首次注册还是重复登录，无法展示新手引导流程。若后续产品要求合并，可在不破坏接口结构的前提下调整业务逻辑。

## 6. 接口详情

---

### 6.1 注册

**路径**：`POST /api/v1/auth/register`  
**鉴权**：不需要

#### 请求参数（Body JSON）

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `phone` | string | **是** | 手机号，11 位纯数字。MVP 阶段注册必填，不允许省略。 |
| `password` | string | 是 | 原始密码，长度 8~50 位，后端加密存储 |
| `nickname` | string | 否 | 昵称；不传则系统随机生成 |

> **说明**：`email` 字段在注册接口中不开放，当前版本不接受该参数。

#### 请求示例

```json
{
  "phone": "13800138000",
  "password": "MyPassword123",
  "nickname": "云淡"
}
```

#### 返回参数

| 字段 | 类型 | 说明 |
|------|------|------|
| `user_id` | string (UUID) | 注册成功后生成的用户 ID |
| `nickname` | string | 最终使用的昵称（系统生成或用户传入） |

#### 返回示例（成功）

```json
{
  "code": 20000,
  "message": "success",
  "data": {
    "user_id": "e7a1f3c2-84b0-4d1e-9a23-1c5d8e2f7b90",
    "nickname": "云淡"
  },
  "trace_id": "req-reg-001"
}
```

#### 错误情况

| HTTP 状态码 | 内部 code | 说明 |
|-------------|-----------|------|
| 400 | 40001 | 手机号格式错误或密码不符合规则 |
| 409 | 40901 | 手机号已被注册 |

---

### 6.2 登录

**路径**：`POST /api/v1/auth/login`  
**鉴权**：不需要

#### 请求参数（Body JSON）

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `phone` | string | 是 | 注册时使用的手机号 |
| `password` | string | 是 | 原始密码 |

#### 请求示例

```json
{
  "phone": "13800138000",
  "password": "MyPassword123"
}
```

#### 返回参数

| 字段 | 类型 | 说明 |
|------|------|------|
| `access_token` | string | JWT Token，后续所有受保护接口均需携带 |
| `token_type` | string | 固定值 `"bearer"` |
| `expires_in` | integer | Token 有效期，单位：秒 |

#### 返回示例（成功）

```json
{
  "code": 20000,
  "message": "success",
  "data": {
    "access_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
    "token_type": "bearer",
    "expires_in": 2592000
  },
  "trace_id": "req-login-001"
}
```

#### 错误情况

| HTTP 状态码 | 内部 code | 说明 |
|-------------|-----------|------|
| 400 | 40001 | 请求参数缺失或格式错误 |
| 401 | 40102 | 手机号或密码错误 |
| 403 | 40301 | 账户处于 `banned` 状态，拒绝登录 |

---

### 6.3 获取当前用户信息

**路径**：`GET /api/v1/users/me`  
**鉴权**：需要（`Authorization: Bearer <token>`）

#### 请求参数

无请求体。用户身份从 Token 解析获取，不接受通过 URL 或 Body 传入 `user_id`。

#### 返回参数

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | string (UUID) | 用户唯一 ID |
| `phone` | string | 手机号（脱敏，中间四位替换为 `****`） |
| `nickname` | string | 当前昵称 |
| `avatar_url` | string \| null | 头像地址，无头像时为 null |
| `status` | string | 账户状态，返回值为小写：`active` 或 `banned` |
| `created_at` | string (ISO 8601) | 注册时间 |
| `last_login_at` | string \| null | 最近登录时间 |

#### 返回示例（成功）

```json
{
  "code": 20000,
  "message": "success",
  "data": {
    "id": "e7a1f3c2-84b0-4d1e-9a23-1c5d8e2f7b90",
    "phone": "138****8000",
    "nickname": "云淡",
    "avatar_url": null,
    "status": "active",
    "created_at": "2024-05-20T10:30:00Z",
    "last_login_at": "2024-06-01T08:15:22Z"
  },
  "trace_id": "req-me-001"
}
```

#### 错误情况

| HTTP 状态码 | 内部 code | 说明 |
|-------------|-----------|------|
| 401 | 40101 | Token 未携带 |
| 401 | 40102 | Token 已过期或无效 |

---

### 6.4 修改当前用户信息

**路径**：`PUT /api/v1/users/me`  
**鉴权**：需要（`Authorization: Bearer <token>`）

> 使用 `PUT` 语义，本接口仅支持对 `nickname` 和 `avatar_url` 两个字段的覆盖更新。调用时若某字段不传，则该字段保持不变；若显式传入 `null`，则视为清空该字段（`avatar_url` 适用此逻辑，`nickname` 不允许为 null）。

#### 请求参数（Body JSON，全部可选）

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `nickname` | string | 否 | 新昵称，长度 1~50 字符，不允许为 null |
| `avatar_url` | string | 否 | 新头像地址；头像文件上传逻辑由独立的文件上传接口处理，此处只接收已上传后的 URL；可传 null 以清空头像 |

#### 请求示例

```json
{
  "nickname": "霜降"
}
```

#### 返回参数

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | string (UUID) | 用户 ID |
| `nickname` | string | 更新后的昵称 |
| `avatar_url` | string \| null | 更新后的头像地址 |
| `updated_at` | string (ISO 8601) | 本次更新时间 |

#### 返回示例（成功）

```json
{
  "code": 20000,
  "message": "success",
  "data": {
    "id": "e7a1f3c2-84b0-4d1e-9a23-1c5d8e2f7b90",
    "nickname": "霜降",
    "avatar_url": null,
    "updated_at": "2024-06-02T09:00:00Z"
  },
  "trace_id": "req-update-001"
}
```

#### 错误情况

| HTTP 状态码 | 内部 code | 说明 |
|-------------|-----------|------|
| 400 | 40001 | 字段格式不合规（如昵称超长或为 null） |
| 401 | 40102 | Token 无效或过期 |

## 7. Token 方案建议

**方案**：采用 JWT（JSON Web Token），签发后存储于客户端，服务端无状态验证。

| 参数 | 建议值 | 说明 |
|------|--------|------|
| 算法 | HS256 | 对称签名算法，适合单体服务 |
| 有效期 | 30 天 | MVP 阶段，减少用户频繁重新登录的摩擦感 |
| Payload 内容 | `user_id`、`exp`（过期时间） | 不在 Token 中存储敏感信息（如密码、手机号） |
| 携带方式 | `Authorization: Bearer <token>` | 标准 HTTP 头部，全局统一 |
| 验证位置 | FastAPI 全局依赖注入（`Depends`） | 不在各路由函数内部单独验证 |

**设计权衡说明：**

> **关于是否引入 Refresh Token 双 Token 机制**  
> 建议：MVP 阶段不引入。双 Token 机制需要维护 Refresh Token 的服务端存储（Redis 或数据库），以及 Token 轮换的完整逻辑，开发成本较高。  
> 原因：30 天的 Access Token 有效期在 C 端产品中已足够覆盖大部分用户的正常使用周期。若后续有安全审计要求，再引入 Token 吊销机制（黑名单策略）即可。

## 8. 用户信息接口设计建议

1. **身份来源唯一化**：所有需要当前用户身份的接口，`user_id` 必须且只能从 JWT Token 解析获取，禁止通过 URL 路径（如 `/api/v1/users/{user_id}`）或请求体传入用户 ID。这是防止越权访问（IDOR）的基本措施。

2. **手机号脱敏**：`GET /api/v1/users/me` 返回的手机号须在应用层进行中间四位脱敏处理（如 `138****8000`），不返回完整手机号。

3. **密码修改接口独立**：本文档中不包含修改密码的接口，建议将其作为独立接口（如 `POST /api/v1/auth/change-password`）在后续安全模块中设计，避免与基础资料编辑接口耦合。

4. **软删除用户**：当某账户执行注销操作时，对 `users` 表的 `deleted_at` 字段写入当前时间即可，不做物理删除；对应的该手机号可通过部分索引（`WHERE deleted_at IS NULL`）的唯一约束允许重新注册。

## 9. MVP 优先级

| 优先级 | 接口 | 原因 |
|--------|------|------|
| P0（必须先做） | `POST /api/v1/auth/register` | 没有注册则系统无法启动 |
| P0（必须先做） | `POST /api/v1/auth/login` | 没有登录则所有鉴权接口无法验证 |
| P0（必须先做） | `GET /api/v1/users/me` | 所有页面渲染当前用户信息依赖此接口 |
| P1（第一版应完成） | `PUT /api/v1/users/me` | 用户昵称是 AI 对话中的称呼来源，需在聊天前可配置 |

## 11. 风险点与实现建议

### 风险 1：密码明文传输
- **风险描述**：前端若通过 HTTP 明文传输原始密码，存在中间人攻击风险。
- **建议**：生产环境必须启用 HTTPS。开发测试阶段可暂时豁免，但部署前须强制落地。

### 风险 2：弱密码策略缺失
- **风险描述**：若不设置最低密码复杂度要求，用户可能设置极简密码（如 `12345678`），导致账户安全性极低。
- **建议**：在接口参数验证层设置密码规则（如：长度至少 8 位，须包含字母与数字的组合），通过 Pydantic 的字段校验器在入参阶段拦截。

### 风险 3：注册接口被批量调用（撞库/注册滥用）
- **风险描述**：注册接口无鉴权保护，可能被脚本批量请求，消耗服务器资源或制造大量无效账户。
- **建议**：MVP 阶段可通过 IP 频率限制（Rate Limiting）中间件做基础防护，后续可升级为短信验证码校验。

### 风险 4：Token 无法主动失效
- **风险描述**：JWT 的无状态特性意味着一旦签发，在过期前无法主动吊销（如用户修改密码后，旧 Token 仍然有效）。
- **建议**：MVP 阶段可接受此限制，但需在文档中记录此已知约束。若后续出现账户安全事件，可通过引入 Token 黑名单（存储于 Redis）的方式补救，无需修改接口结构。

### 风险 5：账户状态校验遗漏
- **风险描述**：若 Token 验证中间件只校验签名有效性而不检查 `users.status`，则被封禁的用户仍可凭旧 Token 访问受保护接口。
- **建议**：在鉴权依赖注入逻辑中，完成 Token 签名验证后，须额外查询数据库确认当前用户的 `status` 字段值为 `active`，否则返回 `403`。代码层可使用枚举常量（如 `UserStatus.ACTIVE`）进行比对，但比对目标的实际值应为小写字符串 `"active"`，与落库值保持一致。
