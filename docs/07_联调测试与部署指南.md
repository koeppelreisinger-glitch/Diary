# 测试联调跑通：从聊天到生成总结闭环

本文档带你完整游走一遍 **新建对话 -> 产生记录 -> 完结会话 -> 查看总结** 的数据流。

## 准备工作
确保服务器正在运行：
```bash
.venv\Scripts\uvicorn.exe app.main:app --host 127.0.0.1 --port 8000 --reload
```
打开浏览器访问：`http://127.0.0.1:8000/docs`
并在顶部 `Authorize` 按钮输入已注册账号（例如前期的 13900001111 手机号）完成授权。

## 联调核心步骤（4步）

### 第一步：创建今天会话
- **路径**：`POST /api/v1/conversations`
- **操作**：直接点击 `Execute`。
- **预期结果**：
  返回类似于下方的成功响应，状态确保是 `recording`，并请**复制保存这个 `id` 值**。
  ```json
  {
    "code": 20000,
    "message": "success",
    "data": {
      "id": "e456d9a4-1234-abcd-8f90-111122223333",
      "status": "recording",
      "record_date": "2024-06-01",
      "created_at": "2024-06-01T10:00:00Z"
    }
  }
  ```

### 第二步：发生聊天互动，传递有效数据源信息
- **路径**：`POST /api/v1/conversations/{conversation_id}/messages`
- **操作**：
  1. 将第一步的 `id` 填入 `conversation_id` 框。
  2. 在 `Request body` 中粘贴：
  ```json
  {
    "content_type": "text",
    "content": "今天很累，开了很多会，还买了一杯咖啡花了32元"
  }
  ```
  3. 点击 `Execute`。
- **预期结果**：
  同时返回你的输入内容 (`user_message`) 及 AI 虚拟的占位反馈 (`ai_message`)。这是总结词条拉取信息的核心。

### 第三步：完结对话，触发后台结构化运算入库
- **路径**：`POST /api/v1/conversations/{conversation_id}/complete`
- **操作**：
  保持上文那个 `conversation_id` 填入，直接 `Execute`。
- **预期结果**：
  服务端会先把会话置为 `completing`，并把总结生成放入后台任务。成功返回 `20000`，但 `daily_record` 可能是 `null`，这属于正常现象。随后通过第四步轮询今日记录，直到生成完成。

### 第四步：从日历首页拉取核实数据完整度
- **路径**：`GET /api/v1/daily-records/today`
- **操作**：直接 `Execute`。
- **预期结果**：
  返回带有今日综合数据的查询体，且 `has_record` 为 `true`，`is_generating` 为 `false`。
  内部的 `record` 对象里不仅有主文，还带出了 `events`, `emotions`, `expenses`, `locations`, `inspirations` 等丰富明细。至此闭环彻底通顺。

# Echo 详细联调操作教程

本文面向前后端联调、演示验收和问题排查。内容已按当前仓库真实代码整理，重点覆盖认证、用户设置、今日对话、AI 生成日记、历史查询、图片上传和回忆功能。

## 1. 当前系统口径

### 1.1 技术栈与入口

- 后端：FastAPI，入口为 `app/main.py`
- API 前缀：`/api/v1`
- 数据库：PostgreSQL + SQLAlchemy AsyncSession + asyncpg
- AI 服务：TokenHub Chat Completions，统一封装在 `TokenHubChatService`
- 前端：`frontend` 目录下的静态 HTML/CSS/JS，通用请求封装在 `frontend/js/api.js`
- 本地静态资源：`/frontend` 和 `/uploads`，Vercel 环境下不挂载本地上传目录

### 1.2 关键行为差异

- `POST /api/v1/conversations/{conversation_id}/complete` 只负责把会话置为 `completing`，并把总结生成放入后台任务。接口会快速返回，`daily_record` 通常为 `null`。
- 前端或联调人员需要轮询 `GET /api/v1/daily-records/today`。当 `is_generating=true` 时说明后台仍在生成；当 `has_record=true` 时说明日记已落库。
- 新注册用户默认时区是 `UTC`，建议联调前先用 `PUT /api/v1/settings` 改为本地时区，例如 `Asia/Shanghai`。
- 已完成会话继续补写时，`is_supplement` 是消息请求体字段，不是 query 参数。
- 当前结构化五类子表是 `events / emotions / expenses / locations / inspirations`，没有独立的 `tags` 接口。历史页里“标签/灵感云”对应的是 `inspirations`。
- 轻量编辑接口使用 `inspirations_to_add` 和 `inspirations_to_remove`，不是 `tags_to_add` / `tags_to_remove`。

## 2. 联调目标

完整闭环如下：

1. 注册和登录，取得 JWT。
2. 设置用户时区和偏好。
3. 创建今日会话。
4. 发送文本或图文消息，并收到 AI 追问。
5. 结束今日会话，触发后台日记生成。
6. 查询今日记录，确认主日记和五类结构化数据已生成。
7. 查询历史列表、日历、时间轴和五表主视图。
8. 可选验证补写、正文重建、轻量编辑、图片上传和回忆功能。

## 3. 启动与准备

### 3.1 启动后端

在项目根目录执行：

```bash
.venv\Scripts\uvicorn.exe app.main:app --host 127.0.0.1 --port 8000 --reload
```

启动后访问：

- 健康检查：`http://127.0.0.1:8000/health`
- Swagger：`http://127.0.0.1:8000/docs`
- 本地前端：`http://127.0.0.1:8000/frontend/login.html`

### 3.2 环境变量

联调至少需要确认：

- `DATABASE_URL` 或本地 PostgreSQL 连接项可用。
- `SECRET_KEY` 不应使用生产默认值。
- 如果要测试真实 AI，配置 `TOKENHUB_AUTHORIZATION` 或 `TOKENHUB_API_KEY`。
- 如果未配置 AI 鉴权，总结服务会在部分场景使用本地 fallback 规则兜底，但真实效果会弱很多。

### 3.3 推荐测试文本

为了同时覆盖事件、情绪、消费、地点和灵感，可以发送：

```json
{
  "content_type": "text",
  "content": "今天在公司开了很多会，还见了客户，整个人有点累。",
  "is_supplement": false
}
```

```json
{
  "content_type": "text",
  "content": "晚上去咖啡店买了一杯咖啡花了32元，回家路上想到以后要把每天的小事都记下来。",
  "is_supplement": false
}
```

## 4. 认证与授权

### 4.1 注册

接口：`POST /api/v1/auth/register`

示例：

```json
{
  "phone": "13900001111",
  "password": "Password123!",
  "nickname": "联调测试"
}
```

预期：

- HTTP `200`
- `code=20000`
- 返回脱敏手机号、昵称、用户 ID 等信息
- 后端会自动创建一条默认 `UserSetting`

### 4.2 登录

接口：`POST /api/v1/auth/login`

示例：

```json
{
  "phone": "13900001111",
  "password": "Password123!"
}
```

预期：

- 返回 `access_token`
- `token_type=bearer`
- 前端会把 token 存入 `localStorage.echo_token`

Swagger 可使用右上角 `Authorize`。项目提供了隐藏的 `swagger_login` 适配 Swagger 的表单授权格式。

### 4.3 验证当前用户

接口：`GET /api/v1/users/me`

预期：

- 已授权时返回当前用户信息。
- 未授权或 token 失效时返回 `401`，前端会清空 token 并跳回登录页。

## 5. 用户设置

### 5.1 查询设置

接口：`GET /api/v1/settings`

用于确认时区、输入偏好、提醒设置、深色模式等用户偏好。

### 5.2 修改设置

接口：`PUT /api/v1/settings`

推荐联调请求：

```json
{
  "timezone": "Asia/Shanghai",
  "input_preference": "text",
  "reminder_enabled": false,
  "is_dark_mode": false
}
```

注意：

- 今日会话归属日期依赖用户时区。
- 如果开启 `reminder_enabled=true`，通常需要同时传合法 `reminder_time`。

## 6. 今日对话主链路

### 6.1 查询今日会话

接口：`GET /api/v1/conversations/today`

无今日会话时：

```json
{
  "has_today": false,
  "conversation": null
}
```

已有会话时记录 `conversation.id`，后续称为 `conversation_id`。

### 6.2 创建今日会话

接口：`POST /api/v1/conversations`

请求体为空。

预期：

- 返回 `status=recording`
- 返回 `record_date`
- 服务端会写入一条 `system` 消息作为 AI 对话提示词
- 如果当天已有会话，当前服务实现会返回已有会话信息，避免重复创建导致 500

### 6.3 发送消息

接口：`POST /api/v1/conversations/{conversation_id}/messages`

示例：

```json
{
  "content_type": "text",
  "content": "今天在公司开了很多会，还见了客户，整个人有点累。",
  "is_supplement": false,
  "mode": "chat"
}
```

预期：

- 服务端先保存用户消息。
- 然后把当前会话消息上下文交给 `AICompanionService`。
- AI 回复生成后保存为下一条消息。
- 响应中同时返回 `user_message` 和 `ai_message`。

字段说明：

- `content_type` 当前主要使用 `text`。
- `image_url` 可传已上传图片 URL。
- `mode` 可选：`expense / inspiration / learning / chat`，用于快速记录场景。
- `is_supplement=true` 时，允许向已 `completed` 的会话继续补写。

### 6.4 拉取消息列表

接口：`GET /api/v1/conversations/{conversation_id}/messages?limit=100`

预期：

- `messages` 按 `sequence_number` 升序返回。
- 包含系统消息，因此纯聊天两轮后，总数通常会大于 4。
- 前端展示时主要关心 `role=user/ai` 的消息。

### 6.5 结束今日记录

接口：`POST /api/v1/conversations/{conversation_id}/complete`

当前真实流程：

1. 校验会话归属当前用户。
2. 校验状态不能是 `completing` 或 `completed`。
3. 将会话状态改为 `completing`。
4. 注册后台任务 `run_background_summary`。
5. 快速返回 `conversation_id / status / updated_at / daily_record=null`。
6. 后台任务读取全部消息，调用 AI 生成日记 payload。
7. 落库 `daily_records` 主表和五类子表。
8. 将会话状态改为 `completed`。

因此联调时不要只看 `complete` 的响应判断是否完成，要继续查今日记录。

## 7. 日记生成与查询

### 7.1 查询今日记录

接口：`GET /api/v1/daily-records/today`

可能状态：

- 还在记录：`has_record=false, is_generating=false`
- 正在生成：`has_record=false, is_generating=true`
- 已完成：`has_record=true, is_generating=false, record={...}`

建议轮询间隔：2 到 5 秒。

完成后重点检查：

- `record.body_text`：完整日记正文。
- `record.summary_text`：列表摘要。
- `record.emotion_overall_score`：1 到 10 的整体情绪分。
- `record.keywords`：最多 8 个关键词。
- `events`：事件。
- `emotions`：情绪标签和强度。
- `expenses`：金额、币种、分类、描述。
- `locations`：地点。
- `inspirations`：灵感或感悟。

### 7.2 手动补录过往日记

接口：`POST /api/v1/daily-records/manual`

示例：

```json
{
  "record_date": "2026-04-20",
  "body_text": "今天补录一篇过去的日记，上午整理资料，晚上散步，心情比较平静。"
}
```

预期：

- 指定日期不能已有记录。
- 后端会创建一个 `completed` 状态的会话。
- 根据正文同步提取结构化数据并返回完整详情。

### 7.3 手动修改正文并重建

接口：`PUT /api/v1/daily-records/{record_id}/body`

示例：

```json
{
  "body_text": "今天主要在公司开会，也见了客户。晚上去咖啡店买了咖啡，回家后还是感觉有点累。"
}
```

预期：

- 更新 `body_text`。
- 同步重新生成 `summary_text / emotion_overall_score / keywords`。
- 软删除旧的五类子表条目。
- 写入新的五类结构化数据。
- 整个过程失败则回滚。

### 7.4 保存补写并重建

接口：`POST /api/v1/daily-records/{record_id}/supplement`

示例：

```json
{
  "note": "本次补充来自今日页补写"
}
```

预期：

- 后端从关联会话中读取完整消息历史。
- 重新生成日记 payload。
- 全量重建主表摘要和五类子表。
- 如果没有消息历史，则基于现有 `body_text` 重新提取。

### 7.5 轻量编辑

接口：`PUT /api/v1/daily-records/{record_id}`

示例：

```json
{
  "user_note": "这天工作密度很高",
  "keywords": ["工作", "会议", "咖啡"],
  "inspirations_to_add": ["工作虽忙，但值得复盘"],
  "inspirations_to_remove": []
}
```

预期：

- 更新备注和关键词。
- 增删灵感条目。
- 不触发正文和结构化数据重建。

## 8. 历史记录接口

### 8.1 历史列表

接口：`GET /api/v1/history/daily-records`

常用参数：

- `page`
- `page_size`
- `start_date`
- `end_date`
- `keyword`
- `tag`，实际按 `inspirations.content` 精准筛选。
- `min_emotion_score`
- `max_emotion_score`

日期支持：

- `YYYY-MM-DD`
- `YYYY/MM/DD`
- `YYYY.MM.DD`
- `YYYYMMDD`

### 8.2 指定日期详情

接口：`GET /api/v1/history/daily-records/{record_date}`

示例：

- `/api/v1/history/daily-records/2026-04-20`
- `/api/v1/history/daily-records/20260420`

预期返回完整 `DailyRecordDetailResponse`，包含五类子表。

### 8.3 日历视图

接口：`GET /api/v1/history/calendar?year=2026&month=4`

用于前端绘制某月有记录的日期点、摘要预览和情绪分。

### 8.4 时间轴

接口：`GET /api/v1/history/timeline?start_date=2026-04-01&end_date=2026-04-30&limit=100`

后端按月分组返回，适合历史流式浏览。

### 8.5 五表主视图

事件：

```text
GET /api/v1/history/events?start_date=2026-04-01&end_date=2026-04-30&keyword=开会
```

灵感：

```text
GET /api/v1/history/inspirations?start_date=2026-04-01&end_date=2026-04-30&keyword=复盘
```

情绪：

```text
GET /api/v1/history/emotions?start_date=2026-04-01&end_date=2026-04-30&emotion_label=疲惫&min_intensity=1&max_intensity=5
```

地点：

```text
GET /api/v1/history/locations?start_date=2026-04-01&end_date=2026-04-30&name=咖啡
```

消费：

```text
GET /api/v1/history/expenses?start_date=2026-04-01&end_date=2026-04-30&category=餐饮&min_amount=1&max_amount=100
```

所有五表接口都返回：

- `total_count`
- `total_pages`
- `current_page`
- `page_size`
- `records`

## 9. 图片与回忆功能

### 9.1 上传图片

接口：`POST /api/v1/media/upload`

请求格式：`multipart/form-data`

字段：

- `file`：图片文件。
- `conversation_id`：可选，关联当前会话。

限制：

- 支持 `image/jpeg / image/png / image/webp / image/gif`
- 最大 10MB

预期：

- 立即返回图片 `id / url / width / height / file_size`
- 后台异步生成缩略图、主色调和 AI 图片描述
- 本地环境 URL 为 `/uploads/...`
- Vercel 环境当前本地文件不持久化，`url` 可能为空，后续应接云存储

### 9.2 历史图片流

接口：`GET /api/v1/media/history`

参数：

- `start_date`
- `end_date`
- `page`
- `page_size`

返回按日期倒序分组的图片列表。

### 9.3 指定日期图片

接口：`GET /api/v1/media/history/{record_date}`

用于日记详情页展示当天图片。

### 9.4 历史上的今天

接口：`GET /api/v1/media/on-this-day`

返回跨年同月同日的历史图片，用于回忆版块。

### 9.5 删除图片

接口：`DELETE /api/v1/media/{image_id}`

当前为软删除，只允许删除本人图片。

## 11. 常见问题

### 11.1 结束会话后没有马上看到日记

这是正常现象。`complete` 已转为后台生成。继续轮询：

```text
GET /api/v1/daily-records/today
```

看到 `is_generating=true` 表示后台还在跑；看到 `has_record=true` 才表示完成。

### 11.2 `40903` 会话已处于结束状态

常见于重复调用 `complete`，或已完成会话继续发普通消息。

补写时请在消息请求体里传：

```json
{
  "is_supplement": true
}
```

### 11.3 `40002` 或 `50003` AI 配置问题

常见原因：

- 未配置 `TOKENHUB_AUTHORIZATION`
- 未配置 `TOKENHUB_API_KEY`
- TokenHub 网络或鉴权失败

处理：

- 检查 `.env`
- 检查启动日志
- 看返回的业务错误码和 `message`

### 11.4 日期不符合预期

优先检查：

- 用户设置里的 `timezone`
- 当前测试账号是否新注册但未改时区
- 历史接口日期格式是否属于支持范围

### 11.5 图片上传本地可见，线上不可见
