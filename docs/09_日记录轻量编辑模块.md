# 《09_日记录轻量编辑模块》后端设计与开发规范

> 本文档为 Echo MVP 阶段“日记录轻量编辑模块”提供实施脚本，聚焦于只允许修改备注、关键词和标签的最小可用编辑能力，避免引入复杂数据结构修改和历史重写。

---

## 1. 模块目标

本模块提供单条日记录的轻量编辑能力：支持用户对现有日记录的 `user_note`、`keywords`、`tags_to_add` 以及 `tags_to_remove` 进行安全、可控的调整。只做“轻量编辑”，是为了保持接口简单、降低业务风险，不触碰 summary_text、emotion_overall_score、events/emotions/expenses/locations 等复杂结构化资产。

## 2. 文件清单

- `app/schemas/daily_record.py`
  - 负责定义日记录相关请求模型，以及复用已有详情响应模型。
- `app/services/daily_record_service.py`
  - 负责单条日记录的业务逻辑，包括今日记录、单日详情与轻量更新。
- `app/api/routes/daily_records.py`
  - 负责路由层接收请求、调用 Service、统一封装 `ApiResponse`。

## 3. Schema 设计脚本

- 需要定义请求模型：`UpdateDailyRecordRequest`
- 添加字段：
  - `user_note`
  - `keywords`
  - `tags_to_add`
  - `tags_to_remove`
- 返回继续复用现有 `DailyRecordDetailResponse`
- 每个字段语义和允许值：
  - `user_note`
    - 可为空
    - 显式传 `null` 表示清空备注
    - 不传表示保持现有值不变
  - `keywords`
    - 字符串数组
    - 整体覆盖当前关键词
    - 传空数组表示清空关键词
    - 不传表示保持现有关键词
  - `tags_to_add`
    - 字符串数组
    - 新增标签名列表
    - 新增前应 `strip()` 去除首尾空白
    - 空字符串标签直接忽略
    - 请求内部应去重，避免重复脏数据
  - `tags_to_remove`
    - 标签 ID 数组
    - 对已有标签做软删除
    - 只能删除当前记录下的当前用户标签
    - 不传表示不修改标签删除状态

## 4. Service 设计脚本

- 需要的方法：`update_record(session, user_id, record_id, req)`
- 入参：
  - `AsyncSession`
  - 当前用户 ID
  - `record_id`
  - `UpdateDailyRecordRequest`
- 出参：
  - `DailyRecordDetailResponse`
- 更新步骤：
  - 查询目标 `DailyRecord`，并确认：
    - 记录存在
    - 记录未被软删除
    - `record.user_id == current_user_id`
  - `user_note`
    - 若请求体包含该字段，则写入该值
    - 显式传 `null` 时清空备注
  - `keywords`
    - 若请求体包含该字段，则整体替换当前关键词
    - 传空数组即清空关键词
    - 不追加，严格“整体覆盖”语义
  - `tags_to_add`
    - 对请求数组中的每个标签名先调用 `strip()`
    - 忽略空字符串标签
    - 去重后新增标签
    - 避免与当前记录中已有有效标签同名重复
  - `tags_to_remove`
    - 只针对当前记录、当前用户、且未软删的标签
    - 通过写入 `deleted_at` 实现软删除
- 详情查询规则：
  - 详情返回的 `tags` 必须只包含 `deleted_at is NULL` 的有效标签
  - 已软删标签不应再次出现在接口返回中
- 权限检查：
  - 必须保证只能改自己的记录
  - 若记录归属不符，应拒绝请求
- 标签冲突处理：
  - `tags_to_add` 需去重、避免重复插入
  - 若数据库唯一约束触发，应捕获并返回业务错误，而不是崩溃

## 5. Router 设计脚本

- 路径：`PUT /api/v1/daily-records/{record_id}`
- 建议函数名：`update_record`
- Router 仅负责：
  - 解析 `record_id`
  - 解析 `UpdateDailyRecordRequest`
  - 注入 `SessionDep` 与 `CurrentUser`
  - 调用 `DailyRecordService.update_record(...)`
  - 封装 `ApiResponse(data=...)`
- 为什么 Router 不写更新逻辑：
  - 业务逻辑必须集中在 Service 层，避免路由层膨胀和重复逻辑
  - Router 只做请求到服务的转发与统一响应包装
