# 《04_用户设置模块》后端设计文档

> 本文档为「回响（Echo）」MVP 阶段的用户设置模块设计文档。  
> 关联文档：`docs/01_后端全局约束与命名规范.md`、`docs/02_数据库总览与关系设计初稿.md`、`docs/03_认证与用户模块.md`  
> 本文档不包含代码、ORM、SQL，仅输出接口设计与模块规范。

---

## 1. 模块目标

本模块负责管理用户的个人化配置项，是聊天行为、推送提醒与 AI 人格注入的配置中枢。

具体职责包括：
- 提供当前用户设置的读取能力
- 提供当前用户设置的更新能力（覆盖式局部更新）
- 管理每日提醒开关与提醒时间的配置
- 管理用户输入偏好（文字 / 语音 / 混合）
- 提供用户主动发起的"删除本人全部数据"入口

本模块不包含用户基础信息（如昵称、头像）的管理，该部分归属于认证与用户模块（`users` 表）。

## 3. `user_settings` 表结构建议（设计说明）

以下为字段级别的设计说明，不含 SQL 或 ORM 代码。

| 字段名 | 类型建议 | 允许为空 | 说明 |
|--------|----------|----------|------|
| `id` | UUID | 否 | 主键，使用 UUID v4 |
| `user_id` | UUID | 否 | 外键，关联 `users.id`；须建唯一索引以确保 1:1 关系在数据库层面得到强制保障 |
| `timezone` | VARCHAR(50) | 否 | 用户本地时区，遵循 IANA 时区标识格式（如 `Asia/Shanghai`）；影响自然日划分与每日会话归属；初始化时建议默认设为 `Asia/Shanghai` |
| `reminder_enabled` | BOOLEAN | 否 | 是否开启每日提醒推送。此处沿用既定字段名 `reminder_enabled`，不再改名。 |
| `reminder_time` | TIME | 是 | 每日提醒时间，格式为 `HH:MM`（本地时间，不含时区）；仅当 `reminder_enabled` 为 `true` 时生效；`reminder_enabled` 为 `true` 时此字段不得为 `NULL`。 |
| `input_preference` | VARCHAR(20) | 否 | 用户输入偏好，枚举值：`TEXT`（纯文字）、`VOICE`（纯语音）、`MIXED`（混合）；落库及接口返回值均使用**小写**字符串（`text` / `voice` / `mixed`），代码层枚举成员名使用大写（如 `InputPreference.TEXT`） |
| `ai_companion_name` | VARCHAR(50) | 是 | **可选扩展字段（P1）**。用户为 AI 伴侣设定的名称；注入至 LLM System Prompt 中；为空时系统使用默认名称（如「回响」）。MVP 第一版不作为聊天模块的必要依赖。 |
| `created_at` | TIMESTAMPTZ | 否 | 记录创建时间，系统在用户注册时自动写入 |
| `updated_at` | TIMESTAMPTZ | 否 | 最近一次更新时间，由系统自动维护 |
| `deleted_at` | TIMESTAMPTZ | 是 | 软删除标记，默认为 NULL；执行删除用户数据时写入当前时间戳 |

**默认值建议：**

| 字段名 | 建议默认值 | 说明 |
|--------|----------|------|
| `timezone` | `Asia/Shanghai` | MVP 阶段面向中文用户 |
| `reminder_enabled` | `false` | 新注册用户默认不开启提醒 |
| `reminder_time` | `NULL` | 提醒关闭状态下无需配置时间；`reminder_enabled` 为 `false` 时允许为 `NULL` |
| `input_preference` | `text` | 默认纯文字输入，语音功能按需启用 |
| `ai_companion_name` | `NULL` | 可选扩展字段，默认使用系统名称 |

## 4. 与 `users` 表的关系

`user_settings` 与 `users` 之间为 **1:1 关系**，通过 `user_settings.user_id` 外键向 `users.id` 建立关联。

| 约束 | 建议 |
|------|------|
| 外键约束 | `fk_user_settings_user_id_users`，引用 `users.id` |
| 唯一约束 | `uq_user_settings_user_id`，保证一个用户最多只有一条配置记录 |
| 初始化时机 | 用户注册成功后，在同一事务内同步写入一条默认 `user_settings`；若写入失败，整笔注册事务回滚 |
| 数据清除一致性 | 执行删除用户数据时，`user_settings` 记录**不执行软删除**，而是在同一事务内将所有可配置字段重置为默认值，保持记录行存续；其余业务数据表的软删除与该重置操作须在同一事务内完成 |

> **说明**：`user_settings` 的分离建表理由已在 `docs/02_数据库总览与关系设计初稿.md` 第 3.2 节阐明，此处不重复论述。

## 5. 接口清单

| 序号 | 方法 | 路径 | 是否需要鉴权 | 说明 |
|------|------|------|--------------|------|
| 01 | GET | `/api/v1/settings` | 是 | 获取当前用户的全部设置项 |
| 02 | PUT | `/api/v1/settings` | 是 | 覆盖更新当前用户的设置项（局部字段更新） |
| 03 | DELETE | `/api/v1/users/me/data` | 是 | 删除当前用户的全部业务数据（软删除） |

> **路径设计说明**：`GET /api/v1/settings` 与 `PUT /api/v1/settings` 均以当前 Token 所归属的用户为操作对象，不接受通过路径参数传入 `user_id`，与 `GET /api/v1/users/me` 的设计原则一致。  
> `DELETE /api/v1/users/me/data` 置于 `/users/me` 路径下，明确表示该操作是当前用户针对自身发起的，语义上与其他用户管理接口对齐。

## 6. 接口详情

---

### 6.1 获取当前用户设置

**路径**：`GET /api/v1/settings`  
**鉴权**：需要（`Authorization: Bearer <token>`）

#### 请求参数

无请求体。用户身份从 Token 解析获取，不接受通过 URL 或 Body 传入 `user_id`。

#### 返回参数

| 字段 | 类型 | 说明 |
|------|------|------|
| `user_id` | string (UUID) | 所属用户 ID |
| `timezone` | string | IANA 时区标识，如 `Asia/Shanghai` |
| `reminder_enabled` | boolean | 是否开启每日提醒 |
| `reminder_time` | string \| null | 提醒时间，格式 `HH:MM`（24 小时制本地时间）；`reminder_enabled` 为 `false` 时可为 `null` |
| `input_preference` | string | 输入偏好，返回值为小写：`text` / `voice` / `mixed` |
| `ai_companion_name` | string \| null | 可选扩展字段（P1）；AI 伴侣名称；未配置时为 `null` |
| `updated_at` | string (ISO 8601) | 最近一次更新时间 |

#### 返回示例（成功）

```json
{
  "code": 20000,
  "message": "success",
  "data": {
    "user_id": "e7a1f3c2-84b0-4d1e-9a23-1c5d8e2f7b90",
    "timezone": "Asia/Shanghai",
    "reminder_enabled": true,
    "reminder_time": "21:00",
    "input_preference": "text",
    "ai_companion_name": null,
    "updated_at": "2024-06-01T13:00:00Z"
  },
  "trace_id": "req-settings-001"
}
```

#### 错误情况

| HTTP 状态码 | 内部 code | 说明 |
|-------------|-----------|------|
| 401 | 40101 | Token 未携带 |
| 401 | 40102 | Token 已过期或无效 |
| 404 | 40401 | 当前用户的设置记录不存在（理论上不应出现，应在注册阶段同步初始化） |

### 6.2 更新当前用户设置

**路径**：`PUT /api/v1/settings`  
**鉴权**：需要（`Authorization: Bearer <token>`）

> 本接口使用 `PUT` 语义，支持对所有可配置字段的局部覆盖式更新。调用时若某字段不传，则该字段保持不变；若显式传入 `null`，则视为清空该字段（适用于 `reminder_time`、`ai_companion_name`）。  
> `reminder_enabled` 与 `input_preference` 不允许显式传入 `null`，必须提供合法值。  
> **提醒规则（已定稿）**：若请求中 `reminder_enabled` 为 `true`，则 `reminder_time` 必须同时提供合法的 `HH:MM` 值；不满足此条件时接口返回 `400`。

#### 请求参数（Body JSON，全部可选）

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `timezone` | string | 否 | IANA 时区标识，如 `Asia/Shanghai`；应用层须校验是否为合法 IANA 时区字符串 |
| `reminder_enabled` | boolean | 否 | 提醒开关；不允许为 `null` |
| `reminder_time` | string \| null | 否 | 提醒时间，格式 `HH:MM`（24 小时制）；`reminder_enabled` 为 `true` 时必填且不得为 `null`；`reminder_enabled` 为 `false` 时可传 `null` 以清空 |
| `input_preference` | string | 否 | 合法值：`text` / `voice` / `mixed`；不允许为 `null` |
| `ai_companion_name` | string \| null | 否 | 可选扩展字段（P1）；AI 伴侣名称，长度 1~20 字符；显式传入 `null` 可清空（系统恢复默认名称） |

#### 请求示例（开启提醒）

```json
{
  "reminder_enabled": true,
  "reminder_time": "21:30"
}
```

#### 请求示例（仅更新输入偏好）

```json
{
  "input_preference": "mixed"
}
```

#### 返回参数

更新成功后，返回当前用户设置的完整最新状态（与 `GET /api/v1/settings` 返回结构一致）。

| 字段 | 类型 | 说明 |
|------|------|------|
| `user_id` | string (UUID) | 所属用户 ID |
| `timezone` | string | IANA 时区标识 |
| `reminder_enabled` | boolean | 是否开启每日提醒 |
| `reminder_time` | string \| null | 提醒时间；`reminder_enabled` 为 `false` 时可为 `null` |
| `input_preference` | string | 输入偏好，返回小写值 |
| `ai_companion_name` | string \| null | 可选扩展字段（P1）；AI 伴侣名称；未配置时为 `null` |
| `updated_at` | string (ISO 8601) | 本次更新后的时间戳 |

#### 返回示例（成功）

```json
{
  "code": 20000,
  "message": "success",
  "data": {
    "user_id": "e7a1f3c2-84b0-4d1e-9a23-1c5d8e2f7b90",
    "timezone": "Asia/Shanghai",
    "reminder_enabled": true,
    "reminder_time": "21:30",
    "input_preference": "mixed",
    "ai_companion_name": null,
    "updated_at": "2024-06-02T10:00:00Z"
  },
  "trace_id": "req-settings-update-001"
}
```

#### 错误情况

| HTTP 状态码 | 内部 code | 说明 |
|-------------|-----------|------|
| 400 | 40001 | 字段格式不合规（如 `timezone` 非法、`input_preference` 枚举值越界、`reminder_time` 格式错误） |
| 400 | 40002 | `reminder_enabled` 为 `true` 但 `reminder_time` 为 `null` 或未传入 |
| 401 | 40102 | Token 无效或过期 |

### 6.3 删除当前用户全部数据

**路径**：`DELETE /api/v1/users/me/data`  
**鉴权**：需要（`Authorization: Bearer <token>`）

> 本接口为高危操作，一旦执行将对当前用户的所有业务数据执行软删除，并将 `user_settings` 重置为默认值。  
> 本接口**不注销账户**（`users` 表用户行保持不变，`user_settings` 记录行保持不变），用户执行后仍可继续登录并重新开始记录。  
> 若需账户注销功能，应由独立的账户注销接口处理（超出本模块范围，不在此设计）。  
> **二次确认为前端交互责任**：接口本身不通过请求体接收确认字段，前端须在 UI 层完成用户的二次确认后方可调用本接口。

#### 请求参数

无请求体。用户身份从 Token 解析获取。

#### 返回参数

| 字段 | 类型 | 说明 |
|------|------|------|
| `cleared_at` | string (ISO 8601) | 本次数据清除操作的执行时间戳 |

#### 返回示例（成功）

```json
{
  "code": 20000,
  "message": "success",
  "data": {
    "cleared_at": "2024-06-10T08:30:00Z"
  },
  "trace_id": "req-delete-data-001"
}
```

#### 错误情况

| HTTP 状态码 | 内部 code | 说明 |
|-------------|-----------|------|
| 401 | 40102 | Token 无效或过期 |
| 500 | 50001 | 数据清除事务执行失败（任一操作失败时整体回滚） |

## 7. 字段设计建议

### 7.1 `reminder_enabled` 命名说明

此处沿用既定字段名 `reminder_enabled`，不再改名。  
**说明**：《01_后端全局约束与命名规范》第 10 节建议布尔值字段以 `is_` 或 `has_` 开头；本项目在早期设计中已使用 `reminder_enabled` 作为约定字段名，为保持命名一致性，本模块不执行重命名，后续其他模块引用此字段时应与此保持一致。

---

### 7.2 `reminder_time` 字段设计建议

**建议**：使用数据库 `TIME` 类型存储，格式为 `HH:MM`（24 小时制），不携带日期信息，不携带时区。  
**原因**：提醒时间本质上是一个"本地墙钟时刻"，所对应的实际 UTC 时间需结合 `user_settings.timezone` 在推送任务触发时动态计算。若将其存储为带时区的 `TIMESTAMPTZ`，则每次用户更改时区后都需要联动更新提醒时间字段，维护成本较高。

**关联逻辑**：推送系统在生成每日提醒任务时，从 `user_settings` 同时读取 `reminder_time` 与 `timezone`，在计算层将其转换为实际的 UTC 触发时刻，不在 `user_settings` 中存储预计算结果。

**约束规则（已定稿）**：`reminder_time` 仅在 `reminder_enabled` 为 `false` 时允许为 `NULL`。当 `reminder_enabled` 为 `true` 时，`reminder_time` 必须为合法的 `HH:MM` 值；接口层在业务校验阶段强制拦截不满足此条件的请求，返回 `400`。

---

### 7.3 `input_preference` 字段设计建议

**建议**：枚举值落库及接口返回均使用小写字符串：`text` / `voice` / `mixed`；代码层枚举成员名使用全大写（如 `InputPreference.TEXT`）。  
**原因**：与《01_后端全局约束与命名规范》第 11 节枚举值规范保持一致；同时与 `users.status` 字段的大小写处理策略对齐，降低接口消费方的理解成本。

**设计权衡**：

> **建议**：不使用数据库原生 `ENUM` 类型，以定长 `VARCHAR(20)` 存储。  
> **原因**：若后续新增输入偏好类型（如 `image`），`VARCHAR` 方案只需在应用层调整枚举定义，无需执行 `ALTER TYPE`；数据库 `ENUM` 类型的新增选项需要对整个类型执行 DDL 修改，在生产环境中存在锁表风险。

---

### 7.4 `timezone` 字段设计建议

**建议**：使用 IANA 时区字符串（如 `Asia/Shanghai`、`America/New_York`），而非 UTC 偏移量（如 `+08:00`）。  
**原因**：UTC 偏移量在夏令时地区会随季节变化，导致同一地区的用户在夏令时开始/结束时需要主动更新配置。IANA 时区标识已内置夏令时规则，可在应用层（依赖 `pytz` 或 `zoneinfo`）自动处理偏移量计算。

**应用层校验**：`timezone` 字段写入前，应在应用层校验字符串是否为合法 IANA 时区标识，不合法时返回 `400`。

## 8. 删除用户数据接口的处理建议

### 8.1 操作范围

本接口执行的是**业务数据清除**，不注销用户账户。执行完成后：
- `users` 表中的用户行保持不变（`deleted_at` 仍为 `NULL`，用户仍可登录）
- `user_settings` 记录行保持不变，所有可配置字段重置为注册时的默认值（`timezone` 恢复 `Asia/Shanghai`、`reminder_enabled` 恢复 `false`、`reminder_time` 恢复 `NULL`、`input_preference` 恢复 `text`、`ai_companion_name` 恢复 `NULL`）
- 第 2 节所列全部业务数据表（`conversations`、`conversation_messages`、`media_files`、`daily_records`、`record_*` 系列、`weekly_reports`）中与该 `user_id` 关联的所有行，`deleted_at` 字段统一写入当前 UTC 时间戳

### 8.2 事务边界

`user_settings` 字段重置与全部关联业务表的软删除操作须在**单一数据库事务**内完成。任意操作失败（如锁等待超时、约束冲突）应触发整体回滚，向调用方返回 `500`，不允许出现部分清除的中间状态。

### 8.3 二次确认机制

二次确认为**前端交互责任**，不体现在接口请求参数中。前端须在 UI 层完成用户的明确二次确认（如弹窗提示）后，方可向后端发起本接口的调用。后端接口不接收任何确认字段，不依赖请求体做确认校验。

### 8.4 OSS 文件的处理

`media_files` 表的软删除操作仅标记元数据记录，不触发对象存储（OSS）上实际文件的物理删除。物理文件的清理应由独立的后台清理任务（如定期扫描 `deleted_at` 非空的 `media_files` 记录并调用 OSS 删除接口）处理，不在本次接口的同步链路内执行，以避免拖慢接口响应时间。

### 8.5 日志审计

本接口属于高危操作，建议在服务器日志中记录以下信息（不含敏感数据）：操作时间、`user_id`、请求 `trace_id`、各表软删除行数。

## 9. MVP 优先级

| 优先级 | 接口 | 原因 |
|--------|------|------|
| P0（必须先做） | `GET /api/v1/settings` | 聊天模块在初始化会话时需读取 `timezone` 以确定归属日期；`ai_companion_name` 为可选扩展字段，不作为第一版必要依赖 |
| P1（第一版应完成） | `PUT /api/v1/settings` | 用户需能配置时区、提醒时间与输入偏好，是基础体验保障 |
| P2（可推迟） | `DELETE /api/v1/users/me/data` | 满足合规需求与用户数据自主权；MVP 初期用户基数小，实施窗口相对宽松，但应在正式上线前完成 |

## 11. 风险点与实现建议

### 风险 1：`user_settings` 记录缺失

- **风险描述**：若注册流程中 `user_settings` 初始化失败且事务未正确回滚，可能导致 `GET /api/v1/settings` 返回 404，影响聊天模块正常启动。
- **建议**：在注册事务中强制同步初始化 `user_settings`，确保两个写入操作处于同一数据库事务。同时在 `GET /api/v1/settings` 接口中加入兜底逻辑：若查询结果为空，自动为当前用户创建默认配置记录并返回，不直接返回 404。

---

### 风险 2：`timezone` 非法值写入

- **风险描述**：若应用层未对 `timezone` 字段的合法性进行校验，写入非法 IANA 时区字符串（如 `UTC+8`、`China Standard Time`）后，时区相关计算（归属日期判定、提醒触发时刻计算）将产生预期外的错误。
- **建议**：在 `PUT /api/v1/settings` 接口的入参校验阶段（Pydantic 验证器层），调用标准时区库（如 `zoneinfo` 或 `pytz`）对传入的时区字符串进行合法性验证，不合法则返回 `400`，不进入数据库写入流程。

---

### 风险 3：`reminder_enabled` 为 `true` 但 `reminder_time` 为 `NULL`

- **风险描述**：若接口入参校验不完整，可能出现 `reminder_enabled=true` 与 `reminder_time=NULL` 同时落库的情况，推送系统读取时触发计算错误。
- **建议**：接口层在业务校验阶段强制拦截此组合（见 6.2 节错误码 `40002`），不允许该状态落库。推送任务调度层同时加入防御性判断作为双重保障：若遇此状态则跳过并记录日志，不抛出异常。

---

### 风险 4：数据清除事务过大导致超时

- **风险描述**：若某用户的历史数据量极大（如数百条 `conversation_messages`），单一事务中覆盖所有关联表的软删除操作可能触发事务超时。
- **建议**：MVP 阶段用户数据量有限，单事务方案可接受。若后期出现超时问题，可将清除操作拆分为：首先在事务内完成 `user_settings` 重置（阻断新数据写入），其余关联表的软删除通过异步后台任务分批处理。但此方案引入了中间状态，需要配套补偿机制，MVP 阶段不建议提前引入。

---

### 风险 5：输入偏好与实际可用功能不一致

- **风险描述**：若用户将 `input_preference` 设置为 `voice`，但语音输入功能在当前版本尚未上线，可能导致前端出现功能不可用但配置已保存的状态矛盾。
- **建议**：MVP 初期若语音功能未上线，建议在前端层面限制 `voice` 和 `mixed` 选项的可选性；后端接口层不对此做额外拦截，正常存储即可，不引入版本标记字段。该字段已预留，待功能上线后前端同步开放。
