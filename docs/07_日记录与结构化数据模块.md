> 本文档为「回响（Echo）」MVP 阶段的日记录与结构化数据模块设计文档。  
> 关联文档：`docs/01_后端全局约束与命名规范.md`、`docs/02_数据库总览与关系设计初稿.md`、`docs/03_认证与用户模块.md`、`docs/04_用户设置模块.md`、`docs/05_聊天与会话模块.md`

## 1. 模块目标

本模块负责每日聊天结束后，承接基于对话生成的总结内容及其结构化衍生数据的存储、查询与用户轻量修正。

具体职责包括：
- 接收总结生成任务的处理结果，将 AI 生成的今日总结及提取的结构化数据一次性统一入库
- 提供今日总结及历史某日记录的查询能力
- 允许用户对 AI 生成结果进行轻量的修正（灵感、备注、关键词）

本模块不包含：
- 跨日聚合列表与历史检索（归属于历史模块）
- 周报生成与查询（归属于周报模块）
- 聊天对话与会话状态管理（归属于聊天与会话模块）

## 3. `daily_records` 表结构建议（设计说明）

**设计原则**：MVP 阶段不在本表引入独立的状态体系，总结生成过程的状态以 `conversations.status` 为准。`daily_records` 仅在总结生成成功后执行**一次性写入**，因此不包含状态字段，且核心内容字段不需要允许为空。

| 字段名 | 类型建议 | 允许为空 | 说明 |
|--------|----------|----------|------|
| `id` | UUID | 否 | 主键，使用 UUID v4 |
| `user_id` | UUID | 否 | 外键，关联 `users.id`；须建普通索引 |
| `conversation_id` | UUID | 否 | 外键，关联 `conversations.id`；须建唯一索引，强制与会话的 1:1 关系 |
| `record_date` | DATE | 否 | 归属的用户本地自然日（`YYYY-MM-DD`）；从 `conversations.record_date` 冗余写入；须建普通索引 |
| `summary_text` | TEXT | 否 | AI 生成的今日总结正文 |
| `emotion_overall_score` | INTEGER | 否 | 展示型综合情绪分（1~10），详见第 12 节 |
| `keywords` | JSONB | 否 | AI 提取的当日关键词数组（字符串列表），详见第 12 节 |
| `user_note` | TEXT | 是 | 用户自行填写的备注或感悟，初始为 NULL |
| `extra_json` | JSONB | 是 | 扩展字段，容纳 AI 返回中超出主结构的附属信息 |
| `created_at` | TIMESTAMPTZ | 否 | 记录创建时间 |
| `updated_at` | TIMESTAMPTZ | 否 | 最近一次更新时间 |
| `deleted_at` | TIMESTAMPTZ | 是 | 软删除标记，默认为 NULL |

**约束建议：**

| 约束类型 | 命名 | 说明 |
|----------|------|------|
| 主键 | `pk_daily_records` | `id` 字段 |
| 外键 | `fk_daily_records_user_id_users` | 引用 `users.id` |
| 外键 | `fk_daily_records_conversation_id_conversations` | 引用 `conversations.id` |
| 唯一索引 | `uq_daily_records_conversation_id` | 保证一个会话最多对应一条日记录 |
| 联合唯一 | `uq_daily_records_user_id_record_date` | 同一用户同一自然日最多一条记录；部分索引（`WHERE deleted_at IS NULL`） |
| 普通索引 | `ix_daily_records_user_id` | 按用户查询历史记录 |
| 普通索引 | `ix_daily_records_record_date` | 按日期查询 |

## 9. `daily_records` 与 `conversations` 的关联及状态流转建议

`daily_records` 与 `conversations` 之间为**强 1:1 关系**，通过 `daily_records.conversation_id` 外键关联。

### 状态流转机制

本模块不再引入独立的状态流转，完全以 `conversations` 表的当前状态作为单点事实来源：

1. **触发与锁定**：当用户调用聊天模块的“结束今日记录”(`complete`)接口时，聊天模块同步将 `conversations.status` 更新为 `completing`。此状态代表“总结生成任务进行中”。
2. **异步执行与落库**：总结生成任务成功后，在同一数据库事务中：
   - 将 AI 生成的结果一次性写入 `daily_records` 记录。
   - 插入各相关子表的记录。
   - 将 `conversations.status` 推进为 `completed`。
3. **异常处理**：若任务失败，`conversations.status` 暂时停留在 `completing`；此时不会有 `daily_records` 行被写入，交由重试策略或系统告警处理。

这种设计确保了状态转移的原子性，避免了结构化日志数据处于“部分写入”或“生成失败”等复杂的中间状态体系。

## 10. 接口清单

| 序号 | 方法 | 路径 | 是否需要鉴权 | 说明 |
|------|------|------|--------------|------|
| 01 | GET | `/api/v1/daily-records/today` | 是 | 获取今日总结详情；若仍在生成中则明确返回等待状态 |
| 02 | GET | `/api/v1/daily-records/{record_date}` | 是 | 获取指定日期的日记录详情；`record_date` 格式为 `YYYY-MM-DD` |
| 03 | PUT | `/api/v1/daily-records/{record_id}` | 是 | 更新 `user_note`、`keywords` 及灵感增删（MVP 严格限制在此范围） |

> **路径设计说明**：
> - `/daily-records/today` 须在路由注册时先于 `/{record_date}` 注册。
> - GET 两个查询接口通过路径参数区分：`today` 隐含当前用户的系统本地日期转换，`{record_date}` 使用显式日期字符串请求。

## 11. 接口详情

---

### 11.1 获取今日总结

**路径**：`GET /api/v1/daily-records/today`  
**鉴权**：需要（`Authorization: Bearer <token>`）

#### 业务逻辑说明
1. 通过 `user_settings.timezone` 将当前服务器时间换算为用户的本地日期。
2. 查询该日期的 `conversations` 表：
   - 若不存在记录，或 `status` 为 `recording`：返回相应的表示“今日暂无可查看的总结”的空或引导状态（`has_record: false`, `is_generating: false`）。
   - 若 `status` 为 `completing`：表示仍在生成过程，返回 `has_record: false`, `is_generating: true`。
   - 若 `status` 为 `completed`：查询并组装 `daily_records` 及关联子表的所有数据并返回。

#### 请求参数
无请求体。

#### 返回参数

| 字段 | 类型 | 说明 |
|------|------|------|
| `has_record` | boolean | 今日总结是否已生成可查阅（`conversations.status` 为 `completed` 时为 true） |
| `is_generating` | boolean | 总结是否正在生成中（`conversations.status` 为 `completing` 时为 true） |
| `record` | object \| null | 日记录详情对象；仅当 `has_record` 为 true 时存在，否则为 `null` |
| `record.id` | string (UUID) | 日记录 ID |
| `record.record_date` | string (YYYY-MM-DD) | 归属日期 |
| `record.summary_text` | string | AI 总结正文 |
| `record.emotion_overall_score` | integer | 展示型综合情绪分（1~10） |
| `record.keywords` | array | 关键词字符串数组 |
| `record.user_note` | string \| null | 用户感悟或备注；未填写时为 `null` |
| `record.events` | array | 事件条目列表（每条含 `id`、`content`、`source`、`is_user_confirmed`、`created_at`） |
| `record.emotions` | array | 情绪条目列表（每条含 `id`、`emotion_label`、`intensity`、`source`、`is_user_confirmed`、`created_at`） |
| `record.expenses` | array | 消费条目列表（每条含 `id`、`amount`、`currency`、`category`、`description`、`source`、`is_user_confirmed`、`created_at`） |
| `record.locations` | array | 地点条目列表（每条含 `id`、`name`、`source`、`is_user_confirmed`、`created_at`） |
| `record.inspirations` | array | 灵感条目列表（每条含 `id`、`content`、`source`、`created_at`） |
| `record.created_at` | string (ISO 8601) | 记录创建时间 |
| `record.updated_at` | string (ISO 8601) | 最近更新时间 |

| 500 | 50002 | 用户时区配置缺失或非法，无法确定今日日期归属；已记录高优先级告警 |

### 11.2 获取指定日期的日记录详情

**路径**：`GET /api/v1/daily-records/{record_date}`  
**鉴权**：需要（`Authorization: Bearer <token>`）

#### 路径参数

| 参数名 | 类型 | 说明 |
|--------|------|------|
| `record_date` | string | 目标日期，格式 `YYYY-MM-DD` |

#### 返回参数

结构与 `GET /api/v1/daily-records/today` 中 `data` 的结构一致。通常在此接口作为回溯场景下，只返回已有记录状态。无记录时可以表现为 `has_record: false`。

#### 错误情况

| HTTP 状态码 | 内部 code | 说明 |
|-------------|-----------|------|
| 400 | 40001 | `record_date` 格式非法（非 `YYYY-MM-DD`） |
| 401 | 40102 | Token 无效或过期 |
| 403 | 40301 | 越权查询（记录并不属于当前用户） |
| 404 | 40401 | 该日期无可用记录 |

### 11.3 更新日记录（范围严控）

**路径**：`PUT /api/v1/daily-records/{record_id}`  
**鉴权**：需要（`Authorization: Bearer <token>`）

> **MVP 修改范围说明及限制**：
> 本接口仅允许以下更新：
> 1. `user_note` 的文本更替或清空
> 2. `keywords` 整体覆盖或清空
> 3.对 `record_inspirations` 维度的条目增（`inspirations_to_add`）删（`inspirations_to_remove`）
> 
> **限制边界**：
> 暂不支持对结构化子条目（`events` / `emotions` / `expenses` / `locations`）的具体内容及数值（如修改消费金额或情感标记）进行编辑交互。该部分能力在 MVP 后续若有需求，会以独立的条目级专门接口提供，不由本 PUT 提供支持。

#### 路径参数

| 参数名 | 类型 | 说明 |
|--------|------|------|
| `record_id` | string (UUID) | 目标日记录 ID |

#### 请求参数（Body JSON，全部可选）

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `user_note` | string \| null | 否 | 用户备注；显式传入 `null` 可清空；未传则保持不变 |
| `keywords` | array \| null | 否 | 关键词字符串数组；整体覆盖现有值；显式传 `null` 清空；未传不变 |
| `inspirations_to_add` | array | 否 | 新增的灵感内容列表（字符串数组）；按 `source=user` 写入 `record_inspirations` |
| `inspirations_to_remove` | array | 否 | 需移除的灵感 ID 列表（UUID 数组）；对相关条目进行软删除验证 |

#### 请求示例（同时更新备注并增修灵感和关键词）

```json
{
  "user_note": "这是很有意义的一天，值得好好记录。",
  "keywords": ["重逢", "老友", "感恩"],
  "inspirations_to_add": ["珍贵", "友谊"],
  "inspirations_to_remove": ["tg-uuid-0001"]
}
```

#### 返回参数

更新成功后，返回当前日记录的完整且最新的所有对象（与 11.1 中的 `record` 结构一致），以便前端重置状态。

#### 错误情况

| HTTP 状态码 | 内部 code | 说明 |
|-------------|-----------|------|
| 400 | 40001 | 请求参数格式非法（如包含未规定的修改结构） |
| 401 | 40102 | Token 无效或过期 |
| 403 | 40301 | `record_id` 或相关的需移除 tags_id 越权 |
| 404 | 40401 | `record_id` 不存在 |

## 12. 结构化数据字段设计建议

### 12.1 `keywords` 适用 JSONB 的考量

**结论：MVP 阶段适合存储为 JSONB。**

`keywords` 字段承载当日内容的概要性字符串标注（如 `["重逢", "散步", "满足"]`）。

**理由：**
- **作为一个整体展示结构使用**，没有基于单一关键词的高频独立检索、过滤需求。
- 用户每次操作均是整体替换，无需精确定位并编辑其中某一项元组。
- 符合项目《后端全局约束与命名规范》中，高内聚且无需深度关系过滤场景下允许使用 JSONB 降低独立建表冗余的要求。

*补充：当后续确实出现“关键词作为用户灵感筛选的全局搜索依据”时，才需将其升级拆解为关联词典表。*

---

### 12.2 `emotion_overall_score` 的明确定义

**建议：INTEGER 存储，取值范围 1~10。它是一个“展示型综合分”。**

- **纯展示属性**：它是一个直观的综合评分（1=极度负面，5=中性，10=极度正面），方便前端直接映射为情绪温度计或其他展示效果。
- **静态一次性生成**：本分数仅在 AI **生成当日总结过程中**一次性计算并得出写入。
- **无需且不联动重算**：在 MVP 阶段，它作为一个当期的快照写入。之后如果用户在其余流程上针对 `record_emotions` 等内容或者强度有轻微认同度或状态调整，该 `emotion_overall_score` **不进行自动联动重算**。借此将复杂的主子关联重新计分的交互开销削减，极大降低数据修改链路的复杂度。

---

### 12.3 哪些字段适合保留 `is_user_confirmed`

此字段仅保留在AI提取存在不确定且未来需要作为严谨统计数据支持的**业务级子表**内。

| 模块范畴 / 表 | 是否保留 | 原因说明 |
|---------------|----------|----------|
| `record_events` | ✅ 必须保留 | 事件存在理解或概括误差，业务需要体现其为可信资产 |
| `record_emotions` | ✅ 必须保留 | 情绪判定多为主观推论，被确认的数据才能纳入正式的情绪报表 |
| `record_expenses` | ✅ 必须保留 | 金额和货币是强准确敏感型数据，未确认的内容绝不能并入真正的收支统计 |
| `record_locations` | ✅ 必须保留 | 模糊地名的有效地址解析确认需要存在依赖 |
| `record_inspirations` | ❌ 不设置 | 灵感体系以自然存在（或不进行软删除）即可默认其为合理分类器，不增加中间认知障碍 |
| *主表总结明细* | ❌ 不设置 | 对系统叙述不需要布尔化认同，仅通过 `user_note` 补充主观定调 |

---

### 12.4 为何维持专门的业务结构化子表替代单一 JSONB

1. **为跨日计算铺路**：历史与周报模块一定会面临“提取上月超过某金额的 `expenses` 金额相加”、“提取过去七天的所有正向 `emotions` 组合”等查询需求。在 JSONB 内执行计算、聚合查询不但可读性极差且性能损耗严重。
2. **便于后续实现数据点精修与独立的软删除控制**：每个事件或消费是业务上的一个独立 Entity，应当能让用户未来实现“仅删除特定一笔花费”并在日志表中留下标准的操作时间轨迹与 `deleted_at` 追溯，而大型 JSONB 使得更新特定局部数组面临“先读取再序列化存回”的脏写风险。
3. **严格化字段约束**：`amount` 为 `NUMERIC(10, 2)` 需要绝对规避使用类似弱类型进行串改的不良边界，这仅能在严格的字段定义上强制。

## 15. 风险点与实现建议

### 风险 1：事务中断带来的系统性数据断层

- **风险特征**：总结完成后大量表需要连环下盘（主表以及 5 个子表）。若期间连接闪断或插入异常，很容易出现生成了一半的内容存库断层差异。
- **处置必须**：必须依赖完备的数据库本地事务执行原子级的单一写入封装；发生任何外键依赖问题抑或超时报错立马进行整体回滚，并将 `conversations.status` 保持为 `completing` 而不变，在重试排查后重新由调度中心触发即可。

### 风险 2：灵感软删除越权问题

- **风险特征**：由于 PUT `/daily-records/{record_id}` 的提交仅以一个 ID 数组 `inspirations_to_remove` 进行，容易导致传输他人的灵感 ID 造成越界软删除。
- **处置必须**：软删除语句必须绑定类似 `WHERE id IN (受检集合) AND record_id = {传入record_id} AND user_id = {当前验权user_id}`。保证越界的任何处理都不产生有效执行。

### 风险 3：时区容错失效

- **风险特征**：获取不正确的时区可能会导致自然日转换时跨日历错归，甚至将该日数据强加错配给上一日。
- **处置必须**：获取 `today` 查询若未能顺利解析出有效规范的 `timezone` 或者该请求无时区支撑时，应该阻断获取直接抛出 `500`。
