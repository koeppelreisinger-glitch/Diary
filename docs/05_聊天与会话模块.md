# 《05_聊天与会话模块》后端设计文档

> 本文档为「回响（Echo）」MVP 阶段的聊天与会话模块设计文档。  
> 关联文档：`docs/01_后端全局约束与命名规范.md`、`docs/02_数据库总览与关系设计初稿.md`、`docs/03_认证与用户模块.md`、`docs/04_用户设置模块.md`  
> 本文档不包含代码、ORM、SQL，仅输出接口设计与模块规范。

---

## 1. 模块目标

本模块负责用户与 AI 之间每日对话的全生命周期管理，是 MVP 核心闭环的主体执行层。

具体职责包括：
- 管理每日会话的创建与状态流转
- 承载用户消息与 AI 回复的双向消息写入
- 提供会话消息的有序列表查询能力
- 接收用户发出的"结束今日记录"指令，并触发后续总结生成流程

本模块不包含：
- AI 总结内容的生成逻辑（由总结生成任务异步执行，归属于日记录模块的写入侧）
- 历史记录的查看（归属于历史模块）
- 周报生成（归属于周报模块）
- 媒体文件的上传与 ASR 转写（归属于媒体文件模块）

## 2. 涉及的数据表

本模块直接操作以下数据表：

| 表名 | 读/写 | 说明 |
|------|-------|------|
| `conversations` | 读写 | 每日会话主表，本模块的核心操作对象 |
| `conversation_messages` | 读写 | 会话内消息流水记录表 |
| `user_settings` | 只读 | 读取 `timezone` 字段，用于确定用户当地自然日归属 |
| `users` | 只读（鉴权） | 通过 Token 解析 `user_id` |

以下表由本模块触发动作，但不在本模块内部执行写入，写入由异步任务或其他模块负责：

| 表名 | 触发方式 | 说明 |
|------|----------|------|
| `daily_records` | 结束记录接口触发异步任务 | 总结生成完成后由任务写入，本模块不直接操作 |
| `media_files` | 消息发送时引用已上传的媒体文件 | 仅读取 `media_file_id`，不在本模块内写入 |

## 3. `conversations` 表结构建议（设计说明）

以下为字段级别的设计说明，不含 SQL 或 ORM 代码。

| 字段名 | 类型建议 | 允许为空 | 说明 |
|--------|----------|----------|------|
| `id` | UUID | 否 | 主键，使用 UUID v4 |
| `user_id` | UUID | 否 | 外键，关联 `users.id`；须建普通索引以支持按用户查询 |
| `record_date` | DATE | 否 | 会话归属的用户本地自然日（格式 `YYYY-MM-DD`）；由系统在创建时依据 `user_settings.timezone` 换算用户当地日期后写入，不依赖服务器时区 |
| `status` | VARCHAR(20) | 否 | 会话状态，详见第 5 节 |
| `extra_json` | JSONB | 是 | 可选扩展字段，用于承载模型调用参数快照等非结构化附属信息，不得写入需参与过滤或关联查询的字段 |

| `updated_at` | TIMESTAMPTZ | 否 | 最近更新时间，系统自动维护 |
| `deleted_at` | TIMESTAMPTZ | 是 | 软删除标记，默认为 NULL |

| 约束类型 | 命名 | 说明 |

| 普通索引 | `ix_conversations_user_id` | 支持按用户查询历史会话列表 |

|--------|----------|----------|------|

| `created_at` | TIMESTAMPTZ | 否 | 消息写入时间，系统自动写入 |
| `updated_at` | TIMESTAMPTZ | 否 | 最近更新时间，系统自动维护 |
| `deleted_at` | TIMESTAMPTZ | 是 | 软删除标记，默认为 NULL |

**约束建议：**

| 约束类型 | 命名 | 说明 |
|----------|------|------|
| 主键 | `pk_conversation_messages` | `id` 字段 |
| 外键 | `fk_conversation_messages_conversation_id_conversations` | 引用 `conversations.id` |
| 外键（可选） | `fk_conversation_messages_media_file_id_media_files` | 引用 `media_files.id`；允许为 NULL |
| 联合唯一 | `uq_conversation_messages_conversation_id_sequence_number` | 保证同一会话内 `sequence_number` 不重复 |
| 普通索引 | `ix_conversation_messages_conversation_id` | 支持按会话拉取全部消息列表 |

## 5. 会话状态设计建议

`conversations.status` 字段定义会话的生命周期阶段。落库及接口返回值均使用小写字符串，代码层枚举成员名使用大写。

| 状态值（落库/返回） | 枚举成员名 | 含义 |
|---------------------|-----------|------|
| `recording` | `ConversationStatus.RECORDING` | 记录进行中，接受用户输入与 AI 回复 |
| `completing` | `ConversationStatus.COMPLETING` | 用户已触发结束，AI 总结生成任务正在异步执行；此状态下拒绝新消息写入 |
| `completed` | `ConversationStatus.COMPLETED` | AI 总结已生成完毕，`daily_records` 已写入；会话生命周期结束 |

**状态流转路径：**

```
recording  →（用户调用结束接口）→  completing  →（总结任务完成）→  completed
```

**约束说明：**
- 仅 `recording` 状态下允许接受 `POST /messages` 请求；其他状态下返回 `409`。
- `completing` → `completed` 的状态推进由异步任务（总结生成服务）完成，不由本模块的接口直接触发。
- 不存在从 `completed` 回退到 `recording` 的路径；若用户在已完成的当日再次进入，前端应直接读取历史记录，不创建新会话。

**设计权衡：**

> **关于是否增加 `failed` 状态**  
> **建议**：MVP 阶段不引入 `failed` 状态。  
> **原因**：总结生成失败时可通过重试机制在 `completing` 状态下重复触发，无需引入独立失败状态。后续若需要精细化的任务状态管控，可在不破坏当前接口结构的前提下扩展新状态。

## 6. 消息顺序字段设计建议

`conversation_messages.sequence_number` 为会话内消息的有序序号，从 `1` 开始单调递增。

**设计选择：使用 `sequence_number` 而非单纯依赖 `created_at` 排序**

| 对比维度 | `sequence_number` | 仅用 `created_at` |
|----------|-------------------|-------------------|
| 排序稳定性 | 确定，业务层保证单调递增 | 受服务器时钟精度影响，毫秒级并发可能出现相同时间戳 |
| 并发写入 | 需要在业务层保证序号分配的原子性（见风险 4） | 无需额外保证 |
| AI 消息插入 | 可明确表达"AI 回复紧跟用户消息之后"的位置关系 | 依赖时间先后，AI 响应写入时间难以精确紧跟 |
| 消息计数 | 直接从序号推断当前会话消息总数 | 需额外 COUNT 查询 |

**分配规则**：
- 用户消息写入时分配下一个可用序号；AI 回复消息紧跟用户消息序号 +1 写入。
- 同一次"发送消息"请求中，用户消息序号与 AI 回复消息序号应在同一业务操作中连续分配，不允许中间插入其他序号。
- 序号的实际分配方式（如数据库行锁、原子递增等）属于实现细节，不在本文档中展开。

## 7. 接口清单

| 序号 | 方法 | 路径 | 是否需要鉴权 | 说明 |
|------|------|------|--------------|------|
| 01 | GET | `/api/v1/conversations/today` | 是 | 获取今天的会话状态；若今日尚无会话，返回空状态而非 404 |
| 02 | POST | `/api/v1/conversations` | 是 | 创建今天的会话；今日会话已存在时返回 409 |
| 03 | GET | `/api/v1/conversations/{conversation_id}/messages` | 是 | 获取指定会话的消息列表，按 `sequence_number` 升序排列 |
| 04 | POST | `/api/v1/conversations/{conversation_id}/messages` | 是 | 发送用户消息，同步返回 AI 回复 |
| 05 | POST | `/api/v1/conversations/{conversation_id}/complete` | 是 | 结束今日记录，触发 AI 总结生成异步任务 |

> **路径设计说明**：`/conversations/today` 作为具名子资源路径，语义上明确表示"今日会话"这一业务实体，优于通过 query 参数过滤（如 `?date=today`）。注意路由注册顺序须确保 `/today` 在 `/{conversation_id}` 之前匹配，防止字符串 `today` 被误解析为 UUID。

## 8. 接口详情

---

### 8.1 获取今日会话状态

**路径**：`GET /api/v1/conversations/today`  
**鉴权**：需要（`Authorization: Bearer <token>`）

#### 请求参数

无请求体。用户身份从 Token 解析，今日日期由系统读取 `user_settings.timezone` 后换算。

#### 返回参数

| 字段 | 类型 | 说明 |
|------|------|------|
| `has_today` | boolean | 今日是否已存在会话记录 |
| `conversation` | object \| null | 今日会话对象；`has_today` 为 `false` 时返回 `null` |
| `conversation.id` | string (UUID) | 会话 ID |
| `conversation.status` | string | 会话状态：`recording` / `completing` / `completed` |
| `conversation.record_date` | string (YYYY-MM-DD) | 会话归属的用户本地日期 |
| `conversation.message_count` | integer | 当前会话的消息总数（通过 `sequence_number` 最大值推算） |
| `conversation.created_at` | string (ISO 8601) | 会话创建时间 |
| `conversation.updated_at` | string (ISO 8601) | 会话最近更新时间 |

#### 返回示例（今日已有会话）

```json
{
  "code": 20000,
  "message": "success",
  "data": {
    "has_today": true,
    "conversation": {
      "id": "c1a2b3d4-0000-4e5f-9abc-111122223333",
      "status": "recording",
      "record_date": "2024-06-01",
      "message_count": 6,
      "created_at": "2024-06-01T10:00:00Z",
      "updated_at": "2024-06-01T10:45:00Z"
    }
  },
  "trace_id": "req-today-001"
}
```

#### 返回示例（今日尚无会话）

```json
{
  "code": 20000,
  "message": "success",
  "data": {
    "has_today": false,
    "conversation": null
  },
  "trace_id": "req-today-002"
}
```

#### 错误情况

| HTTP 状态码 | 内部 code | 说明 |
|-------------|-----------|------|
| 401 | 40101 | Token 未携带 |
| 401 | 40102 | Token 已过期或无效 |
| 500 | 50002 | 用户时区配置缺失或非法，无法确定今日日期归属；已记录高优先级告警 |

### 8.2 创建今日会话

**路径**：`POST /api/v1/conversations`  
**鉴权**：需要（`Authorization: Bearer <token>`）

> 今日会话已存在（`deleted_at IS NULL` 且 `record_date` 与用户当地日期匹配）时返回 `409`，不执行重复创建。调用方在创建前可先调用 `GET /api/v1/conversations/today` 检查状态，避免产生无效请求。

#### 请求参数

无请求体。会话的 `user_id` 从 Token 解析，`record_date` 由系统依据 `user_settings.timezone` 计算。

#### 返回参数

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | string (UUID) | 新建会话 ID |
| `status` | string | 固定为 `recording` |
| `record_date` | string (YYYY-MM-DD) | 会话归属的用户本地日期 |
| `created_at` | string (ISO 8601) | 会话创建时间 |

#### 返回示例（成功）

```json
{
  "code": 20000,
  "message": "success",
  "data": {
    "id": "c1a2b3d4-0000-4e5f-9abc-111122223333",
    "status": "recording",
    "record_date": "2024-06-01",
    "created_at": "2024-06-01T10:00:00Z"
  },
  "trace_id": "req-conv-create-001"
}
```

#### 错误情况

| HTTP 状态码 | 内部 code | 说明 |
|-------------|-----------|------|
| 401 | 40102 | Token 无效或过期 |
| 409 | 40901 | 今日会话已存在，不允许重复创建 |
| 500 | 50002 | 用户时区配置缺失或非法，无法确定今日日期归属；已记录高优先级告警 |

### 8.3 获取会话消息列表

**路径**：`GET /api/v1/conversations/{conversation_id}/messages`  
**鉴权**：需要（`Authorization: Bearer <token>`）

> 接口须校验 `conversation_id` 对应的会话归属当前用户，防止 IDOR 越权访问。

#### 路径参数

| 参数名 | 类型 | 说明 |
|--------|------|------|
| `conversation_id` | string (UUID) | 目标会话 ID |

#### 查询参数（可选）

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `limit` | integer | 否 | 每次返回的消息条数；默认值建议为 50，上限 100 |
| `before_sequence` | integer | 否 | 返回 `sequence_number` 严格小于此值的消息，用于向前翻页加载更早内容。**若不传此参数**，服务端取当前会话中序号最大的最近 `limit` 条消息；**无论何种取法，返回列表均按 `sequence_number` 升序排列**，以保证前端渲染顺序的一致性。 |

#### 返回参数

| 字段 | 类型 | 说明 |
|------|------|------|
| `conversation_id` | string (UUID) | 会话 ID |
| `total_count` | integer | 当前会话消息总数 |
| `messages` | array | 消息列表，按 `sequence_number` 升序排列 |
| `messages[].id` | string (UUID) | 消息 ID |
| `messages[].role` | string | 发送方角色：`user` / `ai` |
| `messages[].content_type` | string | 内容类型：`text` / `voice` |
| `messages[].content` | string | 消息文本内容 |
| `messages[].media_file_id` | string (UUID) \| null | 语音类型消息的媒体文件 ID；文字消息为 `null` |
| `messages[].sequence_number` | integer | 消息在会话内的有序序号 |
| `messages[].created_at` | string (ISO 8601) | 消息写入时间 |

#### 返回示例（成功）

```json
{
  "code": 20000,
  "message": "success",
  "data": {
    "conversation_id": "c1a2b3d4-0000-4e5f-9abc-111122223333",
    "total_count": 6,
    "messages": [
      {
        "id": "m0001-uuid",
        "role": "user",
        "content_type": "text",

        "media_file_id": null,
        "sequence_number": 1,
        "created_at": "2024-06-01T10:05:00Z"

      {
        "id": "m0002-uuid",
        "role": "ai",
        "content_type": "text",
        "content": "听起来是段久违的重逢，见面时心情怎么样？",
        "media_file_id": null,
        "sequence_number": 2,
        "created_at": "2024-06-01T10:05:03Z"
      }
    ]
  },
  "trace_id": "req-msgs-001"
}
```

#### 错误情况

| HTTP 状态码 | 内部 code | 说明 |
|-------------|-----------|------|
| 401 | 40102 | Token 无效或过期 |
| 403 | 40301 | `conversation_id` 不属于当前用户 |
| 404 | 40401 | `conversation_id` 不存在 |

### 8.4 发送用户消息并获取 AI 回复

**路径**：`POST /api/v1/conversations/{conversation_id}/messages`  
**鉴权**：需要（`Authorization: Bearer <token>`）

> 本接口采用同步响应模式，详见第 9 节设计建议。  
> 仅 `status` 为 `recording` 的会话接受本接口调用；`completing` 或 `completed` 状态下返回 `409`。  
> 接口须校验 `conversation_id` 归属当前用户。

#### 路径参数

| 参数名 | 类型 | 说明 |
|--------|------|------|
| `conversation_id` | string (UUID) | 目标会话 ID |

#### 请求参数（Body JSON）

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `content_type` | string | 是 | 消息内容类型：`text` / `voice` |
| `content` | string | 是（`text` 类型） | 消息文本内容；`content_type` 为 `text` 时必填，不允许为空字符串 |
| `media_file_id` | string (UUID) | 是（`voice` 类型） | `content_type` 为 `voice` 时必填，对应已上传的 `media_files.id`；`text` 类型时不传 |

#### 请求示例（文字消息）

```json
{
  "content_type": "text",
  "content": "今天下午去见了老朋友，聊了很久。"
}
```

#### 请求示例（语音消息）

```json
{
  "content_type": "voice",
  "media_file_id": "f1a2b3c4-0000-4d5e-8abc-999988887777"
}
```

> **说明**：语音消息的 `content` 字段由系统在 ASR 转写完成后写入，前端不传。若 ASR 尚未完成，后端应等待 ASR 结果后再构建 LLM 请求，不使用空内容调用模型。

#### 返回参数

| 字段 | 类型 | 说明 |
|------|------|------|
| `user_message` | object | 本次写入的用户消息对象 |
| `user_message.id` | string (UUID) | 用户消息 ID |
| `user_message.role` | string | 固定为 `user` |
| `user_message.content_type` | string | `text` / `voice` |
| `user_message.content` | string | 消息文本（语音消息为 ASR 转写结果） |
| `user_message.sequence_number` | integer | 用户消息的序号 |
| `user_message.created_at` | string (ISO 8601) | 消息写入时间 |
| `ai_message` | object | AI 回复消息对象 |
| `ai_message.id` | string (UUID) | AI 消息 ID |
| `ai_message.role` | string | 固定为 `ai` |
| `ai_message.content_type` | string | 固定为 `text` |
| `ai_message.content` | string | AI 回复文本 |
| `ai_message.sequence_number` | integer | AI 消息的序号（= 用户消息序号 +1） |
| `ai_message.created_at` | string (ISO 8601) | AI 消息写入时间 |

## 9. 发送消息并同步返回 AI 回复的接口设计建议

### 9.1 同步模式的实现路径

本模块 MVP 阶段采用**同步响应模式**：

1. 后端接收用户消息，写入 `conversation_messages`（`role=user`）
2. 从 `conversations` 加载当前会话的消息历史，构建 LLM 请求上下文
3. 调用 AI 服务，等待模型返回回复内容
4. 将 AI 回复写入 `conversation_messages`（`role=ai`）
5. 将用户消息与 AI 回复作为整体一次性返回给前端

**优点**：前端实现简单，无需轮询或 WebSocket；接口状态明确，成功即表示 AI 已完成回复。  
**缺点**：接口响应时间受 LLM 调用延迟影响，通常在 2~10 秒；需设置合理超时时间（建议 30 秒）。

### 9.2 超时处理建议

**设计权衡：**

> **建议**：将后端 LLM 调用的超时时间设置为 30 秒，接口整体超时时间设置为 35 秒，并向调用方返回专用错误码（内部 code `50401`）。  
> **原因**：同步模式下若不设置明确的超时时间，网络波动或模型负载高峰期可能导致请求长时间挂起，占用服务器连接资源。前端应在收到超时响应后提示用户重试，不允许前端自行重传相同消息（重传应由用户主动发起，以避免重复写入）。

### 9.3 关于流式响应（Streaming）的扩展说明

**设计权衡：**

> **建议**：MVP 阶段不引入流式（Server-Sent Events / WebSocket）响应。  
> **原因**：流式响应需要前端处理增量文本拼接、断线重连、消息边界判定等复杂逻辑，同时需要后端处理流式写入与错误中断的补偿策略。MVP 阶段产品验证优先，同步接口在功能闭环上完全满足需求。若后续用户反馈等待体验差，可在接口层以相同路径引入 `Accept: text/event-stream` 的内容协商机制，不需要变更接口路径设计。

### 9.4 LLM 上下文构建建议

后端在调用 AI 服务前，需从 `conversation_messages` 中读取当前会话的历史消息作为上下文。

**设计权衡：**

> **建议**：按 `sequence_number` 升序取最近 N 条消息（建议 N=20）作为上下文窗口，不传递全量历史。  
> **原因**：随着对话轮次增加，全量历史消息的 Token 数量可能超出模型上下文限制，且会显著增加每次调用的成本。固定窗口大小是 MVP 阶段最简单有效的兜底策略；后续可根据实际对话长度调整或引入动态截断机制。

## 10. 「结束今日记录」接口应触发的动作

`POST /api/v1/conversations/{conversation_id}/complete` 接口的执行链路如下：

### 10.1 接口同步执行（在接口响应前完成）

1. **校验会话归属与状态**：确认 `conversation_id` 属于当前用户，且 `status` 为 `recording`
2. **状态流转**：将 `conversations.status` 由 `recording` 更新为 `completing`，同步更新 `updated_at`
3. **返回响应**：向调用方返回含 `status: "completing"` 的成功响应

### 10.2 接口异步触发（后台任务，不阻塞接口响应）

4. **触发总结生成任务**：向后台任务队列投递一条包含 `conversation_id` 的总结生成请求

> 任务执行内容属于日记录模块职责，本模块只负责投递任务，不关心任务内部实现。任务完成后，后台任务负责：将 `conversations.status` 推进至 `completed`；向 `daily_records` 写入 AI 生成的总结内容及结构化子表数据。

**设计权衡：**

> **关于是否在接口中同步等待总结生成完成**  
> **建议**：MVP 阶段异步触发，接口立即返回 `completing` 状态，不等待总结完成。  
> **原因**：总结生成需要读取全量对话消息并调用 LLM，耗时通常在 5~30 秒，不适合在接口链路中同步等待。前端可通过轮询 `GET /api/v1/conversations/today` 的 `status` 字段感知总结是否完成。

### 10.3 对消息写入的影响

接口成功返回后，`conversations.status` 已变更为 `completing`，此后所有发送消息的请求（`POST /messages`）均应返回 `409`，不再接受新消息写入。

| 优先级 | 接口 | 原因 |

## 13. 风险点与实现建议

### 风险 1：用户时区配置缺失或非法

- **风险描述**：若 `user_settings.timezone` 缺失或为非法 IANA 时区字符串，系统无法正确换算用户当地自然日，造成 `record_date` 写入错误数据。
- **建议**：不允许在时区字段缺失或非法时静默降级为默认值后继续写入。应立即终止当前操作，向调用方返回 `500`（内部 code `50002`），并记录高优先级告警供运维排查。时区字段的合法性保障应在 `PUT /api/v1/settings` 的入参校验阶段前置拦截（见用户设置模块），本模块以强依赖方式处理此异常，不做兜底。

---

### 风险 2：同一用户同一自然日创建重复会话

- **风险描述**：前端在网络超时场景下可能重复调用 `POST /api/v1/conversations`，导致尝试写入重复会话记录。
- **建议**：在数据库层对 `(user_id, record_date)` 建立联合唯一约束（部分索引：`WHERE deleted_at IS NULL`），当应用层捕获到约束冲突异常时，将其转化为 `409` 响应返回；不允许出现重复会话记录落库的情况。

---

### 风险 3：LLM 调用超时或失败时的消息状态一致性

- **风险描述**：用户消息已写入数据库后，若 AI 调用超时或报错，导致 AI 回复消息未落库，前端将收到错误响应，但用户消息已存在于数据库中，造成"用户消息孤立无 AI 回复"的不一致状态。
- **建议**：用户消息写入与 AI 回复写入须在整体流程上设计补偿逻辑。MVP 阶段建议：用户消息先写入，AI 调用失败时不回滚用户消息（保留原始记录），但在接口响应中返回错误；前端收到错误后提示用户重试，重试时由用户重新发送消息内容，后端将其作为新消息顺序写入，不尝试覆盖已存在的孤立消息。孤立的用户消息不影响摘要生成（总结任务可识别无 AI 回复的消息行并跳过或处理）。

---

### 风险 4：`sequence_number` 并发分配冲突

- **风险描述**：若多设备或网络重试同时对同一会话发送消息，`sequence_number` 的并发分配可能产生冲突，触发 `uq_conversation_messages_conversation_id_sequence_number` 唯一约束违反。
- **建议**：`sequence_number` 的分配应在数据库层以原子操作保障（如使用 `SELECT MAX(sequence_number) FOR UPDATE` 行级锁后递增），不在应用层并发计算后写入。当唯一约束冲突发生时，应用层捕获异常后可选择自动重试一次，重试仍失败时返回 `500`。

---

### 风险 5：结束记录后总结生成任务失败

- **风险描述**：异步总结生成任务在执行中可能失败（LLM 服务不可用、Token 超限等），导致 `conversations.status` 长期停留在 `completing`，不推进至 `completed`。
- **建议**：后台任务须实现幂等重试机制，失败后按退避策略最多重试 3 次。若全部重试均失败，仍将状态保持为 `completing`（不回退至 `recording`），并通过告警通道通知运维介入。前端可对 `completing` 状态设置超时提示（如超过 5 分钟仍未变为 `completed` 时展示"生成中，请稍后查看"），不将此状态视为用户侧操作失败。

---

### 风险 6：`/conversations/today` 路由与 `/{conversation_id}` 的路径冲突

- **风险描述**：若路由注册顺序不当，字符串 `today` 可能被框架误匹配为 `{conversation_id}` 的 UUID 路径参数，导致 `today` 请求进入错误的处理函数并因 UUID 解析失败返回 `400` 或 `422`。
- **建议**：在 FastAPI 路由注册时，确保 `/conversations/today` 的路由定义在 `/conversations/{conversation_id}` 之前注册；同时在 `{conversation_id}` 路径的处理函数中加入 UUID 格式的入参校验，非 UUID 格式的请求在入参层即返回 `422`，不进入业务逻辑。
