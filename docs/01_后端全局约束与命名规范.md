# 《「回响（Echo）」后端全局约束与命名规范》

本规范旨在为「回响（Echo）」MVP 后端应用的开发提供统一的工程化准则。所有后续模块的设计与开发均需严格遵循本约束，以确保基于 FastAPI 与 PostgreSQL 的系统在架构层面上保持高度的一致性、可维护性与规范性。

## 1. 系统边界与 MVP 范围
- **架构模式**：采用单体应用结构（Monolith）提供 RESTful 服务，MVP 阶段不引入微服务体系。
- **存储边界**：以单一 PostgreSQL 数据库作为唯一的数据与文档存储中心，暂不引入消息总线、分布式锁及复杂的分库分表设计。
- **交付目标**：MVP 的核心闭环（登录进入 → 聊天记录 → AI 追问 → 结束记录 → 生成总结 → 查看历史 → 查看周报）均需在此工程边界下以最高效、最原生的方式完成闭环验证。

## 2. API 风格约定
- **协议与风格**：后端接口需严格遵循 HTTP 与 RESTful 架构风格。
- **动静分离**：
  - 资源型操作需映射为标准的 HTTP Method（`GET` 查、`POST` 增、`PUT`/`PATCH` 改、`DELETE` 删）。
  - 对于非标准资源类型的动作（如登录、生成总结等计算型业务），推荐将其视为子资源的动作指令，使用 `POST` 方法并在 URI 结尾附带行为动词。
- **数据交互**：前后端以无状态的 JSON 作为唯一的数据交互格式。

## 3. 路由前缀规范
- 所有的业务 API，必须统一收口于 `/api/v1` 前缀之下。
- 内部探测（如应用心跳、健康检查监控）可放置于根前缀，如 `/health`。

## 4. 统一返回结构
所有 REST 接口（除特殊声明的流式推送外）均需包装在统一格式的外层对象中，以保证前端拦截器的统一收口：
- `code`: 整数型，业务状态码。
- `message`: 字符串，对应的概要信息或错误提示明细。
- `data`: 字典或列表体，承载具体的业务返回载荷。
- `trace_id` (可选): 字符串，全链路追踪 ID，用于线上异常排查。

## 5. 错误码设计建议
采用“HTTP 状态码 + 内部业务错误码”的双层设计：
- **HTTP 状态码**：如实反映网络请求状态（200 成功，400 入参异常，401 鉴权缺失，403 权限不足，404 资源丢失，500 内部服务降级）。
- **内部业务错误码**：建议采用 5 位整数规则（如 `20000` 为一切正常，`40001` 为参数校验失败，`40101` 为 Token 过期，`50001` 为第三方 AI 调用超时）。无论内部业务错误码为何，结构必须通过顶层的 `code` 字段返回。

## 6. Token 鉴权规则
- **交互方式**：采用 JWT (JSON Web Token) ，调用方需将 Token 放入 HTTP 头部的 `Authorization` 字段，遵从 `Bearer <Token>` 格式。
- **验证机制**：鉴权过程不在具体业务函数中单独书写，必须收敛于全局的依赖注入（Dependency Injection）或路由前置拦截器中。

## 7. 时间字段规范
- **存储约定**：数据库中所有的时间类型必须统一设定为“带时区的时间戳”（`timestamptz` / `TIMESTAMP WITH TIME ZONE`），并强制以 UTC 0 时区落盘。
- **传输约定**：出入参必须采用标准的 ISO 8601 字符串格式。终端展现层的时间转化由前端消费端（依据设备本地时区）负责完成。

## 8. 主键/外键/索引/唯一约束命名规范
为了防止在未来进行数据库模式迁移时发生环境名冲突，所有数据库约束都应统一命名基准：
- **主键 (PK)**：`pk_{table_name}`
- **外键 (FK)**：`fk_{table_name}_{column_name}_{referred_table_name}`
- **索引 (IX)**：`ix_{table_name}_{column_name}`
- **唯一 (UQ)**：`uq_{table_name}_{column_name}`

## 9. 表命名规范
- **格式**：全部采用小写字母，单词之间使用下划线分割（Snake Case）。
- **词性**：统一使用名词的**复数**形式代表数据集合。
- **关联体系**：多对多等交叉中间表应按首字母字典序，用下划线串联两侧表清单进行命名。

## 10. 字段命名规范
- 依然遵循标准的小写下划线打断原则（Snake Case）。
- **布尔值强制约束**：表示状态的布尔值字段名称，必须以 `is_` 或 `has_` 作为前缀，用以杜绝语义模糊。
- **外键关联约定**：涉及外键连接的 ID，命名须明确指向单数目标的 ID 关联。

## 11. 枚举值命名规范
- 在代码逻辑中定义的 Enum 枚举选项必须采用纯大写字母，并以下划线分割单词（如 `TYPE_VOICE`, `TYPE_TEXT`）。
- 落地至数据库存储时，应直接作为定长字符串进行保存，避免因使用数据库原生的 Enum 类型而在频繁增删状态项时面临沉重的改表迁移负担。

## 12. 状态字段设计规范
- 凡是描述业务“生命周期”与“流转阶段”的字段，均应统一命名为 `status`。
- 严禁使用生僻的整数进行状态映射，应采用具备直观工程阅读性的大写字符串文本作为状态记录依据。

## 13. 软删除策略规范
- 日常业务产生的数据一律要求采用“软删除”模式，杜绝物理层面的直接擦除。
- 执行途径：在所有核心表内配给 `deleted_at` 字段；未删除的数据该列值为 `NULL`，删除时写入当时的含时区时间戳。所有查询默认加上该字段为空的范围限定。

## 14. JSONB 使用原则
- **使用边界**：仅可用于容纳高内聚、易变、不规则且不需要深度范围过滤特征的数据组（诸如 AI 提炼出的非标准化参数列表、变长标签等）。
- **禁止项**：严禁将涉及账户外键查询、状态流转筛选、系统时间过滤的关键字段包裹进 JSONB 内，防止引发索引失效和查询效率断崖式下跌。

## 15. 日志与审计字段建议
对于后续划分的所有数据表实体，建议全盘标配以下三大基准审计结构：
- `created_at`：记录初始产生的时间。
- `updated_at`：记录该行数据近期发生任何更新的时间戳。
- `deleted_at`：配套如上的软删审计。
如果有更加庞大详尽的管理需求，还可视具体情况扩展追踪操作人来源。

## 16. 各模块文档的后续组织方式建议
随着业务的发展，建议按照业务边界拆分成多个独立的 Markdown 单元化文档存放。每一份子模块文档应遵循内聚编排结构：
- **边界说明**：讲清该模块的承载功能与排除目标。
- **状态流转**：文字或图形说明该模块所含实体的生命周期。
- **数据结构概略**：只说明新引入的关键数据定义。
- **接口契约**：按照请求路由、Method、输入参数（请求体/路径头）到输出返回对象的标准样例顺序进行排列。
