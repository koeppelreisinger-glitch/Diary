# Echo AI 日记产品计划书

## 1. 产品概述

Echo 是一款面向个人日常记录、情绪回顾和生活复盘的 AI 日记产品。它不是要求用户每天写一篇完整作文，而是通过聊天、快速记录、图片上传和后续补写，把零散表达整理成完整日记，并自动提取事件、情绪、消费、地点和灵感。

当前代码已具备 MVP 闭环：

- 用户注册、登录和个人设置。
- 今日对话式记录。
- AI 追问与陪伴式输入。
- 结束记录后自动生成日记正文。
- 自动提取结构化数据。
- 历史记录、日历、时间轴和五表查询。
- 图片上传、图片回忆流和“历史上的今天”。
- 日记正文编辑、补写和结构化重建。

## 2. 产品定位

### 2.1 目标用户

Echo 主要服务三类用户：

1. 想记录生活但难以坚持写长日记的人。
2. 需要回顾情绪、消费、地点和事件轨迹的人。
3. 希望用 AI 帮自己整理日常素材、形成长期个人档案的人。

### 2.2 核心价值

传统日记的主要门槛是“要主动组织语言”。Echo 的价值是把记录拆成低压力的表达：

- 用户只需要像聊天一样说几句。
- AI 负责追问、整理、摘要和结构化。
- 系统把每天沉淀为可回看、可搜索、可分析的个人记忆。

一句话定位：

> Echo 是一个会听你说话、帮你整理生活线索的 AI 日记伙伴。

## 3. 问题与机会

### 3.1 用户痛点

- 写日记门槛高：不知道从哪里开始，也不想每天写长文。
- 信息分散：事件、情绪、消费、地点和照片散落在不同工具里。
- 回顾困难：过往记录无法有效检索，只能靠日期翻找。
- 情绪变化缺少长期视角：用户很难看清自己最近为什么疲惫、焦虑或开心。
- 图片有记忆价值，但很少能和文字记录自然关联。

### 3.2 产品机会

AI 的优势不只是生成文字，更适合做“整理者”：

- 从用户短句中识别关键事件。
- 从语气和内容中判断情绪。
- 从明确金额中提取消费。
- 从自然语言中提取地点和灵感。
- 把多轮对话整合成自然的第一人称日记。
- 后续可扩展为周报、月报、情绪趋势和个人知识库。

## 4. 产品目标

### 4.1 MVP 目标

让用户能完成一次完整记录闭环：

1. 登录。
2. 开始今日记录。
3. 通过对话输入今天发生的事。
4. 上传可选图片。
5. 结束记录。
6. 系统生成完整日记。
7. 用户在历史页回看和检索。

### 4.2 阶段目标

短期目标：

- 稳定日记生成链路。
- 提升生成速度和错误提示。
- 完善图片线上存储。
- 优化补写和编辑体验。

中期目标：

- 增加周/月复盘。
- 增加更清晰的情绪趋势分析。
- 增强消费、地点、事件的聚合统计。
- 建立更完整的回忆版块。

长期目标：

- 形成个人生活记忆库。
- 支持多模态输入：文字、图片、语音。
- 支持智能回顾、主题搜索和年度报告。

## 5. 核心功能设计

### 5.1 账户与用户设置

当前实现：

- `POST /api/v1/auth/register` 注册。
- `POST /api/v1/auth/login` 登录。
- `GET /api/v1/users/me` 获取当前用户。
- `PUT /api/v1/users/me` 更新昵称和头像。
- `GET /api/v1/users/stats` 获取累计记录天数、连续打卡、累计事件数。
- `GET /api/v1/settings` 和 `PUT /api/v1/settings` 管理偏好。

产品意义：

- 通过用户体系保证记录私有化。
- 时区设置决定“今天”的归属日期。
- 偏好设置为后续提醒、输入方式和界面模式提供基础。

### 5.2 今日对话记录

当前实现：

- `GET /api/v1/conversations/today` 查询今日是否已有会话。
- `POST /api/v1/conversations` 创建今日会话。
- `GET /api/v1/conversations/{conversation_id}/messages` 拉取消息。
- `POST /api/v1/conversations/{conversation_id}/messages` 发送消息并同步返回 AI 回复。
- `POST /api/v1/conversations/{conversation_id}/complete` 结束会话并触发后台总结。

产品逻辑：

- 今日记录以“会话”为容器。
- 用户每发一条消息，系统保存用户消息，再生成 AI 回复。
- AI 回复不追求长篇内容，而是用短句接住情绪，并继续引导用户补充事件、感受、消费、地点或灵感。
- 一天通常只对应一个今日会话，避免记录分散。

### 5.3 AI 日记生成

当前实现：

- 结束会话后，后端将会话置为 `completing`。
- 后台任务读取完整消息历史。
- `SummaryGenerationService` 调用 `DiaryAIService` 生成结构化 payload。
- AI 输出包括正文、摘要、情绪分、关键词和五类子表。
- 成功后写入 `daily_records`，并把会话状态改为 `completed`。

AI 输出核心字段：

- `body_text`：完整日记正文。
- `summary_text`：列表展示用摘要。
- `emotion_overall_score`：1 到 10 的整体情绪分。
- `keywords`：关键词。
- `events`：事件。
- `emotions`：情绪。
- `expenses`：消费。
- `locations`：地点。
- `inspirations`：灵感。

产品意义：

- 用户不用自己整理长文。
- 同一份日记既有可阅读的正文，也有可统计分析的结构化数据。

### 5.4 日记编辑与补写

当前实现：

- `PUT /api/v1/daily-records/{record_id}` 轻量编辑备注、关键词和灵感。
- `PUT /api/v1/daily-records/{record_id}/body` 手动修改正文并全量重建结构化数据。
- `POST /api/v1/daily-records/{record_id}/supplement` 保存补写并重建。
- `POST /api/v1/daily-records/manual` 手动补录过往日期。

产品意义：

- AI 生成不是终稿，用户保留最终控制权。
- 补写机制允许用户结束记录后继续添加遗漏内容。
- 正文重建保证用户修改正文后，结构化数据也能同步刷新。

### 5.5 历史记录与寻迹

当前实现：

- `GET /api/v1/history/daily-records` 查询历史列表。
- `GET /api/v1/history/daily-records/{record_date}` 查询某日详情。
- `GET /api/v1/history/calendar` 查询日历视图。
- `GET /api/v1/history/timeline` 查询时间轴。
- `GET /api/v1/history/events`
- `GET /api/v1/history/inspirations`
- `GET /api/v1/history/emotions`
- `GET /api/v1/history/locations`
- `GET /api/v1/history/expenses`

产品意义：

- 历史不只是“按日期翻页”，而是可以按内容类型回看。
- 寻迹页可以把生活拆成不同维度：花了什么钱、去了哪里、发生了什么、情绪如何、有什么灵感。

### 5.6 图片与回忆

当前实现：

- `POST /api/v1/media/upload` 上传图片。
- `GET /api/v1/media/history` 查询历史图片流。
- `GET /api/v1/media/history/{record_date}` 查询某天图片。
- `GET /api/v1/media/on-this-day` 查询历史上的今天。
- `DELETE /api/v1/media/{image_id}` 软删除图片。

后台处理：

- 上传后立即返回。
- 后台并行生成缩略图、主色调和 AI 图片描述。
- 图片可以关联会话，也可按日期归档。

产品意义：

- 图片是情绪和记忆的重要载体。
- “历史上的今天”可以形成自然的回忆入口。
- 后续可把图片描述和日记正文结合，增强多模态记忆。

## 6. 用户流程

### 6.1 新用户首次使用

1. 用户注册并登录。
2. 系统创建默认设置。
3. 用户进入今日页。
4. 用户点击“开始今日记录”。
5. Echo 进入对话状态。
6. 用户输入今天发生的事。
7. Echo 根据内容继续追问。
8. 用户结束记录。
9. 系统生成日记。
10. 用户查看今日总结。

### 6.2 老用户日常使用

1. 打开今日页。
2. 如果已有记录，直接查看和补写。
3. 如果还没记录，进入聊天记录。
4. 可使用快速模式记录消费、灵感、学习或聊天。
5. 可上传图片作为当天素材。
6. 完成后进入历史页或寻迹页回看。

### 6.3 历史回顾

1. 用户进入记录主页或寻迹页。
2. 按日期、关键词、情绪、消费、地点、灵感筛选。
3. 点击某天进入详情。
4. 查看正文、结构化条目和图片。
5. 必要时补写或重建。

## 7. 系统架构

### 7.1 分层结构

当前后端分层清晰：

- `app/api/routes`：HTTP 路由层，负责接收请求和返回统一响应。
- `app/services`：业务服务层，负责核心流程。
- `app/models`：数据库 ORM 模型。
- `app/schemas`：Pydantic 请求和响应模型。
- `app/core`：配置、数据库、鉴权、异常、安全工具。

### 7.2 主要数据模型

核心表：

- `users`：用户。
- `user_settings`：用户偏好。
- `conversations`：每日会话。
- `conversation_messages`：会话消息。
- `daily_records`：日记主表。
- `record_events`：事件。
- `record_emotions`：情绪。
- `record_expenses`：消费。
- `record_locations`：地点。
- `record_inspirations`：灵感。
- `daily_record_images`：日记图片。

设计原则：

- 主日记与五类子表拆分，便于查询和统计。
- 所有核心数据都按 `user_id` 隔离。
- 结构化子表支持 `source` 字段，区分 AI 生成和用户补充。
- 删除以软删除为主，降低误删风险。

### 7.3 通信方式

前端通信：

- 统一使用 `fetch` 调用 `/api/v1`。
- JWT 放在 `Authorization: Bearer <token>`。
- 普通 JSON 请求由 `apiFetch` 封装。
- 图片上传使用 `multipart/form-data`，手动附带 Authorization。

后端通信：

- TokenHub 调用通过 `httpx.AsyncClient`。
- AI 请求统一走 Chat Completions 格式。
- 总结和图片分析使用非流式返回，便于拿到完整 JSON。

统一响应：

```json
{
  "code": 20000,
  "message": "success",
  "data": {}
}
```

## 8. AI 能力设计

### 8.1 AI 陪伴回复

目的：

- 降低用户表达压力。
- 用短句承接用户情绪。
- 引导补充关键信息。

输入：

- 会话内已有消息。
- 系统提示词。
- 可选快速模式。

输出：

- 一条简短 AI 回复。

### 8.2 AI 日记总结

目的：

- 把对话整理成第一人称日记。
- 提取后续可查询和可统计的数据。

实现要点：

- 强制 JSON 输出。
- 对 AI 返回结果做解析和规范化。
- 如果 AI 不可用，部分场景使用本地 fallback。
- 写库过程使用事务，失败回滚。

### 8.3 AI 图片分析

目的：

- 为图片生成描述和标签。
- 提取主色调，支持更丰富的回忆展示。

实现要点：

- 图片上传先返回，不阻塞用户。
- 缩略图和 AI Vision 并行处理。
- 分析失败不影响图片基础记录。

## 9. 非功能需求

### 9.1 性能

- 今日页加载时并行请求今日会话和今日记录。
- 寻迹页并行请求消费、情绪、事件、地点、灵感和日历。
- 总结生成放入后台任务，避免用户等待长请求。
- 图片处理异步执行，上传响应尽快返回。

### 9.2 稳定性

- 数据库会话异常时自动回滚。
- AI 返回不合规时抛出明确错误或 fallback。
- Serverless 环境禁用连接池，降低连接泄漏风险。
- Neon/Supabase 等 SSL 场景有连接参数兼容处理。

### 9.3 安全

- 密码哈希存储。
- JWT 登录态。
- 所有用户数据按当前用户鉴权。
- 手机号展示脱敏。
- 图片删除校验归属。

### 9.4 可维护性

- 路由、服务、模型、schema 分层明确。
- 统一异常格式。
- 统一 API 响应结构。
- 文档按模块拆分，便于继续扩展。

## 10. 当前风险与限制

### 10.1 图片线上存储限制

当前本地环境可通过 `/uploads` 访问图片。Vercel 环境使用 `/tmp/uploads`，不适合持久化，也不能稳定提供公开 URL。

建议：

- 接入对象存储。
- 图片表保留 `storage_key`，URL 由存储服务生成。

### 10.2 后台任务可靠性

FastAPI `BackgroundTasks` 在普通部署中可用，但在 Serverless 场景下存在执行时长和冷启动限制。

建议：

- 中期引入任务队列。
- 或将生成任务拆成可重试的 job 表。

### 10.3 AI 结果稳定性

AI 输出可能出现遗漏或分类不稳定。

建议：

- 继续强化 JSON schema 约束。
- 增加结果校验和人工确认。
- 为关键字段提供用户编辑入口。

### 10.4 语音能力尚未完整闭环

schema 中预留 `voice` 和 `media_file_id`，但当前主要可用能力仍是文字和图片。

建议：

- 后续补齐语音上传、转写、入会话和日记生成。

## 14. 总结

Echo 的核心不是“自动写作文”，而是把用户每天零散的表达变成长期可回看的个人记忆系统。当前代码已经形成清晰 MVP：用户通过聊天输入，系统用 AI 追问和总结，最终沉淀为日记正文、结构化数据和图片回忆。

下一步最重要的是稳定生成链路、补齐线上图片存储、增强历史回顾体验。只要把“记录轻松”和“回看有价值”这两件事做好，Echo 就具备持续使用的产品基础。

# Echo（回响）AI 日记应用 — 项目报告

> 项目版本：v0.1.0-MVP

## 第一章 需求分析

### 1.1 开发背景

在快节奏的现代生活中，情绪管理与自我记录的需求持续增长，但传统日记应用存在以下痛点：
- **门槛高**：用户需主动打开空白文本框，不知道从何下笔
- **坚持难**：结构化输入带来的认知负担让用户难以养成日记习惯
- **价值低**：普通日记 App 仅做文字存档，缺乏结构化数据沉淀与情绪洞察能力

**Echo（回响）** 的核心出发点是：通过 AI 角色扮演"老朋友"的方式引导对话，让"记日记"变成一段轻松的睡前闲聊。对话结束后，AI 自动将内容转化为结构化日记，同时提取情绪、事件、消费、地点、灵感等多维数据，帮助用户构建个人生活数据库，实现长期价值的自然沉淀。

项目定名"Echo（回响）"，寓意每一天的记录都是生命中的一次回声，被永久留存。

### 1.2 面向用户

| 用户画像 | 说明 |
|---|---|
| 核心用户 | 18-35岁，希望养成日记习惯但难以坚持的年轻人 |

#### 3.2.3 主要表结构说明

| 字段 | 类型 | 说明 |

| phone | VARCHAR(20) | 主登录凭据，唯一约束（部分索引，deleted_at IS NULL）|
| email | VARCHAR(100) | 预留，MVP 不启用 |
| hashed_password | VARCHAR(255) | bcrypt 哈希，禁止明文 |
| nickname | VARCHAR(50) | 用户昵称 |
| status | VARCHAR(20) | `active` / `banned` |
| last_login_at | TIMESTAMPTZ | 最近登录时间 |

**`conversations` 表**

| 字段 | 类型 | 说明 |
|---|---|---|
| id | UUID(PK) | 主键 |
| user_id | UUID(FK→users) | 所属用户 |

**`conversation_messages` 表**

| 字段 | 类型 | 说明 |
|---|---|---|
| id | UUID(PK) | 主键 |
| conversation_id | UUID(FK→conversations) | 所属会话 |

**`daily_records` 表**（核心沉淀表）

| 字段 | 类型 | 说明 |
|---|---|---|
| id | UUID(PK) | 主键 |
| user_id | UUID(FK→users) | 所属用户 |
| conversation_id | UUID(FK, UNIQUE) | 关联会话（强 1:1）|
| record_date | DATE | 归属日期（冗余自 conversation）|
| body_text | TEXT | 日记正文（可被用户/AI重写的权威源）|

| emotion_overall_score | INTEGER | 综合情绪分 1-10 |
| keywords | JSONB | 关键词数组（无需深度过滤，适合JSONB）|
| user_note | TEXT | 用户私密备注 |

> 联合唯一约束：`(user_id, record_date) WHERE deleted_at IS NULL`

**五张结构化子表（共同模式）**

| 子表 | 特有字段 | 说明 |
|---|---|---|
| `record_events` | content(TEXT), source, is_user_confirmed | 当日事件 |
| `record_emotions` | emotion_label(VARCHAR), intensity(1-5), source | 情绪标注 |
| `record_expenses` | amount(NUMERIC 10,2), currency, category, description | 消费记录 |
| `record_locations` | name(VARCHAR 200), source | 地点信息 |
| `record_inspirations` | content(TEXT), source | 灵感（无 is_user_confirmed）|

> 所有子表均冗余 `user_id`（违反 3NF 但支持高效跨日聚合查询，是合理的反范式设计）
> 所有子表均有 `record_id`(FK→daily_records)、`source`(`ai`/`user`)、`deleted_at`

#### 3.2.4 数据库设计决策说明

| 决策 | 理由 |
|---|---|
| `keywords` 存 JSONB 而非独立表 | 关键词作为整体展示，无需单词级筛选，JSONB 降低建表冗余 |
| 五子表独立建表而非 JSONB | 支持跨日聚合（如"过去一月消费超 200 元的记录"），结构约束更严格 |
| `user_id` 冗余至子表 | 支持 `WHERE user_id=? AND record_date BETWEEN ? AND ?` 的跨日查询，无需 JOIN `daily_records` |
| `body_text` 独立于 `summary_text` | Path A 用户改正文时，AI 重建不会覆盖摘要；两者语义分离 |
| 软删除全局策略 | 保留审计链，支持越权操作检测，防止事务失败留下脏空洞 |

### 3.3 关键算法与核心技术

#### 3.3.1 AI 日记生成流程

```
会话消息列表（conversation_messages）
       │
       ▼
序列化为对话文本
[用户]: 今天下午去见了老朋友...
[AI助手]: 听起来是久违的重逢...
       │
       ▼
构造 Prompt
  System: 你是专业日记助手，必须输出指定 JSON 格式...
  User:   以下是今日对话记录，请提取 body_text/summary_text/
          emotion_overall_score/keywords/events/emotions/
          expenses/locations/inspirations
       │
       ▼
调用 TokenHub GLM-5
  temperature=0.3, timeout=90s
       │

  │ 成功    │ 失败
  ▼         ▼
JSON 解析  Fallback 规则引擎
```

#### 3.3.2 两条日记更新路径

**Path A（手动改正文）**：
```
用户提交新 body_text
→ 校验非空
→ 更新 daily_records.body_text
→ 软删旧五子表（含 source=user 的，保留用户手动加的 inspirations）

→ 写入新子表记录
→ 派生更新 summary_text / emotion_overall_score / keywords

```

```
→ 同一事务提交
```

#### 3.3.3 连续打卡计算算法

```python
# 取用户近 365 天日记日期，按倒序排列
dates = [record_date₁, record_date₂, ...]  # 降序

streak = 0
expected = today  # 若最近一条不是今天，改为 yesterday

for d in dates:
    if d == expected:
        streak += 1
        expected = expected - 1day
    elif d < expected:
        break  # 日期中断，停止计数
```

### 6.2 克服的主要技术难点

**难点 1：Vercel + asyncpg + Neon 三方兼容性**

Neon PostgreSQL 的标准连接串携带 `sslmode=require` 参数，而 asyncpg 驱动不支持该 URL 参数形式（需通过代码配置 SSL context）。同时，pydantic-settings 在 Vercel Serverless 冷启动时无法稳定读取环境变量。

解决方案：在 `app/core/config.py` 的 `SQLALCHEMY_DATABASE_URI` 属性中，优先通过 `os.environ.get("DATABASE_URL")` 直接读取，再执行 URL 参数剥离（去除 `sslmode/ssl/channel_binding`），并通过 `connect_args` 传入 SSL 配置。

**难点 2：AI JSON 输出的健壮性解析**

GLM-5 模型有时将 JSON 包裹在 markdown 代码块中（` ```json ... ``` `），有时输出被截断，有时字段名大小写不一致。

解决方案：实现多格式兼容的解析链：先尝试直接 `json.loads()`，失败则用正则从代码块中提取，再失败则启动 Fallback 规则引擎（关键词匹配情绪、正则提取金额、句子截断生成事件）。

**难点 3：日记重建的事务一致性**

日记重建涉及：读取旧子表 → 批量软删 → AI 提取新数据 → 写入新子表 → 更新主表，共 5 个步骤。任何一步失败都可能留下数据不一致状态（旧记录被删但新记录未写入）。

解决方案：全部步骤在同一个 SQLAlchemy async session 事务内执行，任何异常触发 `rollback()`，`conversation.status` 保持不变（重建失败时保持 `completing`，前端可重试）。

**难点 4：补充对话与日记状态机冲突**

原有设计中 `completed` 状态的会话拒绝一切新消息（设计合理但需要为「补充场景」开口）。若直接放开 completed 状态会破坏原有逻辑。

解决方案：引入 `is_supplement` 标志而非新增状态值，服务层在检测到该标志时跳过状态校验，最大程度复用原有代码路径，避免引入状态机复杂度。
