# 《06_历史记录模块》后端设计与开发规范/实施清单

## 一、 模块目标
* **角色定位**：历史记录模块在 Echo 后端中承担着“用户记忆检索中枢”的角色，负责结构化、高性能地呈现用户过往累积的日记录数据。
* **独立性**：它绝非 `today record` 的附属接口，因为 `today` 关注的是**当天数据的生成、增量修改与状态维护**；而 `history` 关注的是**跨周期的数据只读、复杂筛选与多维度聚合展示**。因此，必须将它们在架构上独立开来。
* **面向未来的扩展**：模块的底层架构不仅服务于当下的简单列表查询，还会成为未来“时间轴长卷”、“日历视图打卡情况”、“基于灵感与情绪的深度查询”乃至“系统自动生成周报/统计图表”的基座，因此在数据拼装和加载策略上必须解耦且高效。

## 二、 功能分层规划

### 1. 本轮直接实现
以下功能是当前可以直接进入编码阶段的核心链路：
*   **历史记录列表** (`GET /api/v1/history/daily-records`)：支持分页、按时间倒序、多条件筛选（日期范围、关键词、灵感、情绪分区间）。
*   **指定日期详情** (`GET /api/v1/history/daily-records/{record_date}`)：复用 `daily_record` 模型，带出底层五张子表的完整单日数据。
*   **历史日历视图** (`GET /api/v1/history/calendar`)：按月份高亮记录情况，返回含轻量摘要的日子数组。
*   **历史时间轴视图** (`GET /api/v1/history/timeline`)：按日期范围拉取聚合列表（建议按月分组或按日倒序，前端更易处理时间卷轴渲染）。

### 2. 本轮预留不实现（后续扩展）
以下能力在代码注释或 Schema 定义中留出设计空位，但**本轮坚决不落到业务代码里**：
*   全文搜索引擎集成（ES等）
*   复杂灵感交集筛选（本轮只做单一灵感检索）
*   消费统计聚合
*   情绪趋势图（将交由独立的数据图表接口处理）
*   历史导出（PDF/CSV等）
*   相似记录推荐
*   周报/月报联动预计算

## 三、 文件清单
建议针对历史模块实施最小范围变更，不影响既有结构。

1.  **`app/schemas/history.py` (新增/修改)**
    *   **职责**：存放历史专用的 Pydantic Schema。例如 `HistoryListResponse` 及各项查询、视图返回的模型封装。
2.  **`app/services/history_service.py` (新增/修改)**
    *   **职责**：承载只读性质的多日历史筛选、分页查库、日历及时间轴数据转换逻辑。从 `daily_record_service` 抽离，避免逻辑臃肿。
3.  **`app/api/routes/history.py` (新增/修改)**
    *   **职责**：实现历史模块的 `Router` 及参数校验，不写业务逻辑，通过调用 `history_service` 打包 `ApiResponse`。
4.  **`app/api/router.py` (修改)**
    *   **职责**：注册增加 `api_router.include_router(history.router)`。

*注意*：本阶段不要为了拆分出 `history_calendar.py` / `history_timeline.py` 这种子件而过度微服务化，当前所有只读视图都可在 `history_service.py` 承载，保持聚合。

## 四、 Schema 设计脚本

### 1. 历史列表
*   **`HistoryRecordItemResponse`**：
    `id`, `record_date`, `summary_text`, `emotion_overall_score`, `keywords`, `created_at`, `updated_at`。
*   **`HistoryListResponse`**：
    `total_count`, `total_pages`, `current_page`, `page_size`, `records: List[HistoryRecordItemResponse]`。

### 2. 历史详情
*   **必须复用** `app/schemas/daily_record.py` 中的 **`DailyRecordDetailResponse`**，承载底层关联子表（`events`, `emotions`, `expenses`, `locations`, `inspirations`），不要重复造轮子。

### 3. 日历视图
*   **`HistoryCalendarDayItemResponse`**：
    `record_date: date`, `has_record: bool`, `emotion_overall_score: Optional[int]`, `keywords: List[str]`, `summary_preview: str`（取总结前一段字数作预览即可）。
*   **`HistoryCalendarResponse`**：
    `year: int`, `month: int`, `days: List[HistoryCalendarDayItemResponse]`。

### 4. 时间轴视图
*   **`HistoryTimelineItemResponse`**：
    同历史列表类似的基础视图结构。
*   **`HistoryTimelineGroupResponse`**：
    按月聚合计载。`year_month: str` (如 "2026-04"), `items: List[HistoryTimelineItemResponse]`。
*   **`HistoryTimelineResponse`**：
    `groups: List[HistoryTimelineGroupResponse]`。

## 五、 Service 设计脚本 (`HistoryService`)
规划以下核心方法：

1.  **`list_daily_records(...)`**
    *   **入参**：`session`, `user_id`, `page`, `page_size`, `start_date`, `end_date`, `keyword`, `tag`, `min_emotion_score`, `max_emotion_score`。
    *   **出参**：`HistoryListResponse`。
    *   **逻辑**：通过过滤组装 `filters`。第一步通过 `select(func.count)` 取总数；第二步做 `desc(DailyRecord.record_date)` 排序和 `offset`, `limit` 切片。
    *   **查不到时**：直接根据前置检查，若总数为 0，则立刻返回 `total_count=0, records=[], total_pages=0`，不报错且不进行下一道检索逻辑。

2.  **`get_daily_record_detail_by_date(...)`**
    *   **入参**：`session`, `user_id`, `record_date: date`。
    *   **出参**：`DailyRecordDetailResponse`（复用结构）。
    *   **逻辑**：精准匹配。由于历史模块的特殊性，可直接调用复用 `DailyRecordService.get_record_by_date` 方法来实现深度查询。

3.  **`get_calendar_view(...)`**
    *   **入参**：`session`, `user_id`, `year: int`, `month: int`。
    *   **出参**：`HistoryCalendarResponse`。
    *   **逻辑**：构造日期起止判定跨度（获取该月的 1 日和月底）。检索范围内的记录字段，只返回存在记录的日子并映射成 `HistoryCalendarDayItemResponse` 返回，无该日数据则不用包含对象下发过去。

4.  **`get_timeline_view(...)`**
    *   **入参**：`session`, `user_id`, `start_date: date`, `end_date: date`。
    *   **出参**：`HistoryTimelineResponse`。
    *   **逻辑**：查询跨度内日期的核心轻字段元素。推荐用于 Echo 的显示：**按月分组** （`strftime("%Y-%m")`）。这更符合心境与记忆的月度时间轴长卷划分。

## 六、 Router 设计脚本

*   Router 绝不编写 SQL 操作、也不对复杂业务数据流进行拼合重构，纯依靠注入与声明。

1.  **历史列表**：
    *   路径：`GET /api/v1/history/daily-records`
    *   函数名：`get_history_daily_records`
    *   参数：`start_date: Optional[date]`, `end_date: Optional[date]`, 以及 `page`, `page_size`、各种筛选作为 Query 传入。调用对应方法注入。
2.  **详情接口**：
    *   路径：`GET /api/v1/history/daily-records/{record_date}`
    *   函数名：`get_history_daily_record_detail`
    *   参数：`record_date: date` 作为 Path 直接强类型声明传入。将 `current_user` 和它一起推入 Service。
3.  **日历接口**：
    *   路径：`GET /api/v1/history/calendar`
    *   函数名：`get_history_calendar`
    *   参数：`year: int`, `month: int` （利用 `Query(...)` 对参数进行约束检查）。
4.  **时间轴接口**：
    *   路径：`GET /api/v1/history/timeline`
    *   函数名：`get_history_timeline`
    *   参数：`start_date: date`, `end_date: date`。

## 七、 查询与加载策略
1.  **列表接口**：
    *   必须维持其“轻字段”。不能且不应该顺挂着带出五张大尺寸的附属子表属性，以此确保即便每页拉 50-100 条分页依旧极高吞吐率。多条件筛选叠加走 Filter 构建。
2.  **详情接口**：
    *   这完全是相反的策略。为了在拉取当天的细节时保障性能避免延迟递推查库：必须用 `selectinload` 把附属直接查透带出：
        `selectinload(DailyRecord.events)`
        `selectinload(DailyRecord.emotions)`
        `selectinload(DailyRecord.expenses)`
        `selectinload(DailyRecord.locations)`
        `selectinload(DailyRecord.inspirations)`
3.  **日历接口**：
    *   只下发轻量概要摘要。前端绘制日历点阵只需要确认 `has_record` 和情绪等核心标志，绝不要把关联级的数据塞入月度检索流中。
4.  **时间轴接口**：
    *   从体验维度，采取“按月份分组”下发方案更理想。时间流不仅在横向上拉伸，还可以按月建立视图节点边界供前端展示段落点。

## 八、 参数与校验规则
1.  **日期使用规范**：
    *   无论 Router 的任何接口，`record_date`, `start_date`, `end_date` 等，均在 Router 直接用 `date` 原生声明拦截。绝不应当在 Service 去自己引入 `strptime` 解析（FastAPI 原生即可包揽并抛出合规 422 警告）。
2.  **分页防飞坡**：
    *   必须绑定 `page: int = Query(1, ge=1)` 和 `page_size: int = Query(20, ge=1, le=100)` 来抑制恶意暴走拖库请求。
3.  **空结果 fallback**：
    *   任何列表/多项查询当获取到自身约束总数为 `total_count = 0` 时，安全下放且明确返回：`{ "records": [], "total_pages": 0 }` 等符合前端预期的空壳，不能引发框架崩塌或者给前端奇怪的 None。
4.  **强检索规定**：
    *   `keyword` 限定模糊找（针对 `summary_text.ilike()`）。
    *   `tag` 当前按灵感内容筛选日记录。
    *   极值情绪由 `Query(None, ge=... )` 卡住前后区间，然后入 `where()` 。

## 九、 安全与软删除规则
1.  **最底层越界防护**：
    *   在获取、统计或任何一根查询流水里，必须永远绑定：`DailyRecord.user_id == current_user.id`。没绑此项属于重大高风险渗透缺陷！
2.  **关于软删除的基线**：
    *   不要在开发脚本上强制采用像 `.and_(deleted_at == null)` 这类会由于不同框架或环境导致的不稳定字符串。全部查询必须遵循系统现行的基准规范过滤掉已软删除行（以当前项目中统一依赖基线为准）。
    *   附属查询若自带独立的软删属性也要同步过滤，或者依附于未删主表的映射传递。

## 十、 联调顺序
1.  优先利用脚本或者完成好的每日总结流程，给当下账号**灌入或堆叠出跨越十多天的 `daily_records`（包含情绪标签子表的丰满项）**。
2.  **调列表接口**：裸数据量查询无误后，分批组合测试（比如只筛某种关键字、指定三天范围日等），确认 `total_pages` 和 `records` 无误。
3.  **调详情**：用刚刷出的某两天下发到详情 Path。观测所有事件与开支数组是否一并获取、字段类型是否存在缺失。
4.  **本日历与轴**：验证分组正确叠加与缺省日期表现情况。
5.  **测查边界问题**：指定一月查空集、模拟未鉴权用户的串号（期望失败）、送入非常规参数结构等。

## 十一、 风险提醒
1.  **时区模糊的风险**：使用 `date` 时很容易与基于 UTC 格式持久化的系统生成偏移，造成查跨天找不到某一天。一定要警惕并使用原生类型规避时区差解析出错。
2.  **贪图方便造成查询污染**：日后增加时间轴相关，容易手一滑把所有内容塞一起给前端解析，造成庞杂性能过载。必须遵守前文说的“只有在单一详情路由里才能放纵查全子表”。
3.  **组合叠加逻辑运算错误**：在拼接 filter 列表的 `append` 过程中，逻辑的 “AND” 条件和不经意间的“OR”需要通过 SQLAlchemy 严格明确组合规范。
4.  **复用历史方法入侵**：将来出“周报总结”等能力一定不能强行挤入现有的 History 历史纯看读接口组，周报必然带复杂 LLM/Map-Reduce 算力调用。

## 十二、 最终开发顺序建议

*   **第一阶段：基石查询**
    完成 列表 (`GET /api/v1/history/daily-records`) 与 详情 (`GET /api/v1/history/daily-records/{record_date}`) 的底层 Schema 对齐和全功能实装，它是任何模块能够跑通的生命基盘。
*   **第二阶段：多元化视图呈现**
    实装 日历视图 和 时间轴视图 相关的业务与分组计算。
*   **第三阶段：进阶与除虫**
    在保证跑通的业务闭环下，对分页参数等强进行越界调优；补全并反复交叉验证“标签 & 缩紧时间 & 指定心情”等深度组合检索。
