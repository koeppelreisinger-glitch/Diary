# 《「回响（Echo）」数据库总览与关系设计初稿》

> 本文档为「回响（Echo）」MVP 阶段的数据库架构总览文件，供后续各业务模块接口设计阶段引用。  
> 本文档不包含任何代码、SQL、DDL 或接口设计，仅聚焦于实体识别、表职责划分、表间关系与约束设计。  
> 本文档的所有命名与设计准则须与《01_后端全局约束与命名规范》保持一致。

---

## 1. 核心实体识别

本系统的数据实体可划分为三个职责明确的层级：

**第一层：身份与配置层**  
负责用户账户的基础管理与个人化配置，是整个系统权限认证的根基。该层实体具备长周期稳定性，变更频率较低。

**第二层：交互记录层**  
负责承载用户每日与 AI 之间的真实对话过程。该层是系统的核心数据产生源，决定了所有后续沉淀的原始内容质量与结构完整性。

**第三层：沉淀复盘层**  
负责将交互层的原始对话流进行结构化提炼与归档，形成具备查询与统计价值的结构化资产。包含按日维度生成的总结主表、多维度原子化子表，以及跨周期的聚合报表。

## 3. 各表职责说明

### 3.1 `users`
存储用户账户的核心身份信息，包括登录凭据、昵称、状态等。是所有业务数据的最高归属主键来源。所有业务表通过 `user_id` 向此表建立外键关联。

### 3.2 `user_settings`
以 1:1 关系从 `users` 中分离出来的用户偏好配置表。  
之所以独立建表而非合并进 `users`，是因为配置字段的变更频率与账户字段不同，分离后可分别管理其更新逻辑。  
MVP 阶段，该表至少应承载以下配置内容：
- 用户本地时区（影响每日会话边界的判定与历史记录的日期归属）
- AI 伴侣名称与语气偏好（注入到 LLM System Prompt 中）
- 记录提醒时间偏好（用于后续推送通知的触发时机）

### 3.3 `conversations`
每日聊天会话的主体单元。在「回响」的产品设定中，一个用户在同一自然日只允许存在一条主会话记录。  
该表定义会话的整体生命周期状态（如：记录中、总结生成中、已完成），并作为 `conversation_messages` 与 `daily_records` 的关联中枢。

### 3.4 `conversation_messages`
保存会话内所有发生的消息记录，是系统最高频率写入的表。  
每一条消息均携带发送方角色标注（`user` 或 `assistant`），消息的物理写入顺序由时间戳决定。  
该表的数据是 AI 总结生成的直接原材料：在用户触发结束记录时，后端从该表读取当日全量对话流水，作为 LLM 上下文输入进行摘要提炼。

### 3.5 `media_files`
管理用户在聊天过程中上传的媒体资源（如语音录音、图片）的元数据。  
该表并不直接存储文件内容，而是记录文件在对象存储（OSS）中的路径与基础属性（如文件类型、大小、转写状态等）。  
当用户以语音形式输入时，后端需先将语音文件上传并在此表落盘，再调用 ASR（语音转写）服务获取文本，最终与 `conversation_messages` 关联。  
即便 MVP 初期以文字输入为主，该表仍建议先行建立，为后续功能扩展预留接入点。

### 3.6 `daily_records`
每日总结的主记录表，是沉淀复盘层的核心枢纽。  
当用户结束当天记录后，由 AI 异步生成的今日总结文字将写入此表，同时触发对 `record_*` 系列子表的结构化写入。  
该表通过 `conversation_id` 与 `conversations` 建立 1:1 的关联，并作为全部五张结构化维度子表的父节点。

### 3.7 `record_events`
存放从当日对话中提取出的具体独立事件条目，每条事件为一行（如"拜访了老朋友"、"参加了项目汇报"）。  
事件以独立表存在，支持对特定类型事件的跨日历史检索（如查询某人最近一个月的工作相关事件）。

### 3.8 `record_emotions`
存放从当日对话中提取的情绪分析结果，支持情绪类型标注与情绪强度权重数值。  
单日可能包含多种情绪，因此此表为 1:N 结构。  
独立建表的核心价值在于：可支持跨时间轴的情绪曲线分析与周报中的情绪趋势汇总。

### 3.9 `record_expenses`
存放从当日对话中识别出的消费记录，每条消费为一行（如"咖啡 32 元"、"打车 58 元"）。  
独立建表后，可实现跨日消费汇总、周/月消费统计，是周报"本周花销"板块的数据来源。

### 3.10 `record_locations`
存放从当日对话中提取的地点信息，如用户提及的城市、场所或具体地址。  
独立建表后，可支持地点维度的历史轨迹查询（如"我最近去过哪些地方"）。

### 3.11 `record_inspirations`
存放每日记录关联的灵感，兼容 AI 自动提取与用户手动添加的灵感来源。  
灵感结构相对轻量，但独立建表可支持跨日灵感聚合与基于灵感的历史记录过滤检索。

### 3.12 `weekly_reports`
以周为单位存储 AI 聚合生成的阶段性复盘报表。每个用户每周最多对应一条记录。  
该表的内容是静态落盘的生成结果，而非实时计算的视图，避免用户每次打开周报时对 `daily_records` 进行全量实时聚合运算。

## 4. 核心表之间的关系图

以下以文字形式描述各表之间的关联关系：

```
users
├── 1:1 → user_settings         （通过 user_id 关联；用户配置与账户分离管理）
├── 1:N → conversations          （一个用户拥有多个历史会话，按自然日区分）
├── 1:N → daily_records          （一个用户拥有多条每日记录）
└── 1:N → weekly_reports         （一个用户拥有多份历史周报）

conversations
├── 1:N → conversation_messages  （一个会话包含多条消息，按时间戳排序）
└── 1:1 → daily_records          （一个完成的会话对应唯一一条每日主记录）

conversation_messages
└── N:1 → media_files            （一条消息可关联一个或零个媒体文件）

daily_records
├── 1:N → record_events          （一条日记对应多个提取出的事件条目）
├── 1:N → record_emotions        （一条日记对应多个情绪标注条目）
├── 1:N → record_expenses        （一条日记对应多个消费记录条目）
├── 1:N → record_locations       （一条日记对应多个地点信息条目）
└── 1:N → record_inspirations            （一条日记对应多个灵感条目）
```

说明：`weekly_reports` 与 `daily_records` 之间为**逻辑聚合关系**，不建立持久化外键，通过用户 ID 与日期范围在业务层做查询关联。

## 5. 一天一条记录与一次会话的关系设计建议

### 是否允许一个用户一天只有一个主会话

**结论：是。MVP 阶段，一个用户在同一自然日只允许存在一条 `conversations` 记录。**

判定"同一自然日"的依据，来自 `user_settings.timezone`。系统在用户首次发起记录请求时，读取该时区配置，将用户当地的本地日期作为会话的归属日期（即 `record_date` 字段）。

### 如果用户同一天多次进入聊天，底层如何处理

用户在同一日内的多次进入行为（如早晨记录一段、晚间补充一段），均被路由至该日唯一的 `conversations` 记录中，持续追加 `conversation_messages`，直至用户主动触发结束，或系统在跨日时触发定时收尾。

### `daily_records` 是否与 `conversations` 一一对应

**结论：是，且为强 1:1 约束。**  
一条 `conversations` 在完成状态流转后，有且仅有一条对应的 `daily_records` 产生，并通过 `conversation_id` 外键建立明确的父子关联。这一约束须同时在应用层逻辑与数据库唯一约束层面双重保障。

### 设计优点与限制

**优点：**
- 数据模型清晰，无需处理跨会话合并的复杂逻辑
- 便于实现"今日是否已记录"的状态判断
- 日历视角的历史查看逻辑简单，直接以 `record_date` 检索

**限制：**
- 不支持用户在同一天分主题记录多块独立内容（该需求属于 V2 范畴）
- 若用户更换设备且时区配置未同步，存在归日偏差风险（须在时区规范中处理）

## 6. 会话与消息的关系设计

### 一对多关系

`conversations` 与 `conversation_messages` 之间为标准的 1:N 关系，通过 `conversation_id` 外键字段建立关联。一个会话可包含任意数量的消息条目。

### 消息顺序的处理方式

消息列表的排序以 `created_at` 时间戳的升序排列为标准，完整还原对话的时间线顺序。建议在 `created_at` 字段上建立索引，以支持高效的分页与排序查询。

### `conversation_messages` 为何独立存在

消息流水是系统写入最频繁的数据单元，其生命周期与 `conversations` 相对独立：
- 消息本身需要逐条追加，而会话主体字段（如状态）仅在特定节点发生变化
- 消息数量不可预知，嵌套于会话主表会导致行数据无限膨胀，无法有效建立索引
- 独立存在后，可分别对消息层面进行查询、分页、软删除管理，且易于未来引入归档策略

## 7. 日记录与结构化子表的关系设计

### 关联方式

`daily_records` 与五张结构化维度子表（`record_events`、`record_emotions`、`record_expenses`、`record_locations`、`record_inspirations`）之间均为标准的 1:N 关系，通过子表中的 `record_id` 外键字段向 `daily_records` 主键关联。

### 为何这些维度应保持独立表，而非合并进 `daily_records`

将结构化维度数据以独立行存储于独立表，是本系统能够实现数据查询价值的核心前提，原因如下：

1. **支持跨日查询与历史聚合**：若情绪、消费、事件等数据以 JSONB 数组的形式嵌套在 `daily_records` 中，则无法在数据库层面对其内部元素进行有效的过滤、排序和聚合。例如，"查询过去三个月内我所有消费超过 200 元的记录"这类查询，在独立表中为标准 SQL 查询，在 JSONB 中则需依赖 JSON 路径表达式，可读性与维护性极差。
2. **支持多值场景**：单日可能同时包含多种情绪、多个地点、多条事件，独立表的 1:N 结构天然支持多值，无需约定 JSON 数组格式。
3. **便于后续扩展字段**：未来若需为事件增加"持续时长"字段，为情绪增加"触发原因"字段，独立表均可通过 `ALTER TABLE` 完成，而 JSONB 方案则缺乏结构约束。
4. **避免宽表陷阱**：将五类维度数据嵌入同一行会导致 `daily_records` 行宽快速膨胀，降低读取性能与维护清晰度。

### JSONB 与结构化表的边界说明

JSONB 字段仅适合用于 `metadata` / `extra_json` 等**补充性扩展字段**，容纳少量非结构化、暂时未定型的附属数据。它不能替代已明确业务语义、需要独立查询与聚合的主结构维度表。

## 8. 周报如何从日记录聚合

### 以 `daily_records` 为聚合基础

周报的生成应以当周已完成的 `daily_records` 为基础数据源，而非直接读取 `conversation_messages`。

原因如下：
- `daily_records` 是经过 AI 处理与人工确认的结构化归档，已完成去噪与摘要，具备更高的信息密度与可靠性
- `conversation_messages` 包含大量对话过程中的过渡性内容（如 AI 追问、用户的碎片短句），这些原始内容并不适合直接作为周报的输入语料
- 若以 `conversation_messages` 实时聚合，每次生成周报都需对该用户当周所有消息进行全量读取，对 LLM Token 消耗和接口响应时间造成不可接受的压力

### 是否建议落表缓存

**结论：是，必须落表。**  
`weekly_reports` 表存储的是一次性异步生成的静态文本结果，而非每次请求时的实时 SQL 聚合视图。  
生成触发机制建议为：在用户完成每周第一次打开周报时触发异步任务按需生成，或通过定时任务在每周特定时间点（如周日晚间）批量为活跃用户生成。  
生成完成后将结果持久化至 `weekly_reports`，后续用户查看时直接读取该静态记录，无需重复计算。

## 9. 建议保留的扩展字段

### 推荐配置 `metadata` / `extra_json` 字段的表

以下表建议各预留一个 JSONB 类型的扩展字段：

| 表名 | 扩展字段建议名 | 用途说明 |
|------|------------|----------|
| `daily_records` | `extra_json` | 容纳 AI 原始返回中超出主结构的附属信息，如 AI 评分、封面色彩推荐等 |
| `record_emotions` | `extra_json` | 容纳情绪分析的补充参数，如情绪来源描述文本 |
| `conversations` | `extra_json` | 容纳会话级别的模型调用参数快照，便于复盘与调试 |
| `weekly_reports` | `extra_json` | 容纳周报结构化统计摘要，如本周情绪分布比例、总消费金额等 |

### 使用边界规范

- 扩展字段仅用于承载结构尚未稳定、后续可能调整的附属数据
- 不得将具有明确业务查询需求（如需要 WHERE 过滤、JOIN 关联、ORDER BY 排序）的字段写入 JSONB 扩展字段
- 扩展字段的内容须在应用层通过 Pydantic Schema 约束其结构，防止写入任意格式的脏数据

## 11. 第一版不建议加入的复杂设计

以下设计在 MVP 阶段引入的收益不足以覆盖其带来的开发成本，建议推迟至产品验证完成后再评估引入：

1. **多会话并行体系**：允许用户在同一天创建多个主题独立的会话，涉及会话合并、总结整合等复杂逻辑，不适合 MVP 阶段。
2. **消息分区表（Table Partitioning）**：基于时间或用户 ID 的分区表设计，适合超大规模数据场景，MVP 单库单表足以支撑。
3. **复杂多租户体系**：当前为 C 端单用户产品，无需引入租户隔离逻辑。
4. **完整操作审计日志体系**：记录每一次字段变更的前后值、操作者 IP 的审计表，MVP 阶段以 `created_at` 和 `updated_at` 的基础审计字段满足需求即可。
5. **消息已读/回执状态体系**：当前为单用户 AI 对话场景，无需引入多方已读状态机制。
6. **非必要派生视图或汇总表**：如"情绪月报"、"地点热力表"等聚合派生表，在 MVP 阶段应通过对现有表的查询动态满足，而非提前建立静态派生表。

## 12. 数据一致性与唯一约束建议

数据完整性的保障不能仅依赖应用层逻辑，必须在数据库层面建立明确的唯一约束，以防止并发请求、网络重试等场景下的数据重复写入。

### `conversations` 表
建议建立 `(user_id, record_date)` 的联合唯一约束，保证同一用户在同一自然日只允许存在一条主会话记录。

### `daily_records` 表
建议建立 `(user_id, record_date)` 的联合唯一约束，保证同一用户在同一自然日只允许存在一条每日主记录。  
同时，`conversation_id` 字段建议建立唯一索引，确保一个会话不能对应多条日记录。

### `weekly_reports` 表
建议建立 `(user_id, target_year, target_week)` 的联合唯一约束，保证同一用户在同一自然周只允许存在一份周报记录，防止定时任务重复触发时产生数据重复。

### `user_settings` 表
`user_id` 字段建议建立唯一索引，确保配置表与用户表的 1:1 绑定在数据库层面得到强制保障。

### 为什么数据库级唯一约束不可或缺

- 应用层逻辑在高并发场景下（如同一请求的网络重试、多设备同时操作）存在时间窗口竞争，无法完全防止重复写入
- 数据库唯一约束是最后一道防线，由数据库引擎在事务层面保证原子性
- 有了数据库约束，应用层可以在捕获到约束冲突异常时，将其转化为"数据已存在，直接返回"的幂等响应，而非产生脏数据
