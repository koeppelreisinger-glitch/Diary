## 模块一：用户洞察与问题定义

### 1. 目标用户

Echo 主要面向以下用户群体：

| 用户类型 | 典型特征 |
|---|---|
| 想记录生活但难以坚持写长日记的人 | 有记录意愿，但面对空白页面时不知道从哪里开始，容易中断 |
| 关注情绪管理和生活复盘的人 | 希望长期观察自己的情绪、事件、消费和生活状态变化 |
| 希望用 AI 整理个人记忆的人 | 希望把零散表达、图片和日常素材沉淀为可回看、可搜索的个人档案 |

### 2. 用户痛点

传统日记工具通常要求用户主动组织语言，门槛较高。用户既要回忆当天发生的事情，又要把内容写成完整文章，长期坚持的成本较高。与此同时，日常生活中的事件、情绪、消费、地点、灵感和照片常常分散在聊天软件、相册、备忘录或支付记录里，事后回顾时只能按日期翻找，难以形成连续的个人记忆线索。

Echo 要解决的核心问题是：让用户不用“写作文”，只需要像聊天一样表达当天发生的事情，系统就能帮助用户整理成完整日记，并把其中的重要线索结构化保存，方便后续搜索、复盘和回忆。

### 3. 使用场景

| 场景 | 说明 |
|---|---|
| 晚间生活记录 | 用户睡前用几句话告诉 Echo 今天发生了什么，系统通过 AI 对话继续追问，并在结束后生成日记 |
| 情绪复盘 | 用户想知道最近为什么焦虑、开心或疲惫，可以通过历史记录和寻迹页查看情绪线索 |
| 消费与地点回顾 | 用户在日记中自然提到“午饭花了 15 元”“去了图书馆”，系统自动提取为消费和地点记录 |
| 灵感整理 | 用户随手写下想法、感悟或学习收获，系统将其沉淀为可检索的灵感条目 |
| 图片回忆 | 用户上传当天相关图片，图片与当天日记关联，并可在回忆页和“历史上的今天”中再次呈现 |

## 模块二：产品方案设计

### 1. 产品概述

Echo AI 日记应用是一款面向个人日常记录、情绪回顾和生活复盘的智能日记产品。它不要求用户每天写一篇完整文章，而是通过聊天、快速记录、图片上传和后续补写，把零散表达整理成完整日记，并自动提取事件、情绪、消费、地点和灵感等结构化信息。

一句话概括：Echo 是一个会听用户说话、帮用户整理生活线索的 AI 日记伙伴。

### 2. 核心功能

| 功能 | 说明 |
|---|---|
| 用户注册与登录 | 支持手机号注册、登录，使用 JWT 维护登录状态，保护用户个人记录 |
| 今日对话记录 | 用户每天可创建一条今日会话，通过自然聊天输入当天经历 |
| AI 陪伴回复 | 用户发送消息后，系统同步调用 AI 生成简短回复，引导补充事件、感受、地点、消费或灵感 |
| AI 日记生成 | 用户点击“结束今日记录”后，后台读取完整会话，生成日记正文、摘要、关键词和情绪分 |
| 结构化提取 | 自动提取事件、情绪、消费、地点、灵感五类数据，形成可检索、可分析的个人记忆资产 |
| 历史回忆 | 支持历史列表、日历视图、时间轴和关键词搜索，帮助用户回看过去记录 |
| 寻迹分析 | 将消费、情绪、事件、地点和灵感拆成独立维度展示，便于复盘生活轨迹 |
| 图片功能 | 支持上传图片、图片回忆流和“历史上的今天”，并预留 AI 图片描述能力 |
| 日记编辑与补写 | 用户可以手动修改日记正文，或在结束记录后继续补充内容并重建结构化数据 |

### 3. 产品架构与功能图

```text
用户
  |
  | 注册 / 登录 / 今日记录 / 回忆 / 寻迹 / 我的
  v
前端 HTML + CSS + JavaScript
  |
  | fetch 调用 /api/v1，携带 JWT Token
  v
FastAPI 后端接口层
  |
  | 路由层：auth / users / settings / conversations / daily-records / history / media
  v
业务服务层
  |
  | 认证服务、会话服务、日记生成服务、历史查询服务、媒体服务
  v
PostgreSQL 数据库
  |
  | users、conversations、messages、daily_records、五类结构化子表、图片表
  v
AI 服务 TokenHub Chat Completions / GLM 模型
  |
  | 对话回复、日记生成、结构化提取、图片分析
```

核心数据模型采用“日记主表 + 五类结构化子表”的方式设计。`daily_records` 保存完整日记正文、摘要、关键词和整体情绪分；`record_events`、`record_emotions`、`record_expenses`、`record_locations`、`record_inspirations` 分别保存事件、情绪、消费、地点和灵感。这样的拆分既保留了完整文本，又让日记可以按不同维度检索和统计。

### 4. 交互流程

1. 用户打开正式入口并注册或登录。
2. 用户进入今日页，点击“开始今日记录”。
3. 用户像聊天一样输入当天发生的事情，也可以补充消费、地点、情绪、灵感或图片。
4. 后端保存用户消息，并调用 AI 生成简短陪伴回复。
5. 用户点击“结束今日记录”。
6. 后端将会话状态改为生成中，并在后台调用 AI 生成完整日记和结构化数据。
7. 生成结果写入 `daily_records` 主表和五类结构化子表。
8. 用户在今日页查看生成结果，也可进入回忆页、搜索页或寻迹页进行复盘。

### 5. 创新与差异化

| 对比项 | 常见日记工具 | Echo AI 日记应用 |
|---|---|---|
| 输入方式 | 用户主动写完整文本 | 用户通过聊天表达，AI 陪伴式追问 |
| 内容整理 | 主要保存原始文本 | AI 自动生成完整日记正文和摘要 |
| 信息结构 | 以日期和标题为主 | 自动拆解事件、情绪、消费、地点、灵感 |
| 回顾方式 | 按日期翻找 | 支持搜索、时间轴、日历和寻迹维度回看 |
| 用户控制 | 写完后主要靠人工修改 | 支持 AI 生成后手动编辑、补写和结构化重建 |
| 多模态扩展 | 多数只支持文字或图片附件 | 图片与日记日期、对话和回忆流关联，并预留 AI Vision 分析 |

Echo 的创新点不只是“让 AI 写日记”，而是把 AI 放在记录链路中：记录时降低表达压力，结束时整理成文，沉淀时提取结构化线索，回顾时支持多维检索。它将普通日记从静态文本升级为可搜索、可分析、可持续演进的个人记忆系统。

## 模块三：AI 原生能力说明

### 1. AI 核心能力

Echo 使用 AI 的位置主要分为四类：

| AI 能力 | 触发时机 | 作用 |
|---|---|---|
| 纯文字对话追问 | 用户发送文本消息时 | 生成简短、温和的陪伴式回复，引导用户继续补充 |
| 含图片对话追问 | 用户发送含图片消息时 | 结合图片和文字内容进行回应，增强多模态记录体验 |
| 图片后台分析 | 图片上传后异步触发 | 生成图片描述、标签、缩略图和主色调等辅助信息 |
| 日记总结与结构化提取 | 用户结束今日记录后后台触发 | 一次性生成日记正文、摘要、情绪分、关键词和五类结构化数据 |

其中，作品核心功能运行依赖 AI 回复、日记正文生成和结构化信息提取。系统提示词要求 AI 根据用户今日对话内容生成自然日记，并输出 `body_text`、`summary_text`、`emotion_overall_score`、`keywords`、`events`、`emotions`、`expenses`、`locations`、`inspirations` 等字段，供后端校验后入库。

### 2. AI 如何解决痛点

| 用户痛点 | AI 解决方式 |
|---|---|
| 不知道日记从哪里开始 | AI 以聊天方式承接用户输入，让记录从一句话开始 |
| 不想每天写长文 | AI 根据多轮对话自动整理成完整日记正文 |
| 记录内容太散 | AI 从自然语言中提取事件、情绪、消费、地点和灵感 |
| 回顾困难 | 结构化结果进入数据库后，可用于搜索、时间轴、日历和寻迹展示 |
| 图片和文字割裂 | 图片上传后可关联当天日记，并通过 AI 图片描述增强回忆信息 |

AI 能力与用户痛点的绑定关系是本作品成立的关键：如果只有普通文本输入，用户仍然需要自己组织语言；如果只有 AI 生成文本，后续回顾仍然困难。Echo 将“陪伴输入、内容生成、结构化提取、回忆展示”组合成完整闭环，使 AI 直接服务于记录和复盘过程。

### 3. AI 技术方案

系统通过 TokenHub Chat Completions 接入 GLM 模型，后端统一封装 AI 调用逻辑。用户正在等待回复的对话场景采用同步调用，尽量缩短首句等待时间；不需要用户实时等待的日记生成和图片分析放入后台任务，避免阻塞主流程。

日记生成采用“总结 + 结构化提取合并”的方案：用户点击“结束今日记录”后，后端读取当日完整消息历史，单次调用 AI 返回完整 JSON。相较于把正文生成和结构化提取拆成两次调用，合并方案减少 API 请求次数，降低配额消耗，也让字段解析和一致性控制更稳定。

AI 结果不会直接无条件写入。后端会对返回内容进行 JSON 解析、字段校验、去重、截断和异常兜底；写库过程使用事务，保证主表和子表的一致性。如果 AI 输出不规范，系统会进行修正或降级处理，最终以程序校验后的结果为准。

## 模块四：加分项

### 1. 落地可行性

作品已形成 MVP 核心闭环：用户注册登录、创建今日会话、发送消息、收到 AI 回复、结束记录、后台生成日记、查询今日记录、回忆历史、搜索关键词、查看寻迹维度。后端采用 FastAPI，数据库使用 PostgreSQL，前端使用 HTML/CSS/JavaScript，整体技术栈成熟，便于本地运行、联调和继续扩展。

当前本地访问入口如下：

| 入口 | 地址 |
|---|---|
| 正式用户入口 | `http://localhost:8000/frontend/login.html` |
| 今日记录页 | `http://localhost:8000/frontend/today.html` |
| 回忆页 | `http://localhost:8000/frontend/index.html` |
| 寻迹页 | `http://localhost:8000/frontend/trace.html` |
| 后端 API 文档 | `http://localhost:8000/docs` |
| 健康检查 | `http://localhost:8000/health` |

### 2. 商业化思考

Echo 可以面向个人成长、情绪管理、生活复盘和长期记忆管理场景发展。可行的商业化方向包括：

| 方向 | 说明 |
|---|---|
| 订阅制个人工具 | 基础记录免费，高级 AI 总结、月报、年度报告、更多图片和语音额度作为会员能力 |
| 情绪与生活复盘服务 | 提供周报、月报、情绪趋势、消费洞察和主题回顾，满足自我管理用户需求 |
| 多模态记忆库 | 扩展语音、图片、OCR、地点轨迹等能力，形成个人生活知识库 |
| 隐私优先的个人数据产品 | 强调用户数据隔离、可导出、可删除和本地化部署选项，提升信任感 |

### 3. 目标市场

目标市场包括学生、年轻职场人、内容创作者、自我成长用户和情绪管理需求用户。这类用户通常有记录生活、整理思路或复盘状态的需求，但不一定愿意每天打开传统日记软件写长文。Echo 用聊天作为入口，降低了记录门槛，更容易融入日常使用。

### 4. 竞争优势

Echo 的优势主要体现在：

- 从“写日记”改为“聊日记”，更符合用户自然表达习惯。
- AI 不只生成正文，还提取可检索、可统计的结构化数据。
- 回忆页和寻迹页分别承担“情感化回顾”和“数据化复盘”的职责。
- 数据库结构清晰，主日记与五类子表拆分，利于后续扩展周报、月报和趋势分析。
- 支持手动编辑与补写，保留用户对日记正文的最终控制权。
- 图片功能与 AI Vision 能力预留了多模态扩展空间。
